---
title: Build a chat agent
description: Carry model conversation across Runs with a Session.
---

A chat application needs an Agent, a Session, and a loop. karcarthy does not
add a separate chat abstraction.

## Run the example

Set `OPENAI_API_KEY`, start a REPL, and load the example:

```clojure
(load-file "examples/chat/main.clj")
(example.chat/chat!)
```

```text
Type /reset to clear the conversation or /quit to exit.
you> Remember that my project is called karcarthy.
agent> Got it.
you> What is my project called?
agent> karcarthy.
you> /quit
```

The exact wording varies, but the second response can use the first turn
because both Runs receive the same Session.

## The state boundary

```clojure
(def assistant
  (k/agent
   {:name "chat-assistant"
    :model "gpt-5.6"
    :instructions "Be helpful and concise."
    :output-schema string?}))

(def session (k/session))

(defn ask! [input]
  (k/run! assistant input {:session session}))
```

Each top-level call to `run!` establishes a new run. The Session contributes earlier normalized
messages and receives the new turn after it completes.

Inspect or clear the conversation:

```clojure
(k/get-items example.chat/session)
(example.chat/reset-chat!)
```

A Session stores conversation, not arbitrary application state or a Run
checkpoint. [Sessions](/docs/sessions) defines that boundary and explains how
to provide durable storage.
