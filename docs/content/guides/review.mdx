---
title: Code review
description: Let a proposed change choose focused reviewers and verify a finding recursively.
---

This example makes dynamic orchestration observable. One code-review Agent
reads a proposed change, creates two or three relevant reviewers, and runs them
concurrently. The reviewer responsible for the riskiest claim creates a nested
verifier that tries to disprove its strongest finding.

The final answer is a conventional code review. The Run events retain the
generated Clojure and the complete Agent tree that produced it.

## Run the example

Set an OpenAI credential and run:

```bash
clojure -M:examples review
```

The bundled input is deliberately small and self-contained. It states that a
scheduler must:

1. claim only pending jobs;
2. choose the highest-priority jobs before applying the limit; and
3. prevent two workers from claiming the same job.

The diff violates all three requirements. These known defects make the example
possible to evaluate without requiring another checkout or external Tools.

## Read the output

The terminal first displays a live Run tree. Reviewer names vary because the
change determines the concerns, but the structure is explicit:

```text
Run run_7c2e9b… · running
└─ code-reviewer
   ├─ state-reviewer
   ├─ ordering-reviewer
   └─ concurrency-reviewer
      └─ finding-verifier
```

The command then prints `REVIEW`, followed by `Findings` and `Summary`. Each
finding includes severity, file and line, a failure scenario, a minimal fix,
and the verifier verdict when one exists.

Useful questions for this run are:

- Did the final review identify each of the three known defects?
- Did it report unsupported findings?
- Did the nested verifier reject or strengthen its candidate?
- How many Agent calls, model calls, tokens, and eval attempts were used?

## Inspect the generated Clojure

Retain the Run at a REPL:

```clojure
(require '[karcarthy :as k])
(load-file "examples/review/main.clj")

(def result
  (example.review/run-review!
   example.review/default-change))
```

The generated expressions and macroexpansions are retained as events:

```clojure
(->> (k/events result)
     (filter #(= :eval/expanded (:type %)))
     (map #(select-keys % [:agent :depth :expression :expansion])))
```

The first expression creates the top-level reviewers. The nested expression
creates `finding-verifier`. Inspect the Agent lineage independently:

```clojure
(->> (k/events result)
     (filter #(= :agent/started (:type %)))
     (map (juxt :agent :parent-id :depth)))
```

This distinguishes what the Agent generated from what the harness executed.

## Compare orchestration boundaries

Managers, workers, parallelism, and traces are common across agent libraries.
The relevant difference is what may be defined after the orchestrator reads the
diff.

| Library or pattern | Supplied by the application | Chosen at run time |
| --- | --- | --- |
| karcarthy | One code-review Agent and the Clojure runtime | The Agent creates new Agent definitions and their coordination as retained Clojure; created Agents receive the same recursive capability |
| [OpenAI Agents SDK](https://openai.github.io/openai-agents-python/multi_agent/) | Agents exposed as Tools or handoffs, plus optional Python orchestration | A manager chooses among exposed Agents; application code can control chaining and parallelism |
| [Pydantic AI Dynamic Workflow](https://pydantic.dev/docs/ai/harness/dynamic-workflow/) | A catalog of named subagents | The model writes sandboxed Python to fan out, chain, vote, or retry calls to that catalog |
| [LangGraph orchestrator-worker](https://docs.langchain.com/oss/python/langgraph/workflows-agents#orchestrator-worker) | A graph or functional workflow, worker function, and shared state | The orchestrator plans tasks and dispatches workers dynamically |
| [Agent-o-rama](https://github.com/redplanetlabs/agent-o-rama) | A graph of Java or Clojure functions deployed as an agent | The declared graph runs with integrated storage, tracing, evaluation, and deployment |

The closest comparison is Pydantic AI's Dynamic Workflow: in both cases the
model writes coordination code inside one Tool call. karcarthy additionally
allows that code to construct new Agent values and gives those Agents the same
recursive eval capability inside one Run. The tradeoff is direct: karcarthy
evaluates full-trust JVM Clojure, while Pydantic AI runs the generated Python in
a sandbox.

This comparison is about the orchestration boundary, not production readiness.
Libraries with storage, deployment, evaluation UIs, or sandboxing solve a
larger operational problem than this demonstration library intends to solve.

## Reproduce the comparison

Use `example.review/default-change`, the same model and reasoning effort, and no
repository or web Tools. Record:

- recall against the three known defects;
- verified findings and false positives;
- whether reviewers are predefined or created after reading the diff;
- whether a reviewer can create the nested verifier during the same run; and
- model calls, input and output tokens, and elapsed time.

Keep the generated code or trace with the result. This makes the comparison
about observable behavior rather than framework descriptions.

See [Agent orchestration](/docs/agent-orchestration) for the general patterns
and [Eval](/docs/reference/eval) for the exact execution boundary.
