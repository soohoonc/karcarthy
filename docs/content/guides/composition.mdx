---
title: Compose agents
description: Coordinate a fixed Agent architecture with ordinary Clojure.
---

Dynamic eval is useful when the task determines the architecture. When
you already know the architecture, write it directly.

This example runs security and API reviewers concurrently, then gives both
reviews to an editor.

## The coordinator

```clojure
(defn output! [run]
  (if (= :completed (:status run))
    (:output run)
    (throw (ex-info "Agent Run failed" {:run run}))))

(defn review-team [request]
  (let [[security api]
        (->> [security-reviewer api-reviewer]
             (mapv #(future (k/run! % request reviewer-options)))
             (mapv deref)
             (mapv output!))
        editor-input
        (str "Request:\n" request
             "\n\nSecurity review:\n" security
             "\n\nAPI review:\n" api)]
    (output! (k/run! editor editor-input))))
```

The reviewers and editor are Agents. The coordinator is a function. `future`
runs the reviews together; `output!` refuses to pass a failed Run to the
editor.

## Run the complete example

With `OPENAI_API_KEY` set, start a REPL from the repository root:

```clojure
(require '[karcarthy :as k])
(load-file "examples/composition/main.clj")

(def review
  (example.composition/review-system "/workspace/project"))

(def output
  (review "Review the current working tree against origin/main."))

(println output)
```

The complete definitions and approval policy are in
[`examples/composition/main.clj`](https://github.com/soohoonc/karcarthy/blob/main/examples/composition/main.clj).

Each `run!` here establishes an independent run when the function is called
outside an active one. If evaluated code calls the same function, its `run!`
calls participate in that active run.

Use this pattern when the team is part of the application. Use
[dynamic eval](/docs/guides/architect) when the task should determine
which specialists exist.
