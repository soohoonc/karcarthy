---
title: Create agents at runtime
description: Watch one Agent write and run two specialists.
---

This example starts one Agent, `architect`. After it reads the task, it writes
a failure analyst and a rollout planner, runs them concurrently, and uses both
answers in its final response.

## Before you begin

Run from a karcarthy checkout with JDK 21, the Clojure CLI, and
`OPENAI_API_KEY` or `RESPONSES_API_KEY` set.

## Run the demo

```bash
clojure -M:examples architect \
  "Review a migration from synchronous writes to a queue."
```

The terminal redraws the Run while it executes:

```text
Run run_7c2e9b… · running · 18s · 3 model calls · 8,421 tokens · 2 Agent forms
└─ architect · waiting for 2 Agents
   ├─ failure-analyst · calling model
   └─ rollout-planner · calling model
```

Names, usage, timing, and answers vary. The example fixes the number and roles
of the children so the same behavior is visible on every successful run.

When the Run finishes, the command prints `ANSWER` followed by the parent's
synthesis.

## Inspect it at the REPL

Start the bundled REPL:

```bash
clojure -M:examples repl
```

Then create a monitor and keep the Run:

```clojure
(require '[karcarthy :as k])

(def live (k/monitor {:display :tree}))

(def run
  (example.architect/run-architect!
   "Review a migration from synchronous writes to a queue."
   live))
```

Read the final answer and tree:

```clojure
(println (:output run))
(k/monitor live)
```

Inspect the Agent forms the model submitted:

```clojure
(->> (k/events run)
     (filter #(= :program/expanded (:type %)))
     (map :definition))
```

Those forms are retained from the real Run; the documentation does not invent
their names or source.

## What happened

1. The parent model received the task and called its built-in `agent` Tool
   twice.
2. Each call supplied one Clojure Agent form and explicit input.
3. karcarthy read, expanded, checked, and evaluated each form.
4. Both resulting Agents ran inside the parent's Run.
5. Their outputs returned to the parent as Tool results.
6. The parent model wrote the final answer.

The children share the parent's budgets, deadline, approvals, cancellation,
and event history. They do not inherit its conversation: each receives only
the input supplied in its `agent` Tool call.

The [Generated agents](/docs/generated-agents) page explains the mechanism.
[Compose agents](/docs/guides/composition) shows the other case: architecture
known in advance and written directly in Clojure.
