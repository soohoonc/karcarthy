---
title: Let an Agent write the workflow
description: Run the Architect example and inspect same-process Clojure eval.
---

The Architect Agent is told to use `eval` once to create a coordinator. The
coordinator is told to use `eval` again; its expression creates a failure
analyst and rollout planner, starts both with `future`, dereferences their Run
maps, and synthesizes their outputs.

```bash
clojure -M:examples architect \
  "Review a migration from synchronous writes to a queue."
```

```text
Run run_7c2e9b… · running · 18s · 6 model calls · 8,421 tokens · 2 evals
└─ architect · waiting for Agent
   └─ coordinator · waiting for Agents
      ├─ failure-analyst · calling model
      └─ rollout-planner · calling model
```

The root model sees the coordinator's result. Neither the coordinator nor its
specialists were predefined by the application.

## Inspect the expression

```clojure
(->> (k/events result)
     (filter #(= :eval/expanded (:type %)))
     (map :expression))
```

The two traces show the coordinator expression and its nested specialist
expression. Every branch has the root `:run-id`; usage and limits are aggregate.

## What this demonstrates

1. A model-authored Agent can itself author more executable Clojure.
2. The models authored control flow, not merely Agent configurations.
3. Each expression was ordinary Clojure data and then ordinary Clojure code.
4. `future` carried the dynamically bound run into concurrent work.
5. Each `run!` returned a normal Run map while sharing run-wide resources.
6. Both evaluation levels and orchestration stayed in the same JVM.

See [Dynamic eval](/docs/eval) for the programming model and [Eval
reference](/docs/reference/eval) for exact events and limits.
