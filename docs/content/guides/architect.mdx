---
title: Let an Agent write the workflow
description: Run the Architect example and inspect same-process Clojure eval.
---

The Architect Agent is told to use `eval` once. Its expression creates a risk
analyst and rollout planner, starts both with `future`, dereferences their Run
maps, and returns their outputs.

```bash
clojure -M:examples architect \
  "Review a migration from synchronous writes to a queue."
```

```text
Run run_7c2e9b… · running · 18s · 3 model calls · 8,421 tokens · 1 eval
└─ architect · waiting for Agent
   ├─ failure-analyst · calling model
   └─ rollout-planner · calling model
```

The parent model sees the vector returned by its expression and synthesizes the
answer. No child Agent was predefined by the application.

## Inspect the expression

```clojure
(->> (k/events result)
     (filter #(= :eval/expanded (:type %)))
     (map :expression))
```

The trace then shows both Agent branches with the same `:run-id` as the parent.
Usage and limits are aggregate.

## What this demonstrates

1. The model authored control flow, not merely an Agent configuration.
2. The expression was ordinary Clojure data and then ordinary Clojure code.
3. `future` carried the dynamically bound run into concurrent work.
4. Each `run!` returned a normal Run map while sharing run-wide resources.
5. Evaluation and orchestration stayed in the same JVM.

See [Dynamic eval](/docs/eval) for the programming model and [Eval
reference](/docs/reference/eval) for exact events and limits.
