karcarthy is a native, homoiconic Clojure agent harness. Agents are executable
Clojure values and forms. A model can write a new `(agent ...)` form while its
Agent is running, and the same kernel reads, expands, checks, evaluates, and
runs it immediately.

There is no workflow graph or second orchestration language. Humans, macros,
and models all author the same Agent program.

## Run an Agent

```clojure
(require '[karcarthy :as k])

(k/defagent assistant
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Answer clearly and concisely."
   :output string?})

(def run
  (k/run! assistant "Explain continuation-passing style."))

(:output run) ;=> "..."
```

`run!` owns one bounded model/Tool loop and returns a Run containing the output,
usage, events, and any failure.

## The Clojure aha: an Agent can write an Agent

Every model Agent automatically receives a Tool named `agent`. No generation
flag or manually registered Tool is required:

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions
   "Solve the task. Create and run a skeptical reviewer before answering."
   :output string?})

(def run
  (k/run! architect
          "Review this API migration for hidden risks."
          {:limits {:agent-forms 4
                    :depth 3}}))
```

During that Run, the architect's model can call `agent` with two fields. The
`source` field is a complete Clojure program:

```clojure
(agent
 {:name "risk-reviewer"
  :model {:transport :responses :id "gpt-5.6"}
  :instructions "Find hidden assumptions and concrete failure modes."
  :input string?
  :output string?})
```

The `input` field is the complete value given to that new Agent:

```text
Review this API migration for hidden risks.
```

karcarthy executes one path:

```text
parent Agent
  -> model calls the built-in agent Tool
  -> read one `(agent ...)` form
  -> macroexpand, evaluate, and verify it
  -> run the new Agent on `input`
  -> return its output to the parent model
  -> parent continues to its final answer
```

The child is an ordinary Agent, not a workflow node or serialized substitute.
It can use Tools, call known Agents, contain Clojure control flow, or generate
another Agent. Its source is both program and data: a developer can write it, a
macro can transform it, and a model can produce it at runtime. That is why
karcarthy is built in Clojure. Homoiconicity is doing runtime work here, not
just providing Lisp syntax.

The generated Agent receives only the explicit `input`, not the parent's hidden
conversation. Parent and child still share the current Run's limits, usage,
cancellation, approvals, and event lineage.

## Three composition choices

| Need | Mechanism | Who chooses |
| --- | --- | --- |
| Call a specialist known ahead of time | Put it in `:agents` | Parent model |
| Run a fixed topology | Write an Agent body with ordinary Clojure and `run!` | Application program |
| Create a role or workflow after seeing the task | Use the automatic `agent` Tool | Parent model at runtime |

For example, `:agents` makes an existing Agent callable:

```clojure
(k/defagent researcher
  {:description "Research one question."
   :model {:transport :responses :id "gpt-5.6"}
   :instructions "Research the assigned question."
   :output string?})

(k/defagent editor
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Write the answer. Ask researcher when useful."
   :agents [researcher]
   :output string?})
```

This is different from the built-in `agent` Tool: `researcher` already exists,
while `agent` accepts new Clojure source and constructs a new Agent during the
Run.

## The mental model

| Concept | Role |
| --- | --- |
| Agent | Instructions, capabilities, contracts, and optionally a Clojure body. |
| Tool | A model-callable Clojure function with an input schema. |
| Run | One bounded execution with output, usage, events, and failure data. |
| Session | Conversation history shared across otherwise independent Runs. |
| Context | Local dependencies available to code and Tools, never exposed automatically. |
| Model transport | Translation between the kernel's normalized request and a model API. |

The harness owns the model/Tool loop. Clojure owns control flow. Sessions own
conversation history. Transports only translate model I/O.
