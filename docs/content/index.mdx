---
title: The agent architecture is the program
description: Developers and models write the same Clojure Agents.
---

karcarthy is an agent harness for Clojure. An Agent is a Clojure program.

You can write one yourself:

```clojure
(require '[karcarthy :as k])

(k/defagent reviewer
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Review the supplied change. Return concrete findings."
   :input string?
   :output string?})
```

`defagent` creates an Agent value and keeps the Clojure form that created it.
Run the Agent with `run!`:

```clojure
(def run (k/run! reviewer "Replace synchronous writes with a queue."))

(:status run) ;=> :completed
(:output run) ;=> "..."
```

## An Agent can run other Agents

Give an Agent a Clojure body when you want to control how it works:

```clojure
(k/defagent review-team
  {:input string? :output vector?}
  [change]
  (->> [security-reviewer api-reviewer]
       (mapv #(future (k/run! % change)))
       (mapv deref)
       (mapv :output)))
```

This Agent runs two reviewers at once. The code is the whole architecture.
There is no workflow graph behind it. Use normal Clojure for branches, loops,
recursion, macros, and concurrency.

## A model can write Agents too

Every Agent that uses a model gets a Tool called `agent`. The model calls it
with Clojure source and an input:

```clojure
{:source
 "(agent
    {:name \"failure-analyst\"
     :model {:transport :responses :id \"gpt-5.6\"}
     :instructions \"Find one hidden failure mode.\"
     :input string?
     :output string?})"
 :input
 "Replace synchronous writes with a queue."}
```

karcarthy reads the form, evaluates it into an Agent value, and runs it. Its
answer goes back to the parent Agent.

```text
model writes `(agent ...)`
  → karcarthy creates an Agent value
  → karcarthy runs the Agent
  → the answer goes back to the model
```

A developer-written Agent and a model-written Agent produce the same value.
Both can use Tools. Both can call Agents. Both can contain Clojure code. A new
Agent can create more Agents.

That is what **the agent architecture is the program** means. The code you
write, the code the model writes, and the code karcarthy runs are the same kind
of program.

## Watch an Agent create a team

The Architect example asks one Agent to create two more: a failure analyst and
a rollout planner. It runs them together, then combines their answers.

```bash
clojure -M:examples architect \
  "Review a migration from synchronous writes to a queue."
```

The terminal shows both Clojure forms and both Agent Runs:

```text
RUN    architect
MODEL  -> new Agent
(agent {:name "failure-analyst" ...})

MODEL  -> new Agent
(agent {:name "rollout-planner" ...})

  RUN    failure-analyst
  RUN    rollout-planner
  DONE   failure-analyst
  DONE   rollout-planner
DONE   architect
```

[Read the walkthrough](/docs/guides/architect) or inspect the
[`architect` example](https://github.com/soohoonc/karcarthy/blob/main/examples/architect/main.clj).

## What `run!` returns

Each call to `run!` returns a Run. The Agent is the program; the Run records one
execution of it.

```clojure
(select-keys run [:status :output :usage :events :error])
```

A Run sets limits on model calls, tokens, depth, concurrency, generated Agent
forms, and time. Agents created by a model use the same limits and appear in
the same event history.

## Why Clojure

Clojure code is data. karcarthy can keep an Agent's source, let macros change
it, let a model write it, and execute it with Clojure itself. No second language
is needed.

Generated source runs as full-trust JVM Clojure. Contracts catch bad inputs and
outputs, but they are not a sandbox. Isolate the process if you do not trust the
source.
