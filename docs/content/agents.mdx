An Agent is a Clojure value that keeps the form that created it. Its config has
a name, instructions, Tools, input rules, and output rules. It may also have a
Clojure body.

An Agent runs in one of two ways:

| Form contains | What executes it |
| --- | --- |
| Configuration with a model, no body | karcarthy's model and Tool loop |
| Configuration and a Clojure body | The body |

Both forms create the same Agent value. Call either one with `run!`.

## Model-driven Agents

A model-driven Agent has no body:

```clojure
(k/defagent researcher
  {:description "Research one question and return sourced findings."
   :model {:transport :responses :id "gpt-5.6"}
   :instructions "Research carefully. Distinguish evidence from inference."
   :tools [web-search]
   :input string?
   :output string?})
```

When it runs, the harness:

1. validates the input;
2. calls the model with the instructions and callable schemas;
3. executes requested Tools or available Agents;
4. returns those results to the model; and
5. repeats until the model answers or the Run stops.

The Agent declares the loop. The harness owns its execution.

## Agents with Clojure bodies

An Agent may have a one-argument Clojure body:

```clojure
(k/defagent review-team
  {:input string? :output vector?}
  [change]
  (->> [security-reviewer api-reviewer]
       (mapv #(future (k/run! % change)))
       (mapv deref)
       (mapv :output)))
```

The body is the executor. It may use `let`, `if`, `case`, `loop/recur`, macros,
futures, functions, and calls to other Agents. karcarthy validates the input
before the body and the output after it.

The body is the workflow. There is no graph or config file behind it.

## `agent` and `defagent`

The declaration forms differ only in where the value is bound:

```clojure
(k/defagent reviewer
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Review the change."
   :output string?})
```

`defagent` defines a var named `reviewer` and derives the Agent name from it.
Use `agent` inside an expression and supply `:name` explicitly:

```clojure
(def reviewer
  (k/agent
   {:name "reviewer"
    :model {:transport :responses :id "gpt-5.6"}
    :instructions "Review the change."
    :output string?}))
```

Both macros retain their original form and macroexpansion:

```clojure
(k/definition reviewer)
(k/expansion reviewer)
```

## Runs

`run!` passes one input value to an Agent and returns a Run:

```clojure
(def run (k/run! researcher "What changed in the latest Java release?"))

(select-keys run [:status :output :usage :error])
```

A Run is `:completed`, `:failed`, or `:cancelled`. Completion means the Agent
and its contracts completed successfully; it does not mean an external
evaluator judged the answer correct. Check `:status` before consuming `:output`.

Each top-level `run!` creates a new bounded execution with its own root Agent,
limits, events, and optional Session. Calls performed through `:agents` or the
built-in `agent` Tool are nested inside that Run. A program Agent that calls
`run!` explicitly starts another independent Run.

The calls are recorded differently:

| Call site | What happens |
| --- | --- |
| `k/run!` in application or Agent code | New independent Run |
| Model calls an Agent listed in `:agents` | Nested Agent in the current Run |
| Model calls the built-in `agent` Tool | Compiled and nested Agent in the current Run |

## Who writes and chooses Agents

You can write an Agent before a Run. You can let a model choose an Agent you
already wrote. Or you can let the model write a new Agent during the Run. All
three paths use normal Agent values.

### Expose a known specialist

Put existing Agent values in `:agents` when the parent model should decide
whether to call them:

```clojure
(k/defagent support
  {:description "Resolve one billing question."
   :model {:transport :responses :id "gpt-5.6"}
   :instructions "Resolve the assigned billing question."
   :output string?})

(k/defagent manager
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Resolve the request. Ask support when useful."
   :agents [support]
   :output string?})
```

The model sees `support` as a Tool it can call. The specialist receives only
the explicit Tool input, not the parent's conversation. Its output returns to
the parent model. Limits, usage, cancellation, approvals, and events remain in
the same Run.

### Write the coordination in Clojure

Use a program Agent when application code should decide ordering, branching,
or concurrency. The [composition guide](/docs/guides/composition) builds a
parallel review team and handles failed child Runs explicitly.

### Let the model write another Agent

Every model-driven Agent also sees a built-in Tool named `agent`. The model can
call it with a complete Clojure Agent form and explicit input:

```clojure
{:source "(agent {:name \"specialist\" ...})"
 :input "Complete task input"}
```

The harness compiles that form into the same Agent value described on this
page, runs it inside the current Run, and returns its output to the parent
model. Use this when you cannot know what kind of specialist will help until
the model sees the task. The [Agent creates Agents guide](/docs/guides/architect)
shows a live run; [Agent forms](/docs/reference/agent-forms) gives the exact
Tool input and compiler steps.

These choices are not maturity levels. Prefer one Agent when one Agent is
enough.

## Configuration

Common configuration keys are:

| Key | Meaning |
| --- | --- |
| `:name` | Stable runtime name. `agent` requires it; `defagent` derives it. |
| `:description` | Tells a parent model when to call this Agent. |
| `:model` | Selects the model and transport for a model-driven Agent. |
| `:instructions` | Agent-specific instructions appended to the base prompt. |
| `:tools` | Tools the model may call. |
| `:agents` | Existing Agents the model may call. |
| `:input` / `:output` | Contracts checked at the Agent boundary. |
| `:input-schema` / `:output-schema` | Optional JSON schemas shown to the model. |
| `:limits` | Limits that narrow the containing Run for this Agent. |

The [API reference](/docs/reference/api#agent-and-defagent) lists every key.

## Run options

Pass local dependencies through `:context`; karcarthy makes them available to
Tools and program bodies without inserting them into model messages:

```clojure
(k/run! account-agent "Summarize account 123."
        {:context {:account-id "123" :db db}
         :limits {:model-calls 40
                  :depth 4
                  :parallelism 4
                  :agent-forms 8
                  :deadline-ms 300000}
         :cancel cancelled?})
```

Other options include `:session`, `:observe`, `:approval`, `:metadata`,
`:model-transports`, and `:evaluation-namespace`. See [`run!` in the API
reference](/docs/reference/api#run) for their exact shapes.
