title: Run data

## Agent

```clojure
{:karcarthy/type :agent
 :name "researcher"
 :model {:transport :responses :provider :openai :id "gpt-5.6"}
 :instructions "Research carefully."
 :tools [...]
 :input-schema string?
 :output-schema string?
 :definition-ns 'example.research
 :definition '(agent {...})
 :expansion '(...)}
```

Agent configuration is flat. `definition` and `expansion` are additional
inspection data, not a second configuration layer.

Unknown unqualified configuration keys are rejected. Namespaced keyword keys
are preserved as extension data, following the normal Clojure map convention:

```clojure
(assoc researcher :my-app/owner "research")
```

## Tool

```clojure
{:karcarthy/type :tool
 :name "lookup"
 :description "Look up a record."
 :input-schema {:type "object" ...}
 :output-schema map?
 :definition '(tool {...} [input] ...)
 :expansion '(...)
 :execute #object[clojure.lang.IFn ...]}
```

Tool input uses one `:input-schema` for validation and the model interface.
Tools use the same namespaced extension-key convention as Agents and Run
options.

## Run

```clojure
{:karcarthy/type :run
 :id "run_..."
 :status :completed
 :agent "researcher"
 :input "question"
 :output "answer"
 :usage {:model-calls 3
         :input-tokens 12000
         :output-tokens 900
         :evals 1
         :duration-ms 4200.0}
 :events [...]
 :error nil}
```

Participating `run!` calls return the same shape and same `:id`. Their `:events`
cover the Agent call; the first Run retains the aggregate event sequence.

## Session

A Session stores normalized conversation items in chronological order. Text
turns use:

```clojure
{:role :user :content "question"}
{:role :assistant :content "answer"}
```

Tool turns add an assistant Tool call followed by its result:

```clojure
{:role :assistant
 :tool-calls [{:id "call_..." :name "lookup" :input {...}}]}

{:role :tool
 :tool-call-id "call_..."
 :name "lookup"
 :content "..."}
```

Recoverable Tool failures use the same item with `:is-error true` and an error
value in `:content`.

Only the first Agent in a top-level Run reads from and writes to the supplied
Session. Participating Agents start fresh conversations. Top-level Runs using
the same Session instance serialize the complete read/run/append operation.

## Failure

```clojure
{:karcarthy/type :failure
 :kind :schema
 :phase :tool-output
 :message "Value does not satisfy schema for tool-output"
 :data {...}
 :recoverable? false}
```

## Monitor snapshot

`monitor-state` returns process-local data derived from events. Each run entry
contains status, elapsed time, usage, eval count, Agent order, parent IDs,
activities, active Tools, and errors. The completed Run and its events are the
record; the monitor is a projection for display.
