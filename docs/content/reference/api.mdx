## `agent` and `defagent`

```clojure
(k/agent config)

(k/defagent name config)
```

Both forms create an Agent. `defagent` defines a var and supplies `:name` from
its symbol. `agent` creates a value and requires `:name` in the config. Both
require `:model` and `:instructions`.

```clojure
{:name "researcher"
 :description "Research one question."
 :model "gpt-5.6"
 :instructions "Research carefully."
 :context-schema map?
 :input-schema string?
 :tools [search-tool]
 :agents [fact-checker]
 :output-schema ::report
 :max-turns 20
 :input-guardrails [safe?]
 :output-guardrails [grounded?]
 :limits {:model-calls 30}}
```

`:model`, `:instructions`, `:tools`, and `:agents` may be functions of the
current run context. Instructions must resolve to a string and are sent to
the model exactly as returned; karcarthy does not prepend a framework prompt.

The run-context value passed to dynamic configuration and guardrails is:

```clojure
{:run-id "run_..."
 :agent-id "agent_..."
 :parent-id nil
 :depth 0
 :context {...}
 :limits {...}
 :usage {...}
 :agent agent-value}
```

`:agents` makes those Agents callable by the model using the Agent-as-Tool
pattern. It does not create a hierarchy or transfer the conversation.
An object-shaped `:input-schema` becomes the callable schema; otherwise the
model calls the Agent with `{"input": ...}`.

Every model Agent receives an `eval` Tool for one Clojure expression. The
expression may create Agents and Tools and call `run!`.

```clojure
(k/agent? value)
(k/definition agent)
(k/expansion agent)
```

`definition` and `expansion` expose the retained Clojure forms for tooling and
experiments. They are not needed to run an Agent.

## `tool` and `deftool`

```clojure
(k/tool config [input] body ...)
(k/deftool name config [input] body ...)
```

A Tool requires `:name`, `:description`, and `:input-schema`; `deftool`
supplies the name. Optional keys are `:output-schema`, `:needs-approval`,
`:input-guardrails`, `:output-guardrails`, and `:to-model-output`.

Guardrails receive `[run-context value]`. `:needs-approval` accepts `:never`,
`:once`, `:always`, a boolean, or a function of `[run-context input]`.

```clojure
(k/tool? value)
```

## `run!`

```clojure
(k/run! agent input)
(k/run! agent input options)
```

Runs an Agent, waits for it to finish, and returns a Run map. The first call in
a dynamic call chain establishes a run. Calls made during that dynamic extent
participate in it and return the same shape with the same `:id`.

```clojure
(k/output run)
```

Returns the output of a completed Run. A failed or cancelled Run throws
`ExceptionInfo` with the complete Run at `(:run (ex-data error))`, which keeps
ordinary composition concise without discarding failure data.

```clojure
{:context {...}
 :session session
 :limits {:model-calls 100
          :input-tokens 1000000
          :output-tokens 100000
          :depth 8
          :concurrency 6
          :evals 20
          :deadline-ms 300000}
 :on-event event-fn
 :approval approval-handler
 :cancel cancellation-token
 :model-transports {...}}
```

`context` returns the current Run's local dependency value from a Tool or
evaluated expression:

```clojure
(k/context)
```

## Run monitors

```clojure
(def live (k/monitor))
(def displayed (k/monitor {:display :tree :out *out*}))

(k/monitor live)
(k/monitor-state live)
```

A Run monitor is both an event handler and a printable value. Pass it as
`run!`'s `:on-event` option. Passing it back to `monitor` shows the latest tree;
`monitor-state` returns data for every Run it has seen. `:display :tree`
redraws a live terminal tree; omit it for a silent monitor. The tree refreshes
elapsed time once per second and adds model usage when each call completes.
`:out` must be a `java.io.Writer` and defaults to `*out*`.

See [Events and monitoring](/docs/events).

## Sessions

```clojure
(def session (k/session))
(def named (k/session {:id "session_123" :items []}))

(k/session? session)
(k/session-id session)
(k/get-items session)
(k/add-items! session messages)
(k/pop-item! session)
(k/clear-session! session)
```

`session` is thread-safe and process-local. Durable stores implement
`karcarthy.session/Session`. Runs sharing one Session instance serialize their
read/run/append sequence.

## Prompts

```clojure
(k/prompt "Base instructions" (k/prompt-file "AGENTS.md"))
```

`prompt` joins strings and run-context functions. `prompt-file` reads UTF-8
text. The resulting string becomes the Agent's complete model-visible
instructions; karcarthy does not add a base prompt.

## Eval

Eval is deliberately model-facing rather than a separate public compiler API.
Every Agent receives the `eval` Tool with `{:code string}`. The current Agent
input is bound to `input` inside the expression. Code
can use `agent`, `defagent`, `tool`, `deftool`, `run!`, `context`, `definition`,
and `expansion`. See [Eval reference](/docs/reference/eval).

## Models

The concise form selects OpenAI Responses:

```clojure
"gpt-5.6"
```

The advanced form is:

```clojure
{:transport :responses
 :provider :openai
 :id "gpt-5.6"}
```

The Responses transport supports `:base-url`, `:path`, `:url`, `:api-key`,
`:api-key-env`, `:headers`, `:auth?`, `:timeout-ms`, `:stream`, and
`:provider-options`. Direct OpenAI calls read `RESPONSES_API_KEY` or
`OPENAI_API_KEY` by default.

Use a deterministic model in tests:

```clojure
(k/mock-model (fn [request] {:type :final :output "done"}))
```

`model!` calls the active transport from code executing inside a Run:

```clojure
(k/model! normalized-request)
```

## Supplied Tools

```clojure
(k/responses-web-search)
(k/local-tools {:cwd "/workspace"})

(def connection
  (k/connect-mcp! {:name "server"
                   :command "server-command"
                   :args ["--stdio"]}))
(def tools (k/mcp-tools connection))
(k/close-mcp! connection)
```

`local-tools` supplies `read`, `write`, `edit`, `bash`, and `search`. MCP uses
stdio and must be closed by the caller. See [Tools and context](/docs/tools),
[Model providers](/docs/protocols/models), and [MCP tools](/docs/protocols/mcp) for execution and lifecycle
semantics.

## ACP

```clojure
(k/serve-acp! {:agent assistant})
```

Or provide an Agent factory:

```clojure
(k/serve-acp!
 {:models ["gpt-5.6"]
  :agent (fn [{:keys [cwd model-id]}]
           (k/agent
            {:name "acp-agent"
             :model {:transport :responses :id model-id}
             :instructions "Complete the coding request."
             :tools (k/local-tools {:cwd cwd})
             :output-schema string?}))})
```

Client-provided MCP Tools are appended automatically. See [ACP server](/docs/protocols/acp) for
the complete session factory and supported protocol surface.

## Events and schema validation

(k/emit! {:type :application/decision :value value})
(k/events run)
