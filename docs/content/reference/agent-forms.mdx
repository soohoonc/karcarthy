This page specifies how a model submits the same Clojure `agent` form used by
application code.

A developer might write:

```clojure
(k/agent
 {:name "reviewer"
  :model {:transport :responses :id "gpt-5.6"}
  :instructions "Review the supplied change."
  :input string?
  :output string?})
```

A model submits the equivalent unqualified form as source:

```clojure
(agent
 {:name "reviewer"
  :model {:transport :responses :id "gpt-5.6"}
  :instructions "Review the supplied change."
  :input string?
  :output string?})
```

The evaluation namespace supplies the real `agent` macro. Both forms create the
same Agent value. No generation flag or manually constructed Tool is required.

## What the model sees

The built-in `agent` Tool is described as a small programming manual. It tells
the model:

- when a specialist or custom workflow is useful;
- when direct Tool use or an existing Agent is preferable;
- the two accepted `(agent ...)` forms;
- which model configuration, Tools, and Agents generated code may reference;
- what information a generated Agent receives; and
- how its output and failures return.

The actual model, Tool, and Agent catalog is generated for each Agent call, so
the description never advertises an unavailable symbol.

The call itself is:

```json
{
  "source": "(agent {:name \"reviewer\" ...})",
  "input": "Review the proposed change."
}
```

`source` is exactly one top-level Clojure `agent` form. `input` may be any JSON
value; the generated Agent's `:input` contract validates it after compilation.

## Accepted forms

A model-driven Agent:

```clojure
(agent
 {:name "generated-reviewer"
  :description "Review one proposed change."
  :model {:transport :responses :id "gpt-5.6"}
  :instructions "Review the supplied change and return concrete findings."
  :tools [read search]
  :input string?
  :output string?})
```

A Clojure Agent program:

```clojure
(agent
 {:name "generated-review-team"
  :input string?
  :output vector?}
 [change]
 (->> [security-reviewer api-reviewer]
      (mapv #(future (run! % change)))
      (mapv deref)
      (mapv :output)))
```

The form must evaluate to an Agent. `defagent`, Markdown fences, prose, and
multiple top-level forms are rejected. A program body has one input binding and
may use ordinary Clojure, nested `agent` forms, and the symbols advertised in
the Tool description.

## Information boundary

A generated Agent receives exactly the value supplied in `input`. It does not
receive the parent model's messages, reasoning, previous Tool results, or
Session history.

Run-local application context is still accessible to Clojure code and Tools
through `context`. karcarthy does not insert it into the generated Agent's
model messages.

Its own definition supplies its instructions and capabilities. Its final
output becomes the result of the `agent` Tool call; the parent model then
continues and decides how to use that result.

The parent and generated Agent share execution accounting—limits, token usage,
deadline, cancellation, approvals, and events—but those are harness mechanics,
not hidden prompt context.

## Compilation

```text
source string
  → read one top-level Clojure agent list
  → expand the real agent macro
  → evaluate the expansion
  → require an Agent value
  → validate input
  → run the Agent
  → return its output to the parent model
```

Reader evaluation is disabled while reading. Macroexpansion and Clojure's
compiler catch malformed forms; Agent and Tool contracts validate their
boundaries. Clojure is dynamically typed, so this is not whole-program static
type checking.

### Compiler events

In event names, `program` means the submitted Clojure source for a generated
Agent. These events are not emitted for every ordinary Agent or Tool call.

| Event | Meaning | Important fields |
| --- | --- | --- |
| `:program/read` | The compiler received the source string and entered the read phase. It records an attempted read, not proof that parsing succeeded. | `:source` |
| `:program/expanded` | The source was read as one top-level `agent` form and macroexpanded. | `:definition`, `:expansion`, `:namespace` |
| `:program/checked` | The expanded form passed the compiler's structural checks. | `:namespace` |
| `:program/evaluated` | Evaluation produced a valid Agent value. | `:agent`, `:namespace` |
| `:program/failed` | Expansion or evaluation failed. The containing Run also ends with a structured failure. | `:phase`, `:error` |

A successful compilation therefore emits `read`, `expanded`, `checked`, and
`evaluated` in order. The generated Agent's normal `:agent/*`, `:model/*`, and
`:tool/*` events follow when it runs.

## Available symbols

Generated code runs in a fresh evaluation namespace. It receives Clojure core,
the public Agent-program forms (`agent`, `tool`, `run!`, `context`, `model!`,
and `emit!`), public vars from the selected application namespace, and the Tool
and Agent values advertised in the dynamic catalog.

A Run may select the application namespace:

```clojure
(k/run! architect task
        {:evaluation-namespace 'example.generated})
```

## Recursion and limits

A generated model Agent receives the same built-in `agent` Tool and can write
another Agent. Bound recursive programs with Run limits:

```clojure
{:depth 4
 :agent-forms 10
 :model-calls 50
 :input-tokens 500000
 :output-tokens 50000
 :parallelism 4
 :deadline-ms 300000}
```

`:agent-forms` counts each submitted top-level Agent form, including failed
attempts.

## Direct compilation

A Clojure Agent program can use the same compiler directly:

```clojure
(def generated (k/compile-agent! source))
(def result (k/run! generated input))
```

Use this when application code controls where source and input come from.

## Security

Generated source is evaluated as JVM Clojure. It has the authority of its
namespace and process. Reader restrictions, advertised symbols, and contracts
are not a sandbox. Run untrusted generation in a process, container, or VM with
only the filesystem, network, credentials, and namespaces it should have.
