---
title: Why Clojure?
description: Homoiconicity makes dynamic workflows ordinary code instead of graph data.
---

Most agent libraries expose familiar primitives: an Agent, Tools, a run loop,
conversation state, guardrails, and traces. karcarthy keeps that vocabulary.
Clojure changes one boundary: when the right workflow is not known ahead of
time, the model can produce code as data and the running program can evaluate
it directly.

## Agents and Tools are plain data

```clojure
(def reviewer
  (k/agent {:name "reviewer"
            :model "gpt-5.6"
            :instructions "Review the proposed change."
            :input string?
            :output string?}))

(:instructions reviewer)
(assoc reviewer :instructions "Review only the API boundary.")
```

`agent` is a macro so the value can retain its definition and expansion:

```clojure
(k/definition reviewer)
(k/expansion reviewer)
```

The operational configuration remains flat on the map; there is no hidden
`:config` layer.

## The model writes an expression, not a node

The built-in `eval` Tool accepts only `code`; the current Agent input is already
bound to `input`:

```clojure
{:code
 "(let [r (agent {:name \"reviewer\"
                   :model \"gpt-5.6\"
                   :instructions \"Review the change.\"
                   :input string?
                   :output string?})]
    (:output (run! r input)))"}
```

The code may be any one Clojure expression. `let`, `if`, `mapv`, `future`,
macros, recursion, Agent construction, Tool construction, and `run!` retain
their normal meanings. That is stronger than accepting only an Agent-shaped
form: the model can author the coordination itself.

## Homoiconicity is doing real work

The boundary is:

```text
source string → Clojure data → macroexpansion → value
```

The parsed expression and expansion are emitted as `:eval/*` events. Agent and
Tool macros retain their own source and expansion. Evaluation happens in the
same JVM; orchestration does not launch a second Clojure process.

The result is not durable continuation machinery or a workflow graph. It is
the simpler Lisp claim: code is data, and that data can become code in the
currently running process.

## One run follows the code

Clojure dynamic bindings are conveyed to `future`, so `run!` calls made by
generated concurrent code participate in the existing run:

```clojure
(->> input
     (mapv #(future (run! reviewer %)))
     (mapv deref)
     (mapv :output))
```

Those calls share run limits, usage, events, cancellation, approvals, context,
and the run ID. They do not inherit the parent Agent's model conversation.

## Trust boundary

This is full-trust evaluation, not a sandbox. Use it only when the model and
available bindings are trusted for the process. Reader evaluation is disabled,
but evaluated Clojure can perform anything its namespace and JVM permissions
allow.
