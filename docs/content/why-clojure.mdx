---
title: Why Clojure?
description: Agent orchestration is ordinary Clojure.
---

Most agent libraries already provide Agents, Tools, conversation state, limits,
and traces. karcarthy keeps those familiar pieces.

Clojure changes one boundary: when the right orchestration depends on the task, a
model can produce code as data and the running program can execute it directly.

## One language for both cases

Suppose every change must receive correctness and concurrency reviews. Because
those roles are known, write the coordination yourself:

```clojure
(->> [correctness-reviewer concurrency-reviewer]
     (mapv #(future (k/run! % change)))
     (mapv (comp k/output deref)))
```

If the diff should determine which reviews exist, let an Agent generate the
equivalent expression after it sees the change:

```clojure
(let [correctness-reviewer
      (agent {:name "correctness-reviewer"
              :model "gpt-5.6"
              :instructions "Find concrete behavioral defects."
              :input-schema string?
              :output-schema string?})
      concurrency-reviewer
      (agent {:name "concurrency-reviewer"
              :model "gpt-5.6"
              :instructions "Check atomicity under concurrent workers."
              :input-schema string?
              :output-schema string?})]
  (->> [correctness-reviewer concurrency-reviewer]
       (mapv #(future (run! % input)))
       (mapv (comp output deref))))
```

The resulting coordination is Clojure, not a graph node that karcarthy later
has to interpret.

## The eval boundary

An Agent whose `:tools` includes `k/eval` receives an eval Tool with one input
field:

```clojure
{:code "(+ 1 2)"}
```

The current Agent input is already bound to `input`. `code` must contain exactly
one Clojure expression, but that expression may use normal `let`, `if`, `mapv`,
`future`, functions, macros, recursion, Agent construction, Tool construction,
and `run!`. Agents created by the expression do not inherit eval; include
`:tools [eval]` when recursive dynamic orchestration is intentional.

The boundary is small:

```text
source string → Clojure data → macroexpansion → evaluated value
```

The parsed expression and expansion are retained as `:eval/*` events.
Evaluation stays in the current JVM; coordination does not launch another
Clojure process.

## Agents and Tools remain data

Agent configuration is a flat map, so normal data operations still work:

```clojure
(def concurrency-reviewer
  (k/agent {:name "concurrency-reviewer"
            :model "gpt-5.6"
            :instructions "Review the change for races."
            :input-schema string?
            :output-schema string?}))

(:instructions concurrency-reviewer)
(assoc concurrency-reviewer :instructions "Review only the claim transaction.")
```

`agent` and `tool` are macros, so their values also retain source and expansion
for inspection:

```clojure
(k/definition concurrency-reviewer)
(k/expansion concurrency-reviewer)
```

There is no hidden `:config` layer and no separate workflow representation.

## One Run follows the code

Clojure carries dynamic bindings into `future`. Generated `run!` calls therefore
participate in the active Run and share its limits, usage, events, cancellation,
approvals, context, and ID. Each called Agent still begins a fresh model
conversation with only its explicit input.

## Trust boundary

Eval is full-trust JVM Clojure, not a sandbox. Reader evaluation is disabled,
but evaluated code has the permissions of the host process. Isolate the process
when generated code or its available dependencies are not trusted.
