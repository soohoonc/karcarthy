`run!` creates one root execution tree. The root Agent may call Tools, invoke
known children, or create new Agents. Every descendant contributes to the same
usage counters, deadline, cancellation signal, and event log.

karcarthy calls the value carrying that execution context a `Runtime`. Runtime
is an API value passed to Agent bodies and Tools—not a provider, server, or
storage layer.

## Start a Run

```clojure
(def run
  (k/run! researcher
          {:question "How does ACP model selection work?"}
          {:context {:account-id "acct_123"
                     :db db}
           :limits {:model-calls 20
                    :deadline-ms 120000}
           :observe handle-event}))
```

The result contains everything needed to understand that execution:

```clojure
{:karcarthy/type :run
 :id "run_..."
 :status :completed
 :agent "researcher"
 :input {:question "..."}
 :output {:answer "..."}
 :usage {:model-calls 4
         :input-tokens 12000
         :output-tokens 900
         :generated-forms 0
         :duration-ms 4200.0}
 :events [...]
 :error nil}
```

Status is `:completed`, `:failed`, or `:cancelled`. Failed Runs retain their
events, usage, and structured error.

## Model-backed execution

An Agent without a Clojure body uses the built-in model/Tool loop:

1. validate Agent input and local context;
2. resolve model configuration, instructions, and Tools;
3. load messages from the optional Session;
4. call the model with messages and Tool schemas;
5. stream and record model deltas when supported;
6. validate, approve, and execute requested Tools;
7. append Tool results and call the model again;
8. validate the final output and update the Session.

The loop stops on valid output, failure, cancellation, deadline, or
`:max-turns`. `:stop-when` can reject a final response that does not satisfy an
application-specific condition.

## Instructions and local context

`:instructions` is model-visible. `:context` is application data and is never
added to the prompt automatically.

```clojure
(k/defagent account-agent
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions
   (fn [{:keys [context]}]
     (str "Assist account " (:account-id context) "."))
   :context map?
   :tools [lookup-account]
   :output string?})

(k/run! account-agent request
        {:context {:account-id "acct_123"
                   :db db}})
```

Agent bodies and Tools receive `rt` and read the local value with
`(k/context rt)`. Dynamic instructions and Tool resolvers receive an immutable
Runtime view.

## Conversation Sessions

Runs are independent unless a Session is supplied:

```clojure
(def session (k/memory-session))

(k/run! assistant "My preferred format is Markdown."
        {:session session})

(k/run! assistant "Reformat the previous answer."
        {:session session})
```

A Session stores normalized conversation items. `memory-session` is
thread-safe and process-local. Durable applications implement
`karcarthy.session/Session` with their own storage.

The Session belongs to root model conversations. Domain state, Tool side
effects, and Clojure stacks are not Session data.

## Child Agents

Inside an Agent body, `invoke!` performs a synchronous child call:

```clojure
(k/invoke! rt reviewer change)
```

`spawn!` starts a bounded parallel child and returns a handle:

```clojure
(let [children [(k/spawn! rt security-reviewer change)
                (k/spawn! rt api-reviewer change)]]
  (k/await-all! children))
```

`as-tool` and the zero-arity `agent` Tool also create child executions. All
four paths record parent/child lineage in the Run.

## Limits

```clojure
{:model-calls 100
 :input-tokens 1000000
 :output-tokens 100000
 :agent-depth 8
 :parallelism 6
 :generated-forms 20
 :deadline-ms 300000}
```

Limits apply to the complete root tree. A child may receive lower limits but
cannot raise the root grant.

- `agent-depth` bounds child nesting, including generated Agents.
- `generated-forms` counts generated-Agent compilation attempts.
- `parallelism` bounds concurrently executing children and timed Tools.
- token and model-call limits aggregate every descendant.

## Cancellation

Pass a Boolean, predicate, atom, or other dereferenceable value as `:cancel`:

```clojure
(def cancelled? (atom false))

(future
  (k/run! agent input {:cancel cancelled?}))

(reset! cancelled? true)
```

Cancellation is checked around Agent, model, Tool, and child-execution effects.
Cancelled Runs retain events and usage recorded before cancellation.

## Events and streaming

Use `:observe` for live events:

```clojure
(k/run! agent input
        {:observe
         (fn [{:keys [type delta] :as event}]
           (if (= type :model/text-delta)
             (do (print delta) (flush))
             (record-event! event)))})
```

Common families are:

```clojure
:run/started          :run/completed       :run/failed
:agent/started        :agent/completed     :agent/failed
:model/requested      :model/text-delta    :model/completed
:tool/started         :tool/completed      :tool/failed
:session/updated
:program/read         :program/expanded    :program/evaluated
```

Every event has a Run ID. Agent events add Agent ID, parent ID, and depth.
`k/events` returns the recorded vector from a Run or Runtime.
