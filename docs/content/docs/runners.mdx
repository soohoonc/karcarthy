karcarthy owns model calls, tool calls, child Agents, streaming events,
guardrails, approvals, limits, and cancellation. A provider integration only
translates model I/O; it does not execute tools or decide how an Agent proceeds.

## Minimal kernel

| Primitive | Responsibility |
| --- | --- |
| `agent` | Construct an Agent, or expose the same form to a model with `(agent)`. |
| `tool` | Construct a contracted capability backed by Clojure code. |
| `run!` | Create a root Runtime and execute an Agent. |
| `invoke!` | Execute a child Agent in an existing Runtime tree. |
| `compile-agent!` | Read, expand, check, evaluate, and return an Agent form. |
| `emit!` | Record an event. |

`defagent` and `deftool` name Agent and Tool values. `spawn!`, `await-all!`,
and `as-tool` are small composition helpers. There is no Runner protocol,
workflow interpreter, or placeholder handoff operation.

## Default model loop

An Agent without a custom Clojure body uses this loop:

1. Validate its input and local Runtime context.
2. Resolve its model, instructions, and tools.
3. Load root conversation history from an optional Session.
4. Send instructions, messages, tool schemas, and output schema to the model.
5. Record streaming deltas when the model adapter supports them.
6. If the model requests tools, validate, authorize, execute, and append their
   results before the next model turn.
7. If the model returns a final value, validate guardrails and the output
   contract, append accepted history to the Session, and complete.

The loop stops on completion, failure, cancellation, deadline, or
`:max-turns`. A `:stop-when` predicate can reject a premature final response.

## Agent configuration

```clojure
(k/agent
 {:name "researcher"
  :description "Research one question."
  :model {:transport :responses
          :id "gpt-5.6"
          :reasoning {:effort :high}}
  :instructions
  (fn [{:keys [context]}]
    (str "Research carefully for " (:user-id context) "."))
  :context ::research-context
  :input ::request
  :tools [web-search fetch-page]
  :output ::report
  :max-turns 20
  :stop-when accepted-final?
  :guardrails {:input [safe-input?]
               :output [grounded?]}
  :limits {:model-calls 30}})
```

`:model`, `:instructions`, and `:tools` may be functions of the immutable
Runtime view. Instructions must resolve to a string. A custom-body Agent may
omit `:model` and `:instructions` entirely.

## Instructions and context

The names follow the established split used by current Agent SDKs:

- `:instructions` is model-visible policy and prompt text;
- `:context` is local dependency injection and is never exposed to the model
  automatically.

Context may contain database handles, authenticated user data, clocks, or
helper functions. The Agent's `:context` value is its contract; the concrete
value enters through `run!` or a child invocation:

```clojure
(k/run! agent input
        {:context {:user-id "u_123"
                   :db db
                   :http http-client}})
```

Programs and Tools read it with `(k/context rt)`. Dynamic instructions receive
it in the Runtime view. `prompt`, `prompt-file`, and `system-prompt` compose
instruction text without inferring a directory or hidden policy source.

## Sessions

Runs are stateless unless the caller supplies a Session:

```clojure
(def session (k/memory-session))

(k/run! researcher first-request {:session session})
(k/run! researcher next-request {:session session})
```

A Session is only an ordered conversation store. Its protocol consists of
`session-id`, `get-items`, `add-items!`, `pop-item!`, and `clear-session!`.
`memory-session` is thread-safe and process-local. Applications that need
durability implement `karcarthy.session/Session` with their normal database.

The Session belongs to the root model-backed Agent. Child Agents do not
silently share it, and custom Clojure programs remain responsible for their own
domain data. This is not a workflow checkpoint: Clojure stacks, closures, and
arbitrary effects are not serialized or resumed.

Provider continuation identifiers are an internal optimization within one
model/tool loop. Portable Session history remains normalized messages.

## Tools and Agent composition

A Tool declares a model-facing name and description, input/output contracts,
a Clojure implementation, and optional approval, guardrail, timeout, retry,
enablement, and model-output projection policies.

Zero-arity `(agent)` is itself a Tool. It accepts one ordinary `(agent ...)`
source form and input, then reads, expands, checks, evaluates, and invokes it in
the current run tree.

`as-tool` exposes a known Agent to a model-backed parent while the parent keeps
control. Arbitrary orchestration uses `invoke!`, `spawn!`, and ordinary Clojure
control flow. karcarthy does not claim handoff semantics until it can define
actual control and conversation transfer.

## Local tools, MCP, and hosted tools

`local-tools` returns `read`, `write`, `edit`, `bash`, and `search` Tools rooted
at a caller-selected directory. File operations reject paths outside that
root. `bash` is bounded but is not a sandbox.

`mcp-tools` adapts discovered MCP definitions into the same Tool values. The
current MCP client implements the `2025-11-25` stdio lifecycle. Provider-hosted
capabilities such as `responses-web-search` are lowered only by their matching
model adapter and are never pretended to be local functions.

## Streaming

Streaming model adapters emit normalized text and tool-call deltas before
returning an authoritative final response. The Runtime records
`:model/text-delta` and `:model/tool-call-delta` events. The Responses adapter
uses typed SSE events; ACP forwards text deltas as `agent_message_chunk`
updates.

Model-adapter construction is internal. Applications select the built-in
Responses adapter with `:transport :responses`; tests use `fake-model`.

## Limits

Limits apply to the whole root run tree unless a child receives a smaller
grant:

```clojure
{:model-calls 100
 :input-tokens 1000000
 :output-tokens 100000
 :agent-depth 8
 :parallelism 6
 :generated-forms 20
 :deadline-ms 300000}
```

All descendants debit the same usage counters. A child may lower inherited
ceilings but cannot raise them.

## Approval

Approval is evaluated after Tool-input validation and before the effect. The
Run's `:approval` handler receives the Runtime view, Tool, and arguments.
Missing or denied approval fails before execution. Durable approval suspension
and resumption remain roadmap work.

## Events

`run!` retains emitted events and can send each event to an `:observe`
callback. Stable families include:

```clojure
:run/started          :run/completed       :run/failed
:agent/started        :agent/completed     :agent/failed
:model/requested      :model/text-delta    :model/completed
:tool/started         :tool/completed      :tool/failed
:session/updated
:program/read         :program/expanded    :program/evaluated
```

Events carry a Run ID and Agent parent/child identifiers where applicable.
There is no separate hook system or synthetic trace ID.

## ACP boundary

`serve-acp!` exposes an Agent or per-session Agent factory over ACP v1 stdio.
The server maps ACP prompts to `run!`, creates one in-memory Session per ACP
session, forwards cancellation and permission requests, publishes Tool events,
and merges session-provided MCP Tools. It advertises selectable models with ACP
session configuration options and returns aggregate Run token counts in the
prompt response when every model call reported usage. ACP currently marks that
prompt-usage field unstable. karcarthy does not synthesize `usage_update`
notifications: those describe current context occupancy and context-window
size, which a generic model transport may not know. The current
`session/set_config_option` method is canonical; a narrow `session/set_model`
alias supports Harbor's current adapter.

Harbor's generic ACP adapter stores `acp-events.jsonl` and `acp-summary.json`,
then converts those protocol records to a validated `trajectory.json` in
[ATIF](https://harborframework.com/docs/agents/trajectory-format). That
conversion belongs to Harbor. karcarthy does not add a second experiment-file
format or send source/expanded Agent forms as task input. Those forms and their
program events remain observable inside the karcarthy Run when a research
caller chooses to retain them.

ACP is an external client boundary, not the harness underneath karcarthy.
