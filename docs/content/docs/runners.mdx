---
title: Execution Runners
description: Choose the execution backend for a run.
---

A runner runs one agent to completion. Karcarthy uses that same runner for each
agent it encounters while interpreting a workflow.

```clojure
(k/run {:runner runner
        :workflow workflow
        :input input})
```

Agents describe the work:

```clojure
(k/defagent reviewer
  {:instructions "Review the input and return concrete issues."
   :model "sonnet"
   :tools ["Read" "Grep"]})
```

The caller chooses the runner:

```clojure
(k/run {:runner (k/mock-runner)
        :workflow reviewer
        :input "check this design"})

(k/run {:runner (k/claude-cli-runner {})
        :workflow reviewer
        :input "check this design"})

(k/run {:runner (k/acp-runner {:command ["my-acp-agent"]})
        :workflow reviewer
        :input "check this design"})
```

Agents do not carry a backend. If the backend changes at runtime, pass a
different runner value.

Runner configuration belongs in the runner value, agent fields, or run `:options`.
Do not hide execution configuration inside `:input`; `:input` is the task or
state being worked on.

## Runtime Selection

Keep runner choice in application code:

```clojure
(defn runner-for [env]
  (case env
    :test (k/mock-runner)
    :local (k/process-runner "cat")
    :live (k/claude-cli-runner {})))

(defn review [env input]
  (k/run {:runner (runner-for env)
          :workflow reviewer
          :input input}))
```

This keeps the agent data stable. The same agent can run through a mock in
tests, Claude in a live run, or an ACP-compatible process in another
environment.

## Mock Runner

Use `mock-runner` for tests, docs, and local workflow design:

```clojure
(k/run {:runner (k/mock-runner)
        :workflow (k/agent {:name "echo" :instructions "Echo the input."})
        :input "hi"})
```

Output:

```clojure
{:karcarthy/type :result
 :ok? true
 :agent "echo"
 :text "[echo] hi"
 :raw {:runner :mock}}
```

Pass a response function when tests need deterministic named behavior:

```clojure
(def test-runner
  (k/mock-runner
   (fn [{:keys [agent prompt]}]
     (case (:name agent)
       "classifier" "{:route :launch}"
       "brief-writer" "Decision: staged beta."
       (str "[" (:name agent) "] " prompt)))))
```

## Function Runner

Use `fn-runner` when a Clojure function is the backend for agent calls:

```clojure
(def uppercase
  (k/fn-runner clojure.string/upper-case))

(k/run {:runner uppercase
        :workflow (k/agent {:name "shout" :instructions "Uppercase the input."})
        :input "quiet please"})
```

By default the function receives the input string. Use `:call? true` when it
needs the full agent call map, `{:agent agent :input input :options options}`:

```clojure
(def local-runner
  (k/fn-runner
   (fn [{:keys [agent input]}]
     (str (:name agent) ": " input))
   {:call? true}))
```

Use `fn-runner` when the function is the backend. Use `step` when the function
is just one local transform inside a larger workflow.

## Process Runner

Use `process-runner` to wrap one fixed command:

```clojure
(def upper
  (k/process-runner ["tr" "a-z" "A-Z"]))

(k/run {:runner upper
        :workflow (k/agent {:name "shout" :instructions "Uppercase the input."})
        :input "quiet please"})
```

The command receives the input on stdin and returns text on stdout. String
commands go through the configured shell:

```clojure
(k/run {:runner (k/process-runner "tr a-z A-Z")
        :workflow (k/agent {:name "shout" :instructions "Uppercase the input."})
        :input "quiet please"})
```

Different commands mean different runner values. Choose the runner before
calling `run`.

## Claude CLI Runner

Use `claude-cli-runner` when each agent call should run through `claude -p`:

```clojure
(def claude
  (k/claude-cli-runner {:model "haiku"
                        :max-turns 4}))

(k/run {:runner claude
        :workflow (k/agent {:name "reviewer"
                            :instructions "Find one launch risk and one mitigation."
                            :tools ["mcp__feedback__search"]})
        :input "enterprise SSO beta"})
```

`tools` is an allowlist passed to runners that support tool selection. The tool
server, MCP config, and credentials must already exist in that runner's
environment.

## ACP Runner

Use `acp-runner` for any Agent Client Protocol agent process that speaks stdio
JSON-RPC:

```clojure
(def acp
  (k/acp-runner
   {:command ["my-acp-agent"]
    :cwd "."
    :mcp-servers []
    :permission-policy :reject}))

(k/run {:runner acp
        :workflow (k/agent {:name "reviewer"
                            :instructions "Review the current branch."
                            :model "sonnet"})
        :input "Check for correctness and missing tests."})
```

The ACP agent is the subprocess. Karcarthy folds the agent spec's
`:instructions` into the `session/prompt` text and sends `:model` through ACP
session config options when the agent advertises a `model` option.

## OpenAI Agents SDK Runner

`openai-agents-runner` delegates single-agent calls to the OpenAI Agents SDK
bridge:

```clojure
(k/run {:runner (k/openai-agents-runner {})
        :workflow (k/agent {:name "triage"
                            :instructions "Route to refund, sales, or support."})
        :input "I was charged twice."})
```

## Mixed Backends

Most workflows should use one runner for the whole run. If you truly need mixed
backends, make that dispatch visible in a custom runner:

```clojure
(require '[karcarthy.core :as core])

(def hybrid
  (let [mock (k/mock-runner)
        acp  (k/acp-runner {:command ["my-acp-agent"]})]
    (reify core/Runner
      (-run [_ agent input options]
        (if (= "external-reviewer" (:name agent))
          (core/run-agent acp agent input options)
          (core/run-agent mock agent input options))))))

(k/run {:runner hybrid
        :workflow workflow
        :input "review this release"})
```

That is ordinary host code, not workflow data. The workflow remains a visible
description of the work; the runner remains the execution boundary.

## Subagents

Use `subagent` when the selected runner supports native delegation inside one
agent session:

```clojure
(def security-reviewer
  (k/subagent "security-reviewer"
    "Use for auth, secrets, injection, permission, and data exposure review."
    "Review like a security owner. Return concrete findings with file refs."
    :tools ["Read" "Grep" "Glob"]
    :model "sonnet"))

(def claude
  (k/claude-cli-runner {:subagents [security-reviewer]}))

(k/run {:runner claude
        :workflow (k/agent {:name "review-coordinator"
                            :instructions "Review this branch. Use the security-reviewer subagent where useful."
                            :tools ["Agent" "Read" "Grep" "Glob"]})
        :input "Review current changes."})
```

Subagents are runner configuration, not workflow nodes. A workflow branch is
Karcarthy orchestration; a subagent is provider-native delegation inside one
runner call.
