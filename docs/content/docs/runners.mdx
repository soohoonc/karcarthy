---
title: Use Runners
description: Connect workflow data to mock, function, process, CLI, and SDK-backed agent systems.
---

karcarthy coordinates workflows. A runner runs one agent to completion. That
boundary is the core design: workflows stay portable data, while model calls,
tool calls, sessions, streaming, and provider details stay inside the selected
agent system.

## Runner model

```clojure
(k/run runner workflow input)
```

`runner` can be one runner for every agent:

```clojure
(k/run (k/mock-runner)
       (k/pipe researcher summarizer)
       "map the agent framework space")
```

or a registry selected per agent:

```clojure
(def runners
  {:default (k/mock-runner)
   :claude  (k/claude-cli-runner {})
   :acp     (k/acp-runner {:command ["my-acp-agent"]})})

(k/run runners
       (k/pipe
        (k/agent {:name "planner"
                  :instructions "Plan the rollout."
                  :runner :claude})
        (k/agent {:name "checker"
                  :instructions "Check for missing owners."
                  :runner :acp}))
       "enterprise SSO beta")
```

Agent-level `:runner` is a registry key. It does not change the workflow
shape.

This mirrors the Agent Client Protocol split: a client starts or resumes a
session, sends prompt turns, receives updates, and may provide filesystem,
terminal, permission, and MCP surfaces. Karcarthy owns the workflow value above
that boundary. A selected runner owns the provider-native or protocol-native
agent session.

## Subagents

Use `subagent` when the selected runner supports native delegation inside one
agent session:

```clojure
(def security-reviewer
  (k/subagent "security-reviewer"
    "Use for auth, secrets, injection, permission, and data exposure review."
    "Review like a security owner. Return concrete findings with file refs."
    :tools ["Read" "Grep" "Glob"]
    :model "sonnet"
    :effort :high))

(def runner
  (k/claude-cli-runner
   {:subagents [security-reviewer]}))

(k/run runner
       (k/agent {:name "review-coordinator"
                 :instructions "Review this branch. Use the security-reviewer subagent where useful."
                 :tools ["Agent" "Read" "Grep" "Glob"]})
       "Review current changes.")
```

Subagents are runner config, not workflow nodes. A workflow branch is visible
karcarthy orchestration; a subagent is provider-native delegation used by a
single runner call.

| karcarthy field | Claude Code | Codex | OpenAI Agents SDK |
| --- | --- | --- | --- |
| `:name` | subagent name | custom agent name | handoff agent name |
| `:description` | delegation description | custom agent description | handoff description |
| `:instructions` | `prompt` | `developer_instructions` | `instructions` |
| `:tools` | tool allowlist | inherited/configured tools | bridge-specific, not yet exposed |
| `:mcp-servers` | subagent MCP servers | `mcp_servers` config | bridge-specific, not yet exposed |
| `:model` | subagent model | custom agent model | handoff agent model |
| `:effort` / `:reasoning-effort` | effort | `model_reasoning_effort` | bridge-specific |
| `:background?` / `:isolation` | background/worktree controls | sandbox/worktree config | not applicable |

`claude-cli-runner` lowers subagents to Claude Code `--agents` JSON. The main
agent still needs access to the `Agent` tool, usually through its `:tools`
allowlist.

`openai-agents-runner` lowers subagents to Agents SDK handoffs. The current
bridge supports names, instructions, handoff descriptions, and models.

Codex custom agents are file-backed TOML definitions under `.codex/agents/` or
`~/.codex/agents/`, so they belong in a dedicated Codex runner rather than
`process-runner`. The `karcarthy.runner.codex/subagent->config` helper lowers a
`subagent` to Codex custom-agent config keys without writing files.

## Mock runner

Use the mock runner for tests, docs, and local workflow design:

```clojure
(k/run (k/mock-runner)
       (k/agent {:name "echo" :instructions "Echo the input."})
       "hi")
```

Output:

```clojure
{:karcarthy/type :result
 :ok? true
 :agent "echo"
 :text "[echo] hi"
 :raw {:runner :mock}}
```

Pass a response function when tests need deterministic named behavior:

```clojure
(k/mock-runner
 (fn [{:keys [agent prompt]}]
   (case (:name agent)
     "classifier" "launch"
     (str "[" (:name agent) "] " prompt))))
```

## Function runner

Use `fn-runner` when a Clojure function is the runner for agent specs:

```clojure
(def uppercase
  (k/fn-runner clojure.string/upper-case))

(k/run uppercase
       (k/agent {:name "shout" :instructions "Uppercase the input."})
       "quiet please")
```

By default, the function receives only the input string. If it needs the agent
spec or run options, opt into context:

```clojure
(k/fn-runner
 (fn [{:keys [agent input opts]}]
   (str (:name agent) ": " input))
 {:context? true})
```

Use `fn-runner` for a Clojure implementation of the runner boundary. Use
`step` when the function is just a local transform inside a workflow.

## Claude CLI runner

Use Claude CLI when each agent call should run through `claude`:

```clojure
(k/run (k/claude-cli-runner {:model "haiku"
                      :max-turns 4})
       (k/agent {:name "reviewer"
                 :instructions "Find one launch risk and one mitigation."
                 :tools ["mcp__feedback__search"]})
       "enterprise SSO beta")
```

`tools` is an allowlist passed to runners that support tool selection. The
tool server, MCP config, and credentials must already exist in the selected
agent system.

## Process runner

Use `process-runner` to wrap a fixed local command. An argv vector is executed
directly:

```clojure
(def local-model
  (k/process-runner
   ["./bin/local-agent"]
   {:timeout-ms 120000}))

(k/run local-model
       (k/agent {:name "writer" :instructions "Write the final answer."})
       "Summarize the rollout risk.")
```

The command receives the input on stdin and returns text on stdout. Use a string
when you want shell parsing, pipes, redirects, or inline scripts:

```clojure
(k/run (k/process-runner "tr a-z A-Z")
       (k/agent {:name "shout" :instructions "Uppercase the input."})
       "quiet please")
```

`process-runner` does not derive commands from `k/agent`. Select different
commands with the runner registry:

```clojure
(k/run {:default (k/process-runner ["cat"])
        :upper   (k/process-runner ["tr" "a-z" "A-Z"])}
       (k/agent {:name "shout"
                 :instructions "Uppercase input."
                 :runner :upper})
       "quiet please")
```

This keeps process execution as runner configuration. Dedicated CLI bridges
such as `claude-cli-runner` may still be agent-aware because they translate an
agent spec into that CLI's prompt, flags, tools, and session behavior.

## ACP runner

Use `acp-runner` for any Agent Client Protocol agent process that speaks stdio
JSON-RPC:

```clojure
(def acp
  (k/acp-runner
   {:command ["my-acp-agent"]
    :cwd "."
    :mcp-servers []
    :permission-policy :reject}))

(k/run acp
       (k/agent {:name "reviewer"
                 :instructions "Review the current branch."
                 :model "sonnet"})
       "Check for correctness and missing tests.")
```

The ACP agent is the subprocess. Karcarthy folds the agent spec's
`:instructions` into the `session/prompt` text and sends `:model` through ACP
session config options when the agent advertises a `model` option. Other
runner-specific controls can go through `:config-options` or a custom
`:request-handler`.

## OpenAI Agents SDK bridge

`openai-agents-runner` delegates single-agent calls to the OpenAI Agents SDK
bridge. Keep karcarthy responsible for orchestration and keep SDK-specific
agent-loop behavior in that bridge.

```clojure
(k/run (k/openai-agents-runner {})
       (k/agent {:name "triage"
                 :instructions "Route to refund, sales, or support."})
       "I was charged twice.")
```

## Tools

Tools are not provisioned by karcarthy. They are names the selected runner may
use to restrict what an agent system can call.

| Field | Meaning |
| --- | --- |
| `:tools ["mcp__feedback__search"]` | Allow this agent to use a tool already configured in the runner's environment. |
| No `:tools` | Let the runner default decide, or use no tools for mock/offline runs. |
| Unknown tool name | Runner-specific failure or ignored by runners that do not use tools. |

This keeps orchestration portable. The same workflow can run offline with
`mock-runner`, through Claude CLI with MCP tools, or through a process runner
without changing the graph.

## Failure behavior

Top-level `run` fails fast for malformed workflows. Composite workflow nodes
isolate child failures by returning not-ok result maps, so one failed branch
does not crash an entire `branch`.

```clojure
{:karcarthy/type :result
 :ok? false
 :error "..."
 :exception "..."}
```

Use this result shape for retries, audit logs, and UI status displays.
