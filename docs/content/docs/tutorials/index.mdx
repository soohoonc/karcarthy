The tutorials build one realistic workflow: a launch-readiness assistant.

Given a product-change prompt, it:

1. classifies the request as launch or incident,
2. runs product, engineering, security, and support reviewers,
3. synthesizes their notes into a launch brief,
4. asks a critic to accept or request revisions.

The examples run offline by default with deterministic mock responses.

## Core Shape

The Clojure workflow is small:

```clojure
(def reviewers
  [product-reviewer engineering-reviewer security-reviewer support-reviewer])

(def review-packet
  (k/branch reviewers :max-concurrency 4))

(def launch-brief
  (k/revise
   (k/pipe review-packet brief-writer)
   critic
   :max-rounds 2))

(def workflow
  (k/route classifier
           {:launch launch-brief
            :incident incident-responder}
           :default launch-brief))
```

Then the caller chooses a runner:

```clojure
(k/run (k/mock-runner responses)
       workflow
       "Prepare an enterprise SSO beta launch review.")
```

The workflow data does not contain a runner selector. Use a mock, Claude, ACP,
OpenAI, or process runner by passing a different runner to `run`.

## Run The Examples

```bash
clojure -M -e '(load-file "examples/clojure/launch.clj")'
python3 examples/python/launch.py
npx tsx examples/typescript/launch.ts
```

Expected final brief:

```text
Decision: launch to a staged enterprise beta cohort.
Top risks: SSO latency, unverified audit logs, and missing rollback messaging.
Next actions: confirm audit-log coverage, assign rollback owner, publish support docs, and monitor p95 auth latency during rollout.
```
