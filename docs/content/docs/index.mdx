karcarthy is a small agent harness for Clojure. It runs model-backed Agents,
executes their Tools, streams their output, and returns a structured record of
each Run.

Agents and Tools are ordinary Clojure values. Define them in a namespace, run
them from a REPL or an application, and use normal Clojure when you need to
compose them. The library namespace is `karcarthy`.

## Define and run an Agent

```clojure
(ns example
  (:require [karcarthy :as k]))

(k/defagent assistant
  {:model {:transport :responses
           :id "gpt-5.6"}
   :instructions "Answer clearly and concisely."
   :output string?})

(def run
  (k/run! assistant "Explain continuation-passing style."))

(:output run)
(select-keys run [:status :usage])
```

That is the basic programming model: `agent` constructs an Agent, `defagent`
gives one a name, and `run!` starts a Run. A model-backed Agent uses karcarthy's
model and Tool loop; an Agent can also have a Clojure body when its coordination
should be deterministic.

The [Quickstart](/docs/quickstart) covers installation, a typed Tool, REPL use,
conversation history, and recursive Agent creation.

## What the harness provides

- a streaming model and Tool loop;
- typed local Tools and provider-hosted Tools;
- MCP Tool discovery;
- optional Sessions for conversation history;
- limits, cancellation, approvals, events, and structured Run results;
- ACP serving for editors and evaluation systems such as Harbor;
- an `agent` Tool that lets a model define and run another Agent.

karcarthy does not introduce a workflow language. Sequential, conditional, and
parallel composition use ordinary Clojure and the same `run!` function used for
any other Agent.

## Agents can be Tools

Known Agents can be exposed to a model with `as-tool`. karcarthy also provides a
zero-argument `(agent)` form that creates a model-facing Tool:

```clojure
(k/defagent architect
  {:model {:transport :responses
           :id "gpt-5.6"}
   :instructions
   "Complete the task. Create a focused Agent when specialization helps."
   :tools [(k/agent)]
   :output string?})

(k/run! architect "Review this authentication design.")
```

When the model calls that Tool, it supplies an `(agent ...)` form and an input.
karcarthy checks the form, evaluates it as Clojure, verifies that it produced an
Agent, and runs it inside the current Run. The generated Agent uses the same
configuration model, execution path, limits, and events as one written by a
developer.

Read [Agents](/docs/agents) for Agent configuration, Clojure composition,
Agents as Tools, generated Agents, and recursive use.

## Why homoiconicity

karcarthy is inspired by Lisp's simplest idea: code is data. An Agent definition
is both executable Clojure and an inspectable Clojure form. A developer can
write it, a macro can transform it, or a model can produce it through the
`agent` Tool without crossing into a separate workflow representation.

```text
developer or model → Clojure form → Agent → Run
```

Generated Agents retain their source and macroexpanded forms, so experiments
can inspect and compare the behavior that actually ran. Homoiconicity is the
reason model-authored orchestration fits the normal programming model; it is
not another API layered on top of it.
