karcarthy is a small Clojure agent harness. An Agent combines a model,
instructions, and Tools. `run!` gives the Agent a task and returns a complete
record of what happened.

karcarthy owns the model and Tool loop. Model transports only translate model
I/O, so the same Agent semantics apply to local Tools, hosted Tools, and MCP.

## A first Agent

```clojure
(require '[karcarthy :as k])

(k/defagent assistant
  {:model {:transport :responses
           :id "gpt-5.6"}
   :instructions "Answer clearly and concisely."
   :output string?})

(def run
  (k/run! assistant "Explain continuation-passing style."))

(:status run) ;=> :completed
(:output run) ;=> "..."
```

The input to a model-backed Agent is normally a user prompt. A Run contains the
output, usage, events, and structured failure information.

## The library in four values

| Value | Purpose |
| --- | --- |
| Agent | A model and Tool loop, or a Clojure function body. |
| Tool | A typed capability that a model may call. |
| Run | The result of one `run!` call. |
| Session | Optional conversation history shared across Runs. |

Agents and Tools are ordinary Clojure values. Applications use normal Clojure
for sequencing, branching, concurrency, and error handling.

## Agents as Tools

A known Agent can be exposed with `as-tool`. karcarthy also provides a Tool
that lets a model define a new Agent during a Run:

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions
   "Complete the task. Create a focused Agent when specialization helps."
   :tools [(k/agent)]
   :output string?})
```

The zero-argument `(k/agent)` form returns a Tool named `agent`. A model calls
it with a Clojure Agent form and an input. karcarthy checks the form, evaluates
it, and runs the resulting Agent inside the current Run.

## Why Clojure forms matter

An Agent definition is executable code and inspectable data. A developer can
write it, a macro can transform it, or a model can produce it through the
`agent` Tool. Authored and generated Agents therefore use one representation
instead of separate code and workflow formats.

That is the homoiconic motivation behind karcarthy. It enables model-authored
orchestration without adding a workflow language.
