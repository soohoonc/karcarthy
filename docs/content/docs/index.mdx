karcarthy coordinates multi-agent workflows while keeping the whole plan as
plain data. You describe agents as maps and workflows as EDN values. A runner —
Claude, Codex, OpenAI, ACP, a process command, or the offline mock — owns the
model/tool loop underneath; karcarthy owns the coordination value above it.

```clojure
(require '[karcarthy :as k])

(k/defagent researcher
  {:instructions "Research the question and cite sources."})

(k/defagent summarizer
  {:instructions "Summarize the findings in one sentence."})

(k/run {:runner (k/mock-runner)
        :workflow (k/pipe researcher summarizer)
        :input "what is a monad?"})
```

## Everything is data

Most agent frameworks make the workflow a host-language object. In karcarthy
the workflow is a value. You inspect it, diff it, store it, generate it, and
rewrite it:

```clojure
{:karcarthy/type :pipe
 :steps [{:karcarthy/type :agent
          :name "researcher"
          :instructions "Research the question and cite sources."}
         {:karcarthy/type :agent
          :name "summarizer"
          :instructions "Summarize the findings in one sentence."}]}
```

That value can live as EDN, cross a JSON boundary, or be produced by an agent
and parsed without `eval`. You can also rewrite it before execution:

```clojure
(->> workflow
     (k/configure {:model "claude-sonnet-4"
                   :instructions/suffix "State assumptions before final answer."}))
```

## The inner loop belongs to runners

karcarthy does not implement model calls or tool execution. The runner you pass
to `run` owns that loop. karcarthy owns the visible coordination value and the
small interpreter that walks it. Swap the runner and the workflow data does not
change.

## Why Clojure

karcarthy takes Lisp homoiconicity and applies it in a practical way: the agent
program is data. Agents are maps, workflows are EDN trees, and constructors
such as `pipe`, `branch`, and `revise` return values you can print, store,
diff, walk, rewrite, validate, or have another agent generate.

That does not mean karcarthy evaluates arbitrary agent-written code. Dynamic
workflows parse EDN operations with `clojure.edn`, validate them, and interpret
them through the same workflow interpreter. The homoiconic move is that the
language for describing coordination is available as ordinary data, not hidden
inside an opaque object graph.

Clojure fits because EDN is native, maps and vectors are the default building
blocks, structural rewrites are ordinary code, macros such as `defagent` add
ergonomic entry points without changing the data model, and multimethods
dispatch cleanly on tagged workflow nodes.

## Core surfaces

| Surface | Description |
| --- | --- |
| Agents | Plain maps tagged with `:karcarthy/type :agent`; usually defined with `defagent`. |
| Workflows | `pipe`, `branch`, `route`, `revise`, `delegate`, `reduce`, `continue`, `step`, and `dynamic` (`step` is the host-local Clojure escape hatch; `dynamic` is experimental). |
| Subagents | Runner-native delegation config built with `subagent`/`defsubagent`, passed to runners that support native delegation. Distinct from workflow nodes. See [Runners](/docs/runners#subagents). |
| Rewrites | `agents`, `over`, and `configure`. |
| Runners | Mock, Clojure function, process command, ACP, Claude, Codex, and OpenAI. |
| CLI | `bin/karcarthy agent`, `run`, `demo`, and `json`. |

## Current status

karcarthy is early and not yet published as a stable release. The docs treat
the small public vocabulary as the product surface and keep implementation
internals out of the main path.
