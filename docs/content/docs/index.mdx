karcarthy is a small agent harness written in Clojure. It owns the model/tool
loop, and its orchestration language is Clojure itself.

```clojure
(require '[karcarthy :as k])

(k/defagent researcher
  {:model {:transport :responses :id "gpt-5.6"}
   :context "Research carefully and cite evidence."
   :tools [web-search]
   :output ::report})

(k/run! researcher
        {:question "Why are Lisp programs represented as lists?"}
        {:environment {:user-id "u_123"}})
```

The native Agent/Tool kernel, context/environment split, explicit conversation
state, streaming model loop, child execution, generated-form evaluation,
contracts, limits, events, Responses HTTP/SSE transport, local tools, stdio MCP
client, and ACP v1 server are implemented. Crash-safe suspension remains on the
roadmap.

## The premise

An agent is an executable Clojure value. It carries model configuration,
model-visible context, an optional local-environment contract, tools, output
contracts, loop policy, and optionally an arbitrary Clojure body.

```clojure
(k/defagent review-team
  {:input ::review-request
   :output ::review}
  [rt request]
  (let [draft   (k/invoke! rt writer request)
        reviews (k/await-all!
                 (for [reviewer [security-reviewer api-reviewer]]
                   (k/spawn! rt reviewer draft)))]
    (k/invoke! rt editor {:draft draft :reviews reviews})))
```

There is no workflow data language above Clojure. Sequencing is `let`, routing
is `if` or `case`, parallelism is a runtime function, and recursion is ordinary
function or agent invocation.

## Dynamic behavior is already built in

karcarthy does not have a separate `dynamic` feature. A model may write an
`(agent ...)` form, after which the harness reads, expands, checks, evaluates,
and invokes it. The form is recursive: zero-arity `agent` is the model-facing
Tool that accepts another configured `agent` form.

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :context "Design and run the smallest useful child Agent."
   :tools [(k/agent)]})
```

Evaluation causes normal macroexpansion. The resulting child is an ordinary
agent and may construct more agents subject to the run's depth, model-call,
token, concurrency, and time budgets.

## The four values

| Value | Meaning |
| --- | --- |
| Agent | Configuration plus an executable body. Without a custom body, the harness supplies its model/tool loop. |
| Tool | A described, contracted Clojure function the model may invoke. |
| Runtime | Local environment, budgets, cancellation, model access, state checkpoints, and observation for one run tree. |
| Run | Status, typed output, conversation state, usage, events, and the source/expanded forms involved in execution. |

Agent and tool values retain their source forms. Programs are therefore
executable behavior and inspectable research artifacts at the same time.

## The harness owns the loop

For a model-backed agent, karcarthy:

1. resolves tools and model options from the local Runtime environment;
2. assembles `{:system ... :messages [...]}` context for the current turn;
3. calls the configured model and emits streaming deltas;
4. validates and executes requested tools;
5. checkpoints explicit conversation state and enforces budgets;
6. repeats until a typed output, failure, cancellation, deadline, or stop condition;
7. emits a trace for every decision and effect.

The model-provider boundary only performs model I/O. It is not a runner and
does not own tools, delegation, conversation state, or orchestration.

## Engineering and research

Agents retain original and macroexpanded forms; generated programs also place
both forms in the trace. Every run records model and tool events, child-agent
lineage, token usage, latency, approvals, failures, and final output. That makes
the same system suitable for applications and behavioral experiments.

Harbor can evaluate the stable ACP surface. Experiments may compare quality,
cost, failure modes, trace structure, or invariants; they do not need to claim
that a generated program improves anything.

## A small toolset, not a second harness

`local-tools` returns `read`, `write`, `edit`, `bash`, and `search` as
ordinary Tool values. `system-prompt` reads the packaged Markdown prompt;
`prompt-file` and `prompt` let the application explicitly compose any other
prompt sources. This follows the small-kernel pattern used by
[Pi](https://github.com/badlogic/pi-mono/tree/main/packages/coding-agent): keep
the loop fixed and make capabilities explicit. There is no special coding
Agent type. Hosted web search and MCP tools use the same Tool boundary; ACP
remains outside the loop.
