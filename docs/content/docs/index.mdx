karcarthy is an in-process agent harness for Clojure. It provides a model/Tool
loop, typed Tools, conversation Sessions, streaming events, child execution,
MCP, and an ACP server.

Its distinguishing capability is recursive Agent creation: give a model the
built-in `agent` Tool and it can submit a new Clojure Agent definition for the
harness to check, evaluate, and run.

## Define and run an Agent

```clojure
(require '[karcarthy :as k])

(k/defagent researcher
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Answer accurately. Use web search for current facts."
   :tools [(k/responses-web-search)]
   :output string?})

(def run
  (k/run! researcher "What changed in the latest Java release?"))

(:status run) ;=> :completed
(:output run) ;=> "..."
```

An Agent definition contains the same pieces found in other harnesses:

| Concept | karcarthy |
| --- | --- |
| Model and inference options | `:model` |
| System instructions | `:instructions` |
| Callable capabilities | `:tools` |
| Dependency injection | local `:context` |
| Conversation history | optional `Session` |
| Input and output validation | Clojure specs, predicates, or JSON Schema |
| Lifecycle and telemetry | `Run` values and events |

`run!` returns a complete Run value containing status, output, usage, events,
and structured failure data.

## The `agent` form

The same form has three related uses:

```clojure
;; Construct an Agent value.
(def reviewer
  (k/agent {:name "reviewer"
            :model {:transport :responses :id "gpt-5.6"}
            :instructions "Review the proposed change."
            :output string?}))

;; Define a named var containing an Agent.
(k/defagent reviewer
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Review the proposed change."
   :output string?})

;; Return the Tool that lets a model create and run an Agent.
(k/agent)
```

`defagent` is definition syntax: it is the convenient equivalent of defining a
var whose value is created by `agent`, and it supplies the Agent name from the
var name. It does not create a different kind of Agent.

## Let a model create an Agent

Add zero-arity `(k/agent)` to a model-backed Agent's Tool set:

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions
   "Solve the task. Create a specialized child Agent when that is useful."
   :tools [(k/agent)]
   :output string?})
```

The model can now call a Tool named `agent` with a Clojure definition and an
input value. karcarthy:

1. reads exactly one Clojure form;
2. macroexpands and checks it;
3. evaluates it in the configured namespace;
4. verifies that the result is an Agent;
5. invokes the child inside the current Run.

The child shares the Run's budgets, cancellation, approvals, and event tree.
It may receive the same `agent` Tool and create another child, up to configured
depth and resource limits. See [Agents](/docs/agents) for the complete model.

## Clojure composition

Agents can also have a Clojure body. Use this for deterministic coordination
around model-backed children:

```clojure
(k/defagent review-team
  {:input map?
   :output map?}
  [rt change]
  (let [reviews (k/await-all!
                 [(k/spawn! rt security-reviewer change)
                  (k/spawn! rt api-reviewer change)])]
    (k/invoke! rt editor {:change change :reviews reviews})))
```

`invoke!`, `spawn!`, and `await-all!` preserve parent/child lineage and shared
limits. Branching and iteration are ordinary Clojure.

## Included integrations

- OpenAI Responses-compatible HTTP/SSE model transport
- local `read`, `write`, `edit`, `bash`, and `search` Tools
- provider-hosted web search
- MCP `2025-11-25` over stdio
- ACP v1 serving for editors and evaluation harnesses such as Harbor
