[Harbor](https://www.harborframework.com/) runs Agents against tasks in
isolated environments, invokes verifiers, and stores rewards and trajectories.
karcarthy integrates at Harbor's generic ACP boundary:

```text
Harbor task
  -> packaged karcarthy ACP process
  -> executable Clojure Agent program
  -> repository Tools, model calls, and nested Agents
  -> Harbor verifier
  -> reward, logs, and ATIF trajectory
```

Harbor owns task execution, environments, rewards, experiment storage, and
ACP-to-ATIF conversion. karcarthy owns the Agent program and model/Tool loop.
There is no shared workflow DSL.

## From demonstration to search

The Harbor example does not compare two hand-written labels such as `direct`
and `specialist`. It searches complete Clojure Agent forms.

The initial [`seed.clj`](https://github.com/soohoonc/karcarthy/blob/main/examples/harbor/seed.clj)
is a normal Coding Agent. An optimizer can replace its instructions, add or
remove Agents, introduce review and revision, use concurrency, or discover a
different structure. Every candidate remains exactly one executable
`(agent ...)` form compiled by karcarthy.

This follows the important shape of several research systems:

- [AFlow](https://openreview.net/forum?id=z5uVAKwmjf) treats workflow design as
  search over executable code and evaluates generated candidates.
- [Meta Context Engineering](https://arxiv.org/abs/2601.21557) evolves artifacts
  from rollout history and retains the best candidate by validation score.
- [GEPA](https://arxiv.org/abs/2507.19457) uses verifier feedback and full
  trajectories for reflective mutation rather than relying only on sparse
  scalar rewards.

The example uses GEPA's general text-evolution engine, with the text constrained
to a valid Clojure Agent program.

Candidate forms are full-trust JVM Clojure. Harbor isolates the task workspace,
not the host-side candidate compilation step. Use a trusted optimizer and do
not treat this example as a sandbox for adversarial source.

## Experimental loop

`examples/harbor/hillclimb.sh` performs the following loop:

1. Download a published Harbor dataset, defaulting to Terminal-Bench 2.
2. Create deterministic, disjoint train, validation, and test splits.
3. Evaluate the seed and later candidates in isolated Harbor tasks.
4. Feed sampled training rewards, verifier output, and ATIF trajectories to the
   reflective optimizer.
5. Generate and compile a complete offspring Agent form.
6. Rank candidates by mean validation reward and retain the best-so-far.
7. After search, evaluate the selected Agent on the untouched test split.

The optimizer never receives a rule requiring an `agent` Tool call. If a
multi-Agent workflow helps, search can discover it. If it does not, the best
candidate may remain a single Agent.

## Metric

The primary objective is:

```text
validation score = mean Harbor verifier reward over validation tasks
```

By default, the runner reads the verifier's `reward` field. `--metric` selects
another named reward for datasets with multiple metrics.

Training rewards and trajectories generate proposals. Validation rewards select
candidates. Test rewards are reported only after selection. This separation is
what prevents the optimizer from simply choosing the candidate that memorized
its training examples.

ATIF trajectories, verifier output, compile failures, Tool activity, Agent
calls, aggregate tokens, and runtime are diagnostic signals. They help explain
why a candidate worked, but do not earn reward merely for looking agentic.

## Run the search

The search is paid and explicitly opt-in:

```bash
KARCARTHY_LIVE=1 OPENAI_API_KEY=... \
  examples/harbor/hillclimb.sh \
  --train-size 8 --val-size 8 --test-size 8 \
  --max-evals 40 --max-iterations 5
```

The default output is `examples/harbor/runs/search`:

```text
best.clj       selected Agent program
split.json     exact task IDs in every split
summary.json   validation and held-out test metrics
trials/        Harbor results, verifier logs, ACP events, and ATIF trajectories
```

The built-in scheduler task remains only as a free packaging and verifier smoke
test. Research evaluation uses a multi-task Harbor dataset.

## Packaging boundary

karcarthy implements the ACP sessions, prompts, streamed messages, Tool updates,
permissions, model selection, cancellation, usage, and closure used by Harbor's
ACP runner. The normal library artifact contains no examples.

The Harbor directory builds a separate application uberjar containing the
candidate compiler and starts:

```bash
java -jar karcarthy-example-standalone.jar \
  acp main/harbor-agent
```

The development adapter uploads both that application and the candidate source
into each task. It also caches and uploads the matching Linux Temurin 21 JRE, so
benchmark images do not need Java preinstalled. A published karcarthy ACP
distribution could bundle that runtime and remove the local adapter.

## Trace boundary

Harbor's [ATIF format](https://www.harborframework.com/docs/agents/trajectory-format)
records portable messages, Tool calls, observations, and metrics. A runtime
Agent creation appears as an `agent` Tool call containing the submitted Clojure
source and explicit input.

ATIF is not a lossless copy of karcarthy's internal Run events. Compiler phases
and exact nested lineage remain karcarthy events. Use Harbor for portable
behavioral evaluation and persist selected internal events separately when an
experiment needs kernel-level instrumentation.

## What this is not

This is reflective evolutionary harness search, not model training. It does not
update weights, calculate policy gradients, or provide token IDs and log
probabilities. A full Harbor RL integration has additional rollout interface
requirements described in Harbor's [RL workflow](https://harborframework.com/docs/training-workflows/rl).
