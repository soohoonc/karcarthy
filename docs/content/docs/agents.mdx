An Agent describes how to complete a task. The usual Agent has a model,
instructions, Tools, and an output contract:

```clojure
(k/defagent researcher
  {:description "Research one question and return sourced findings."
   :model {:transport :responses :id "gpt-5.6"}
   :instructions "Research carefully. Distinguish evidence from inference."
   :tools [web-search]
   :output string?})
```

## Run an Agent

`run!` passes one input value to an Agent and returns a Run:

```clojure
(def run
  (k/run! researcher "What changed in the latest Java release?"))

(:status run) ;=> :completed
(:output run) ;=> "..."
```

The Run retains the input, output, usage, events, and a structured failure when
execution does not complete:

```clojure
{:karcarthy/type :run
 :id "run_..."
 :status :completed
 :agent "researcher"
 :input "What changed in the latest Java release?"
 :output "..."
 :usage {:model-calls 4
         :input-tokens 12000
         :output-tokens 900
         :agent-forms 0
         :duration-ms 4200.0}
 :events [...]
 :error nil}
```

Status is `:completed`, `:failed`, or `:cancelled`. Check it before using
`:output`; failed and cancelled Runs retain their usage, events, and `:error`.

For an Agent with a model, karcarthy validates the input, calls the model with
the Agent's instructions and callable schemas, executes requested Tools or
Agents, returns their results to the model, and repeats until the model returns
a final answer or the Run stops.

## Run options

Local dependencies belong in `:context`. They are available to Tools and
Clojure Agent programs through `(k/context)` but are not inserted into model
messages:

```clojure
(k/run! account-agent "Summarize account 123."
        {:context {:account-id "123" :db db}
         :limits {:model-calls 100
                  :input-tokens 1000000
                  :output-tokens 100000
                  :depth 8
                  :parallelism 6
                  :agent-forms 20
                  :deadline-ms 300000}
         :cancel cancelled?})
```

`:cancel` may be a Boolean, predicate, atom, or another dereferenceable value.
Cancellation is checked around Agent, model, and Tool calls. Other options are
`:session`, `:observe`, `:approval`, `:metadata`, `:model-transports`, and
`:evaluation-namespace`; the [Library API](/docs/reference/api#run) defines
their exact shapes.

## Two ways to declare an Agent

`defagent` defines a top-level var and uses its name as the Agent name:

```clojure
(k/defagent reviewer
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Review the change."
   :output string?})
```

`agent` creates the same kind of value inside an expression, so `:name` is
explicit:

```clojure
(def reviewer
  (k/agent
   {:name "reviewer"
    :model {:transport :responses :id "gpt-5.6"}
    :instructions "Review the change."
    :output string?}))
```

## Configuration

The common keys are:

| Key | Meaning |
| --- | --- |
| `:description` | Tells a parent model when to use this Agent. |
| `:model` | Selects the model and transport. |
| `:instructions` | Agent-specific instructions appended to karcarthy's base system prompt. |
| `:tools` | Tools that model may call. |
| `:agents` | Other Agents that model may call. |
| `:input` | Optional contract for values passed to the Agent. |
| `:input-schema` | Optional model-facing schema when this Agent is available to another Agent. |
| `:output` | Optional contract for the final answer. |

`defagent` supplies `:name`; `agent` requires it. The [API
reference](/docs/reference/api#agent-and-defagent) lists limits, guardrails,
dynamic instructions, and other options.

## Make specialists available

Put Agent values in `:agents`:

```clojure
(k/defagent support
  {:description "Resolve one billing question."
   :model {:transport :responses :id "gpt-5.6"}
   :instructions "Resolve the assigned billing question."
   :output string?})

(k/defagent manager
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Resolve the request. Ask support when it is useful."
   :agents [support]
   :output string?})
```

When `manager` runs, its model receives a callable function named `support`.
The default input is:

```json
{"input": "the complete input for support"}
```

If the available Agent declares an object-shaped `:input` or `:input-schema`,
that schema is exposed instead and the Agent receives the resulting map
directly.
For example:

```clojure
(k/defagent translator
  {:description "Translate text into a requested language."
   :input {:type "object"
           :properties {"text" {:type "string"}
                        "language" {:type "string"}}
           :required ["text" "language"]
           :additionalProperties false}
   :model {:transport :responses :id "gpt-5.6"}
   :instructions "Translate exactly as requested."
   :output string?})
```

The called Agent's result is returned to the parent model as a Tool result; the
parent then decides what to do next. Both Agents share the same Run limits,
usage, cancellation, approvals, and event stream.

## Use Clojure as the workflow engine

An `agent` form may contain a Clojure program instead of a model configuration:

```clojure
(k/defagent review-team
  {:input string? :output vector?}
  [change]
  (->> [security-reviewer api-reviewer]
       (mapv #(future (k/run! % change)))
       (mapv deref)
       (mapv :output)))
```

This is the escape hatch and the central design: sequencing, branching,
recursion, macros, and concurrency remain Clojure. There is no workflow graph
to translate into.

Calls to `run!` in such a program start independent Runs. Use `:agents` when a
model should choose a specialist inside its current Run; use ordinary Clojure
when the program should choose.

## Let the model write an Agent

Every model Agent can propose and run new Clojure:

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Solve the task. Write a new Agent when useful."
   :output string?})
```

The model always receives a callable capability named `agent`. Its description
explains when generation is useful, when it is unnecessary, the exact Clojure
grammar, execution behavior, and the Tools, Agents, and model configuration
available to generated code. The call has two fields:

```json
{
  "source": "(agent {:name \"specialist\" ...})",
  "input": "the complete input for the new Agent"
}
```

karcarthy requires one top-level `agent` form, expands it, evaluates it,
validates `input`, and runs it. The generated Agent receives that value but not
the parent's conversation, reasoning, previous Tool results, or Session
history. Run-local application context remains available to its Clojure code
and Tools but is not inserted into model messages. Its output returns to the
parent model, which continues.

The new form can contain ordinary Clojure and calls to other Agents. A
generated model Agent has the same `agent` capability, so generation is
recursive and bounded by Run limits.

This executes model-authored JVM Clojure. Use it only in a process or sandbox
with the authority you intend to grant. See [Generated
Agents](/docs/reference/generated-agents) for the exact pipeline.

For conversation history across root Runs, see [Sessions](/docs/sessions). For
incremental output and the event tree, see [Streaming and events](/docs/streaming).
