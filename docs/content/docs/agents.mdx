---
title: Agents
description: Define model-backed Agents, Clojure-body Agents, and Agents that models can call.
---

An Agent is an executable Clojure value. It either runs karcarthy's model and
Tool loop or executes a Clojure body.

## Model-backed Agents

```clojure
(k/defagent researcher
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Research the question and return a concise answer."
   :tools [search-tool]
   :output string?})
```

The core configuration is deliberately small:

| Key | Purpose |
| --- | --- |
| `:name` | Agent identity; `defagent` supplies it from the var name. |
| `:model` | Model and transport configuration. |
| `:instructions` | Model-visible policy for the Agent. |
| `:tools` | Capabilities available to the model. |
| `:output` | Optional contract for the final result. |

The [Library API](/docs/reference/api#agent) lists advanced configuration such
as guardrails, stop conditions, dynamic configuration, and limits.

## Input, context, and output

These are different boundaries:

| Boundary | Who sees it | Normal use |
| --- | --- | --- |
| `run!` input | The Agent | A user prompt for a model-backed Agent. |
| `:context` Run option | Local Clojure code | Dependencies and request data not sent automatically to the model. |
| Agent output | The caller | Text or structured data validated by `:output`. |

For a model-backed Agent, use a string prompt:

```clojure
(k/run! researcher "What changed in the latest Java release?")
```

karcarthy can pass another Clojure value and the Responses transport will
encode it as JSON, but that is not the primary interface. Prefer a clear prompt
and keep databases, authenticated users, and other application data in
`:context`. Instructions may deliberately project selected context into the
prompt when the model needs it.

`:input` is an optional boundary contract. It is most useful for Clojure-body
Agents and Agents exposed as Tools. For an ordinary model-backed Agent,
`:input string?` is usually the only useful constraint.

## `agent` and `defagent`

Use `agent` when constructing a value inside an expression:

```clojure
(def reviewer
  (k/agent
   {:name "reviewer"
    :model {:transport :responses :id "gpt-5.6"}
    :instructions "Review the change."
    :output string?}))
```

Use `defagent` for a named top-level definition:

```clojure
(k/defagent reviewer
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Review the change."
   :output string?})
```

Both forms create the same kind of value. The relationship is like `fn` and
`defn`.

## Agents with Clojure bodies

A Clojure-body Agent receives its input unchanged, so it may accept any
Clojure value:

```clojure
(k/defagent select-request-fields
  {:input map?
   :output map?}
  [request]
  (select-keys request [:task :repository]))
```

The body has exactly one binding, `[input]`. It can call functions, use macros,
and run other Agents. Calls to `run!` start independent Runs; see
[Running agents](/docs/running-agents#compose-agents-with-clojure).

## Expose a known Agent as a Tool

Use `as-tool` when a parent model should call an Agent you already defined:

```clojure
(k/defagent support-manager
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Resolve the request. Use billing for billing questions."
   :tools [(k/as-tool billing-agent
                      {:name "billing"
                       :description "Handle a billing request."})]
   :output string?})
```

The parent remains active after the child finishes. The child shares the
parent Run's limits, cancellation, approvals, usage, and events. Its input is
the Tool input, so give the Agent or `as-tool` a model-usable input schema when
the input is structured.

## Let the model define an Agent

Zero-arity `(k/agent)` returns a Tool named `agent`:

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions
   "Solve the task. Create a focused Agent when specialization helps."
   :tools [(k/agent)]
   :output string?})
```

The model calls the Tool with one `(agent ...)` form and the input for the new
Agent. karcarthy reads, macroexpands, checks, evaluates, verifies, and runs that
Agent inside the current Run. A generated Agent may receive `(agent)` itself,
so the mechanism is recursive and bounded by Run limits.

Generated code is real JVM Clojure. Use an isolated process or sandbox when the
model or task input is not trusted. The [Generated Agents
reference](/docs/reference/generated-agents) documents evaluation and limits.

## Inspect an Agent

Agents retain the form that created them and its macroexpansion:

```clojure
(k/source-form reviewer)
(k/expanded-form reviewer)
```

Those forms are ordinary Clojure data and may be inspected, transformed, or
compared in an experiment.
