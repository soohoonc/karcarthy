title: Tools
description: Define Clojure functions that a model can call.
---

A Tool is a Clojure function with a model-visible name, description, and input
schema.

```clojure
(k/deftool lookup-user
  {:description "Look up a user by ID."
   :input {:type "object"
           :properties {"user-id" {:type "string"}}
           :required ["user-id"]
           :additionalProperties false}
   :output map?}
  [{:keys [user-id]}]
  (lookup! (:db (k/context)) user-id))
```

Expose it by putting the Tool value in an Agent's `:tools` vector:

```clojure
(k/defagent support
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Resolve the user's request."
   :tools [lookup-user]
   :output string?})
```

## What happens on a Tool call

On every model turn, karcarthy sends each Tool's name, description, and JSON
input schema. The model may return a Tool call such as:

```clojure
{:name "lookup-user"
 :input {:user-id "user_123"}}
```

karcarthy validates the input, requests approval if configured, runs the
Clojure function, validates its output, and sends that output back to the
model. The model continues until it returns a final answer.

## `tool` and `deftool`

Use `deftool` for a top-level definition. Use `tool` to create the same value
inside an expression:

```clojure
(def lookup-user
  (k/tool
   {:name "lookup-user"
    :description "Look up a user by ID."
    :input map?
    :input-schema {:type "object"
                   :properties {"user-id" {:type "string"}}
                   :required ["user-id"]
                   :additionalProperties false}
    :output map?}
   [{:keys [user-id]}]
   (lookup! (:db (k/context)) user-id)))
```

The body has one input binding and can use all of Clojure.

## Contracts and schemas

`:input` is the execution contract. If it is JSON Schema, the same value is
shown to the model. Clojure predicates and specs also work as contracts;
karcarthy derives a schema when possible. When the contract cannot describe a
useful model interface, set `:input-schema` separately as in the example above.

The same distinction exists between `:output` and `:output-schema`.

Function Tool schemas should be object-shaped with named properties. That is
the common interface expected by model providers, even though a local Clojure
function can accept any value.

## Context and approval

Pass local dependencies to `run!`:

```clojure
(k/run! support "Look up account 123."
        {:context {:db db}})
```

Tools read them with `(k/context)`. Context is not sent to the model unless
your code deliberately includes it in instructions, messages, or Tool output.

For a Tool with `:approval :always` or `:once`, pass an approval function:

```clojure
(k/run! support prompt
        {:approval
         (fn [{:keys [tool input]}]
           (confirm? (:name tool) input))})
```

Without a handler, approval fails closed.

## Where Tools come from

All callable functions enter an Agent through `:tools`, but they may execute
in different places:

| Source | Execution |
| --- | --- |
| `tool` / `deftool` | A Clojure function in the karcarthy process. |
| `local-tools` | Built-in filesystem, shell, and search functions in the karcarthy process. |
| provider-hosted Tool | The selected model endpoint executes it. |
| MCP Tool | An external MCP server process executes it. |

`local-tools` returns `read`, `write`, `edit`, `bash`, and ripgrep-backed
`search` Tools:

```clojure
(def tools
  (k/local-tools
   {:cwd "/workspace/project"
    :approval {:read :never
               :search :never
               :write :always
               :edit :always
               :bash :always}}))
```

File Tools remain under `:cwd`, including after resolving symlinks. `bash` is
bounded by timeout and output size but is not a sandbox.

Provider-hosted capabilities are declared as Tool values but executed by the
model endpoint; see [Models](/docs/models#provider-hosted-tools).

For MCP, karcarthy starts a stdio MCP server, discovers its remote Tool
definitions, and turns each one into an ordinary Tool value. Connection
lifecycle, approval defaults, naming, supported protocol features, and
ACP-provided MCP servers are documented on the dedicated [MCP page](/docs/mcp).

Agents are declared separately with `:agents`; see [Make specialists
available](/docs/agents#make-specialists-available).
