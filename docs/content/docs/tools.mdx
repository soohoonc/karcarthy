---
title: Tools
description: Define typed capabilities that a model can call.
---

A Tool is a model-callable function. Unlike an Agent prompt, a Tool input is
intentionally structured: its schema tells the model which arguments to
produce and lets karcarthy validate them before execution.

## Define a Tool

```clojure
(k/deftool lookup-user
  {:description "Look up a user by ID."
   :input {:type "object"
           :properties {:user-id {:type "string"}}
           :required ["user-id"]
           :additionalProperties false}
   :output map?}
  [{:keys [user-id]}]
  (lookup! (:db (k/context)) user-id))
```

A Tool body has exactly one binding. Here the model supplies a map and the body
destructures it.

Use `tool` inside an expression and `deftool` for a named top-level definition.
They create the same Tool value; `deftool` supplies the default Tool name from
the var name.

## Schemas and contracts

`:input` is both the execution contract and, when it is JSON Schema, the schema
shown to the model. Use an object with named properties for a model-facing
function Tool. Local contracts may accept other values, but model providers may
reject non-object function schemas.

Clojure predicates and specs can also validate input. If a contract cannot be
translated into a useful model schema, provide the two separately:

```clojure
{:input request?
 :input-schema {:type "object"
                :properties {:query {:type "string"}}
                :required ["query"]
                :additionalProperties false}}
```

The same distinction exists for `:output` and `:output-schema`.

## Tool configuration

| Key | Purpose |
| --- | --- |
| `:name` | Model-visible function name. |
| `:description` | When and how the model should use the Tool. |
| `:input` | Input contract. Required. |
| `:input-schema` | Explicit model-facing schema. |
| `:output` | Optional output contract. |
| `:output-schema` | Explicit output schema. |
| `:approval` | Approval policy. |
| `:timeout-ms` | Execution timeout. |
| `:retry` | Retry count or options. |
| `:to-model-output` | Projection of a rich result into what the model sees. |

The [Library API](/docs/reference/api#tool-and-deftool) lists dynamic enablement,
guardrails, and metadata.

## Local context

Tools read local application dependencies with `context`:

```clojure
(k/run! support-agent "Look up account 123."
        {:context {:db db}})
```

```clojure
(k/deftool lookup-account
  {:description "Look up an account."
   :input {:type "object"
           :properties {:id {:type "string"}}
           :required ["id"]}
   :output map?}
  [{:keys [id]}]
  (lookup! (:db (k/context)) id))
```

Context is not sent to the model automatically.

## Approval

Set `:approval` to `:always` or `:once`, then supply a handler to `run!`:

```clojure
(k/run! agent prompt
        {:approval
         (fn [{:keys [tool input]}]
           (confirm? (:name tool) input))})
```

Without a handler, a Tool requiring approval fails closed.

## Built-in local Tools

```clojure
(def tools
  (k/local-tools
   {:cwd "/workspace/project"
    :approval {:read :never
               :search :never
               :write :always
               :edit :always
               :bash :always}}))
```

`local-tools` returns `read`, `write`, `edit`, `bash`, and ripgrep-backed
`search`. File Tools stay inside `:cwd`, including through symlinks. `bash` is
bounded but is not a sandbox.

Hosted provider Tools and MCP Tools enter the same `:tools` vector. See
[Integrations](/docs/integrations).

## Agents as Tools

`as-tool` exposes an existing Agent. Zero-arity `(agent)` lets the model supply
a new Agent definition. Both are covered in [Agents](/docs/agents#expose-a-known-agent-as-a-tool).
