The examples use one rule throughout: orchestration is Clojure code. Agents may
write more Clojure code, and the harness records both source and expanded forms
alongside the execution trace.

## Examples

| Page | Shows |
| --- | --- |
| [Clojure](/docs/examples/clojure) | Tools, model-backed agents, a custom orchestration body, parallel children, generated agents, and trace capture. |
| [Python](/docs/examples/python) | The ACP client boundary for driving a running Clojure harness from Python or Harbor. |
| [TypeScript](/docs/examples/typescript) | The same ACP boundary from a TypeScript application. |

Python and TypeScript are clients, not alternative authoring languages for the
agent program. The executable program remains Clojure; ACP carries prompts,
events, approvals, and results across the process boundary.

## Pattern: a model-backed leaf

```clojure
(k/defagent researcher
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Research carefully and cite evidence."
   :tools [web-search fetch-page]
   :output ::report})
```

## Pattern: orchestration as code

```clojure
(k/defagent research-team
  {:input ::question
   :output ::report}
  [rt question]
  (let [draft (k/invoke! rt researcher question)
        checks (k/await-all!
                [(k/spawn! rt fact-checker draft)
                 (k/spawn! rt citation-checker draft)])]
    (k/invoke! rt editor {:draft draft :checks checks})))
```

## Pattern: generated behavior

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions "Design and run a child Agent for the task."
   :tools [(k/agent)]})
```

The model calls the same `agent` form used by authored Clojure. The harness
reads, expands, checks, evaluates, and verifies its source before the child
runs.
