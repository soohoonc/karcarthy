This guide introduces the native karcarthy API.

## Requirements

- JDK 21 or newer
- Clojure CLI
- an API key for the model provider you configure
- ripgrep when using `local-tools` search

During development, depend on karcarthy from Git:

```clojure
;; deps.edn
io.github.soohoonc/karcarthy
{:git/url "https://github.com/soohoonc/karcarthy"
 :git/sha "<commit sha>"}
```

## Define a tool

A tool is a Clojure function with model-facing documentation and contracts at
its input and output boundaries:

```clojure
(require '[clojure.spec.alpha :as s]
         '[karcarthy :as k])

(s/def ::query string?)
(s/def ::search-input (s/keys :req-un [::query]))
(s/def ::search-output vector?)

(k/deftool web-search
  {:description "Search the web for relevant sources."
   :input ::search-input
   :output ::search-output}
  [rt {:keys [query]}]
  (search! (:http (k/environment rt)) query))
```

The harness validates tool arguments before execution and validates the return
value before exposing it to the model. Tools may also declare `:approval`,
`:enabled?`, guardrails, timeouts, and a projection controlling what the model
sees.

## Define an agent

```clojure
(s/def ::question string?)
(s/def ::request (s/keys :req-un [::question]))
(s/def ::answer string?)
(s/def ::sources vector?)
(s/def ::report (s/keys :req-un [::answer ::sources]))

(k/defagent researcher
  {:description "Research a question and return a sourced report."
   :model {:transport :responses
           :id "gpt-5.6"
           :reasoning :high}
   :context
   (fn [{:keys [environment]}]
     (str "Research carefully for user " (:user-id environment)
          ". Distinguish evidence from inference."))
   :environment map?
   :input ::request
   :tools [web-search]
   :output ::report
   :loop {:max-turns 20}})
```

With no body after the configuration map, `defagent` installs the standard
model/tool loop.

The built-in `:responses` transport defaults to OpenAI. To use a compatible
gateway, keep the Agent unchanged and configure its model boundary:

```clojure
{:transport :responses
 :provider :anthropic
 :id "anthropic/claude-model"
 :base-url "https://ai-gateway.vercel.sh/v1"
 :api-key-env "AI_GATEWAY_API_KEY"}
```

## Run it

```clojure
(def run
  (k/run! researcher
          {:question "What makes Lisp homoiconic?"}
          {:environment {:user-id "u_123"
                         :http http-client}
           :limits {:model-calls 20
                    :deadline-ms 120000}
           :observe println}))

(:status run) ;=> :completed
(:output run) ;=> {:answer "..." :sources [...]}
```

`run!` returns the complete run record. Tools and parent programs normally use
`invoke!`, which returns the child output while attaching the child's run to
the same trace tree.

To continue the same model conversation later, persist the plain-data snapshot
and pass it back explicitly:

```clojure
(def next-run
  (k/run! researcher next-request {:state (:state run)}))
```

## Add local tools

For a Codex/Pi/Claude-Code-shaped local Agent, compose the provided Tools and
prompt explicitly:

```clojure
(def tools
  (conj (k/local-tools {:cwd "/workspace/project"})
        (k/responses-web-search)))

(def coder
  (k/agent
   {:name "coder"
    :model {:transport :responses
            :id "gpt-5.6"
            :reasoning :high}
    :context
    (k/prompt
     (k/system-prompt)
     (k/prompt-file "/workspace/project/AGENTS.md"))
    :tools tools
    :output string?}))

(k/run! coder "Fix the failing test and verify the smallest relevant target.")
```

`local-tools` returns `read`, `write`, `edit`, `bash`, and ripgrep-backed
`search`, all rooted at `:cwd`. Prompt composition is independent:
`system-prompt` reads the packaged `resources/karcarthy/system.md`, while
`prompt-file` reads only the file the caller names. Hosted web search is an
explicit Responses Tool whose availability depends on the endpoint and selected
model. None of these functions introduces a special Agent.

## Add MCP tools

MCP is an adapter into ordinary Tools:

```clojure
(def server
  (k/connect-mcp!
   {:name "project"
    :command "npx"
    :args ["-y" "@example/project-mcp"]}))

(try
  (let [tools (into (k/local-tools {:cwd "/workspace/project"})
                    ;; :never means this MCP server is explicitly trusted.
                    (k/mcp-tools server {:approval :never}))
        coder (k/agent
               {:name "coder"
                :model {:transport :responses :id "gpt-5.6"}
                :context
                (k/prompt
                 (k/system-prompt)
                 (k/prompt-file "/workspace/project/AGENTS.md"))
                :tools tools
                :output string?})]
    (k/run! coder "Inspect the project service."))
  (finally
    (k/close-mcp! server)))
```

The implemented MCP transport is stdio using protocol version `2025-11-25`.
MCP tools require approval by default. The inner model loop has no special MCP
branch.

## Serve an Agent over ACP

Use a factory when the Agent's local directory and MCP tools depend on the ACP
session:

```clojure
(defn acp-agent [{:keys [cwd mcp-tools]}]
  (let [tools (into (k/local-tools {:cwd cwd}) mcp-tools)]
    (k/agent
     {:name "local-agent"
      :model {:transport :responses :id "gpt-5.6"}
      :context (k/system-prompt)
      :tools tools
      :output string?})))
```

```bash
clojure -M -m karcarthy.acp your.namespace/acp-agent
```

The ACP v1 stdio server supports initialization, new/prompt/close sessions,
cancellation, streaming Agent-message chunks, tool-call updates, client
permissions, and session-provided stdio MCP servers. HTTP/SSE MCP and resumable
ACP sessions are not advertised.

## Compose agents with Clojure

```clojure
(k/defagent answer-and-review
  {:input ::request
   :output ::report}
  [rt request]
  (let [report (k/invoke! rt researcher request)
        review (k/invoke! rt fact-checker report)]
    (if (:accepted? review)
      report
      (k/invoke! rt researcher
                 (assoc request :feedback (:feedback review))))))
```

This is not lowered into `pipe`, `route`, or `revise` nodes. It is compiled and
executed as the Clojure program shown.

## Let an agent create another agent

Add the expansion capability to its tool set:

```clojure
(k/defagent architect
  {:model {:transport :responses :id "gpt-5.6"}
   :context "Design and run the smallest useful child Agent."
   :tools [(k/agent)]})
```

The same `agent` form is both constructor and capability. With a config it
constructs an Agent; with no arguments it returns the model-facing `agent`
Tool. That Tool accepts `{:source <one agent form> :input <value>}` and reads,
expands, checks, evaluates, and invokes the child inside the same trace and
budgets.

The source may contain a normal agent form with arbitrary Clojure control
flow:

```clojure
(agent
  {:name "review-team"
   :input ::request
   :output ::report}
  [rt request]
  (let [draft (invoke! rt researcher request)
        checks (await-all!
                [(spawn! rt fact-checker draft)
                 (spawn! rt citation-checker draft)])]
    (invoke! rt editor {:draft draft :checks checks})))
```

There is no separate dynamic mode or expansion node. The application grants
the language's own `agent` form to the model.
