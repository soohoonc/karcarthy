In this guide you run one agent on the offline mock runner, compose two agents
into a workflow, and run the same thing from the command line. Nothing here
needs an API key until the final section, which is marked as paid.

## Requirements

- JDK 21 or newer
- Clojure CLI
- A checkout of `soohoonc/karcarthy`

## Run one agent

Start a REPL in the checkout and run:

```clojure
(require '[karcarthy :as k])

(k/defagent echo
  {:instructions "Echo the input."})

(k/run {:runner (k/mock-runner)
        :workflow echo
        :input "hi"})
```

The mock runner answers offline by echoing the prompt, tagged with the agent
name:

```clojure
{:karcarthy/type :result
 :ok? true
 :agent "echo"
 :text "[echo] hi"
 :raw {:runner :mock}}
```

That is the smallest workflow: one request, one agent node, one runner.

## Compose two agents

`pipe` runs workflows in sequence and feeds the first result's `:text` into
the second agent:

```clojure
(k/defagent researcher
  {:instructions "Research the question and list the important facts."})

(k/defagent summarizer
  {:instructions "Summarize the previous answer in one sentence."})

(def workflow
  (k/pipe researcher summarizer))

(k/run {:runner (k/mock-runner)
        :workflow workflow
        :input "why does Lisp use code as data?"})
```

## The request map

Every run — including the two above — is one request map:

```clojure
(k/run {:runner runner
        :workflow workflow
        :input input
        :options options})
```

| Key | Meaning |
| --- | --- |
| `:runner` | The backend used for each agent call, such as mock, ACP, Claude, Codex, OpenAI, a process, or a local function. |
| `:workflow` | The agent or workflow data to run. A single agent is already a workflow. |
| `:input` | The initial workflow value. A string is a prompt; EDN is rendered before the first agent call. |
| `:options` | Runtime controls for this run, such as `:observe`, `:state`, timeouts, or runner-specific overrides. |

## Input shapes

The common case is a string prompt:

```clojure
(k/run {:runner runner
        :workflow workflow
        :input "review this change"})
```

Use a map when the initial task has structure:

```clojure
(k/run {:runner runner
        :workflow workflow
        :input {:prompt "review this change"
                :ticket 42
                :risk :high}})
```

That starts the first agent with:

```text
review this change

INPUT EDN:
{:ticket 42, :risk :high}
```

A map without `:prompt` is passed as EDN text. Put execution configuration in
`:options`, runner construction, or agent fields such as `:model` and `:tools`,
not inside `:input`.

## Run from the command line

In a repo checkout, `./bin/karcarthy` runs through the Clojure CLI directly.
Build a standalone jar for faster startup or installed copies:

```bash
clojure -T:build uber
```

Run one mock agent:

```bash
./bin/karcarthy agent echo --instructions "Echo the input." hi
```

Expected output:

```text
[echo] hi
```

## Run a live model (paid)

Everything above runs offline and free. The calls below invoke real model
backends: `claude-runner` shells out to the `claude` CLI and `codex-runner` to
`codex exec`, so each needs its CLI installed and authenticated — and each call
costs money.

Agents stay runner-agnostic; they never carry a backend:

```clojure
(k/defagent reviewer
  {:instructions "Review the input and return concrete issues."
   :model "sonnet"})

(defn review [runner input]
  (k/run {:runner runner
          :workflow reviewer
          :input input}))

(review (k/mock-runner) "check this design")                            ; offline
(review (k/claude-runner {}) "check this design")                       ; paid: claude CLI
(review (k/codex-runner {:dir "."}) "check this design")                ; paid: codex exec
(review (k/acp-runner {:command ["my-acp-agent"]}) "check this design") ; your ACP agent
```

If the runner changes at runtime, pass a different runner value.

The same switch exists on the CLI with `--runner`:

```bash
./bin/karcarthy agent reviewer \
  --runner claude \
  --instructions "Review the input and return concrete issues." \
  "check this design"
```

The CLI's `claude` runner uses lean sub-agent defaults — system-prompt replace
mode, tools disabled — so agents answer directly.
