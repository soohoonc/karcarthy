---
title: Build Workflows
description: Understand how pipe, step, branch, delegate, reduce, revise, route, continue, and dynamic move data through agents.
---

A workflow is a value. `run` interprets that value through a runner and
returns a result map.

```clojure
(k/run runner workflow "input text")
;=> {:karcarthy/type :result
;    :ok? true
;    :text "..."}
```

Every workflow node consumes text and returns a result. Composite nodes decide
how child result text is routed to the next node.

## Mental model

| Node | Use it when | Input behavior | Output behavior |
| --- | --- | --- | --- |
| `pipe` | Steps must happen in order. | First step receives the original input; each later step receives previous `:text`. | Last successful step's result. |
| `step` | Host Clojure code should transform workflow text. | Receives the flowing input string, or an explicit context map. | A normal result map. |
| `branch` | Several reviewers should inspect the same input independently. | Each branch receives the same input. | A result with `:results` and combined `:text`. |
| `delegate` | A planner should create subtasks for a worker. | Planner receives the original input; worker receives one subtask at a time. | A result with `:subtasks`, `:results`, and combined `:text`. |
| `reduce` | Branch or delegate outputs need synthesis by another workflow. | Runs the source workflow, then sends an EDN result summary to the reducer workflow. | Reducer result plus `:source` and `:reduced`. |
| `revise` | A draft needs critique and revision. | Worker receives the input, then revision prompts. | Final draft plus `:rounds` and `:accepted?`. |
| `route` | A classifier should choose the next path. | Source receives original input. | Selected workflow result, or `:error :no-route`. |
| `continue` | One step should hand off to another. | Second workflow receives source result text. | Continued workflow result. |
| `dynamic` | The workflow should decide its next step while it runs. | An agent emits one EDN op at a time. | Final `:complete` result plus the run `:state`. |

## pipe

Use `pipe` for sequential work:

```clojure
(def workflow
  (k/pipe researcher summarizer))

(k/run (k/mock-runner) workflow "what is a monad?")
```

Trace with the mock runner:

```text
input:      what is a monad?
researcher: [researcher] what is a monad?
summarizer: [summarizer] [researcher] what is a monad?
```

The next step receives only the previous result text. If a step fails, the pipe
returns that not-ok result and stops.

## step

Use `step` for local Clojure transforms inside a workflow:

```clojure
(def workflow
  (k/pipe
   writer
   (k/step clojure.string/trim :name "trim")
   reviewer))
```

By default the function receives only the flowing input string. If it needs run
metadata, opt into context:

```clojure
(k/step
 (fn [{:keys [input node]}]
   (str (:name node) ": " input))
 :name "tag"
 :context? true)
```

`step` is deliberately Clojure-local because it stores a function. It is not
EDN/JSON portable. Use `process-runner` when the execution boundary is a fixed
external process selected by the runner registry.

## branch

Use `branch` when independent agents should inspect the same input:

```clojure
(def reviewers
  (k/branch [product-reviewer engineering-reviewer security-reviewer]))
```

Trace:

```text
input: enterprise SSO beta
product-reviewer:     reviews the same input
engineering-reviewer: reviews the same input
security-reviewer:    reviews the same input
```

The result contains ordered child results:

```clojure
{:karcarthy/type :result
 :ok? true
 :results [{:agent "product-reviewer" :text "..."}
           {:agent "engineering-reviewer" :text "..."}
           {:agent "security-reviewer" :text "..."}]
 :text "...joined child text..."}
```

## delegate

Use `delegate` when a planner should create subtasks:

```clojure
(k/delegate planner worker :max-concurrency 8)
```

The planner replies with EDN:

```clojure
{:subtasks ["audit logs" "rollback messaging" "support docs"]}
```

The worker runs once per subtask.

## reduce

Use `reduce` when branch or delegate outputs need synthesis by another workflow:

```clojure
(def review-packet
  (k/reduce
    (k/branch [product-reviewer engineering-reviewer security-reviewer])
    brief-writer))
```

The reducer receives a prompt containing EDN data:

```clojure
{:input "original input"
 :results [{:agent "product-reviewer" :ok? true :text "..."}
           {:agent "engineering-reviewer" :ok? true :text "..."}]}
```

Because reducers are workflows, not host functions, the workflow stays
serializable unless you deliberately add Clojure-local `step` nodes.

## revise

Use `revise` for draft, critique, revise loops:

```clojure
(def launch-brief
  (k/revise brief-writer critic :max-rounds 2))
```

Trace:

```text
round 1 worker:    writes a draft
round 1 evaluator: replies {:accept? true} or {:accept? false :feedback "..."}
round 2 worker:    receives input, previous draft, and feedback
```

The result includes loop metadata:

```clojure
{:karcarthy/type :result
 :ok? true
 :text "final draft"
 :rounds 2
 :accepted? true}
```

Evaluator workflows reply with EDN verdict maps:

```clojure
{:accept? true}
{:accept? false :feedback "name the missing rollback owner"}
```

## route

Use `route` when one workflow chooses the next path:

```clojure
(k/route classifier
         {:launch launch-brief
          :incident incident-plan}
         :default launch-brief)
```

Trace:

```text
input:     Prepare a launch review.
classifier result text: {:route :launch}
selected workflow:      launch-brief
launch-brief input:     Prepare a launch review.
```

## continue

Use `continue` for a direct handoff:

```clojure
(k/continue triage-agent specialist-agent)
```

The second workflow receives the source result text.

## dynamic

Use `dynamic` when the workflow should shape itself during the run:

```clojure
(def workflow
  (k/dynamic workflow-agent :max-steps 25))

(k/run runner workflow "audit this migration")
```

The dynamic workflow agent replies with one EDN op at a time:

```clojure
{:op :define
 :agent {:name "reader"
         :instructions "Inspect one file."
         :runner :claude
         :model "sonnet"
         :tools ["Read"]
         :config {}}}
{:op :define :name "review"
 :workflow {:karcarthy/type :pipe
            :steps [{:karcarthy/type :agent-ref :name "reader"}]}}
{:op :call :workflow "review" :input "src/foo.clj"}
{:op :patch :agent "reader" :merge {:instructions "Inspect one file and list risks."}}
{:op :spawn :agent "reader" :inputs ["src/a.clj" "src/b.clj"]}
{:op :complete :text "done"}
```

`agent-ref` and `workflow-ref` are late-bound. If the agent patches `"reader"`,
later calls to workflows containing `(k/agent-ref "reader")` use the patched
agent. The run state is local to that one workflow run.

## Transform before running

Because workflows are data, application code can inspect and rewrite them
before `run` interprets them:

```clojure
(def production-workflow
  (->> (k/pipe researcher summarizer)
       (k/configure {:runner :claude
                     :model "claude-sonnet-4"
                     :instructions/suffix "State assumptions before final answer."})))

(map #(select-keys % [:name :runner :model])
     (k/agents production-workflow))
;=> ({:name "researcher", :runner :claude, :model "claude-sonnet-4"}
;    {:name "summarizer", :runner :claude, :model "claude-sonnet-4"})

(k/run runner production-workflow "summarize the prior work")
```

That is the point of karcarthy: keep the coordination graph visible and
serializable when it uses portable nodes, while runners own model calls, tools,
and provider-specific agent loops. It is EDN plus an explicit interpreter, not
raw `eval`.
