The Runtime owns one root execution tree: model calls, Tool calls, child
Agents, generated Agents, usage, events, cancellation, and deadlines.

## Execute an Agent

```clojure
(def run
  (k/run! researcher
          {:question "How does ACP model selection work?"}
          {:context {:account-id "acct_123"
                     :db db}
           :limits {:model-calls 20
                    :deadline-ms 120000}
           :observe handle-event}))
```

`run!` returns:

```clojure
{:karcarthy/type :run
 :id "run_..."
 :status :completed
 :agent "researcher"
 :input {:question "..."}
 :output {:answer "..."}
 :usage {:model-calls 4
         :input-tokens 12000
         :output-tokens 900
         :generated-forms 0
         :duration-ms 4200.0}
 :events [...]
 :error nil}
```

Status is `:completed`, `:failed`, or `:cancelled`. Failed Runs retain events,
usage, and structured error data.

## Model/Tool loop

A model-backed Agent follows this lifecycle:

1. validate Agent input and local context;
2. resolve model configuration, instructions, and Tools;
3. load messages from the optional Session;
4. send messages and Tool schemas to the model;
5. stream and record model deltas when supported;
6. validate, approve, and execute requested Tools;
7. append Tool results and continue the model loop;
8. validate the final output and update the Session.

The loop stops on valid output, failure, cancellation, deadline, or
`:max-turns`.

## Instructions and context

`:instructions` is model-visible. `:context` is local application data and is
never added to the prompt automatically.

```clojure
(k/defagent account-agent
  {:model {:transport :responses :id "gpt-5.6"}
   :instructions
   (fn [{:keys [context]}]
     (str "Assist account " (:account-id context) "."))
   :context map?
   :tools [lookup-account]
   :output string?})

(k/run! account-agent request
        {:context {:account-id "acct_123"
                   :db db}})
```

Tools and custom Agent bodies read local context with `(k/context rt)`.
`prompt`, `prompt-file`, and `system-prompt` can compose instruction text.

## Sessions

Runs are stateless unless a Session is supplied:

```clojure
(def session (k/memory-session))

(k/run! assistant "My preferred format is Markdown."
        {:session session})

(k/run! assistant "Format the previous answer again."
        {:session session})
```

A Session is an ordered conversation store. `memory-session` is thread-safe
and process-local. Durable applications implement
`karcarthy.session/Session` with a database or other storage system.

## Child execution

Inside a custom Agent body, use `invoke!` for a synchronous child:

```clojure
(k/invoke! rt reviewer change)
```

Use `spawn!` and `await-all!` for parallel children:

```clojure
(let [children [(k/spawn! rt security-reviewer change)
                (k/spawn! rt api-reviewer change)]]
  (k/await-all! children))
```

Children share usage counters, cancellation, deadline, event storage, and the
parallelism limit. A child may lower inherited limits but cannot raise them.

## Limits and cancellation

```clojure
{:model-calls 100
 :input-tokens 1000000
 :output-tokens 100000
 :agent-depth 8
 :parallelism 6
 :generated-forms 20
 :deadline-ms 300000}
```

`agent-depth` limits child nesting, including model-created Agents.
`generated-forms` limits successful and failed generated-Agent compilation
attempts. Token and model-call counters cover the complete Run tree.

Pass an atom, dereferenceable value, predicate, or Boolean as `:cancel`.
Cancellation is checked by Agent, model, Tool, and child-execution paths.

## Events and streaming

Use `:observe` to receive events as they occur:

```clojure
(k/run! agent input {:observe println})
```

Common event families are:

```clojure
:run/started          :run/completed       :run/failed
:agent/started        :agent/completed     :agent/failed
:model/requested      :model/text-delta    :model/completed
:tool/started         :tool/completed      :tool/failed
:session/updated
:program/read         :program/expanded    :program/evaluated
```

Every event has a Run ID. Agent events also carry Agent ID, parent ID, and
depth. `k/events` returns the recorded event vector from a Run or Runtime.

The Responses transport parses typed SSE events and emits normalized text and
Tool-call deltas. A non-streaming final response remains authoritative.

## Model transport

The built-in Responses-compatible transport is selected in Agent config:

```clojure
{:transport :responses
 :provider :openai
 :id "gpt-5.6"}
```

For a compatible gateway, configure its endpoint and credential environment:

```clojure
{:transport :responses
 :provider :anthropic
 :id "anthropic/claude-model"
 :base-url "https://ai-gateway.vercel.sh/v1"
 :api-key-env "AI_GATEWAY_API_KEY"}
```

Provider transports perform model I/O. Tool execution, Sessions, child Agents,
limits, and events remain in the Runtime.

## Serve over ACP

Expose a static Agent or per-session factory over ACP v1 stdio:

```clojure
(defn acp-agent [{:keys [cwd mcp-tools model-id]}]
  (k/agent
   {:name "coding-agent"
    :model {:transport :responses :id model-id}
    :instructions (k/system-prompt)
    :tools (into (k/local-tools {:cwd cwd}) mcp-tools)
    :output string?}))

(k/serve-acp! {:agent acp-agent
               :models ["gpt-5.6" "gpt-5.5"]})
```

For normal stdio use, launch a var from a separate process:

```bash
clojure -M -m karcarthy.acp example/acp-agent
```

The server supports ACP initialization, session creation and close, prompts,
cancellation, client permission requests, model selection, session-provided
stdio MCP servers, streaming Agent-message chunks, Tool updates, and prompt
token usage when the model transport reports it.

Harbor's generic ACP adapter can launch the same server, capture ACP events,
and produce an ATIF `trajectory.json`. ATIF generation and evaluation records
are owned by Harbor; karcarthy supplies the ACP execution boundary.
