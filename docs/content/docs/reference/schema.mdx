---

karcarthy values are ordinary Clojure values. Some contain functions, closures,
protocol implementations, or compiled code and are intentionally not EDN/JSON
portable.

The shapes here describe the native harness. Some implementation maps carry
additional internal fields.

## Agent

An Agent is a self-describing executable value:

```clojure
{:karcarthy/type :agent
 :name "researcher"
 :config {...}
 :source-form '(agent {...})
 :expanded-form '(...)
 :invoke #object[clojure.lang.IFn ...]}
```

The public configuration surface is:

| Key | Meaning |
| --- | --- |
| `:name` | Stable identifier. Supplied from the symbol by `defagent`. |
| `:description` | Model- and human-facing description used in delegation. |
| `:model` | Transport, optional provider metadata, model identifier, reasoning, sampling, and endpoint options. May be a resolver. |
| `:context` | System-prompt string or per-turn function producing `{:system ... :messages [...]}`. |
| `:environment` | Contract for the local, non-model-visible dependency value. |
| `:input` | Input contract. |
| `:tools` | Tool values or a Runtime-dependent tool resolver. |
| `:output` | Output contract. Text is the default. |
| `:loop` | Turn limit and stop predicate. |
| `:guardrails` | Input and output guardrails. |
| `:limits` | Limits narrower than the current Runtime grant. |
| `:hooks` | Agent lifecycle observers. |

A custom-body Agent may omit `:model` and `:context`.

## Tool

```clojure
{:karcarthy/type :tool
 :name "web-search"
 :description "Search the web."
 :input ::search-input
 :output ::search-output
 :approval :never
 :enabled? #object[clojure.lang.IFn ...]
 :execute #object[clojure.lang.IFn ...]
 :source-form '(tool {...} [rt input] ...)}
```

Tool inputs are validated before approval. Tool outputs are validated before
they enter conversation state or become visible to the model. `:to-model-output` may
project a rich application result into a smaller model-visible value.

## Runtime

A Runtime is passed to agent programs, tools, resolvers, guardrails, and hooks:

```clojure
{:karcarthy/type :runtime
 :run-id "run_..."
 :agent-id "agent_..."
 :parent-id "agent_..."
 :environment {...}
 :limits {...}
 :usage {...}
 :conversation-state state-atom
 :model-transports {:transport-name transport}
 :cancel token
 :approval handler
 :evaluation-namespace 'karcarthy.eval.run_...}
```

The Runtime is an internal map. Public access goes through functions such as
`environment`, `emit!`, `invoke!`, and
`compile-agent!`, not direct mutation of this representation.

## Conversation state

The root model conversation is an explicit, versioned, plain-data snapshot:

```clojure
{:karcarthy/type :conversation-state
 :version 1
 :model {:transport :responses :provider :openai :id "gpt-5.6"}
 :messages [{:role :user :content "..."}
            {:role :assistant :content "..."}]
 :provider-state {:previous-response-id "resp_..."}}
```

`conversation-state?` recognizes this shape. A caller supplies it as the
`:state` Run option and receives the current snapshot in the Run result.

## Limits

```clojure
{:model-calls 100
 :input-tokens 1000000
 :output-tokens 100000
 :agent-depth 8
 :parallelism 6
 :generated-forms 20
 :deadline-ms 300000}
```

Limits and usage counters are shared across the root run tree. A child may
lower inherited ceilings but cannot raise them.

## Run

`run!` returns:

```clojure
{:karcarthy/type :run
 :id "run_..."
 :status :completed
 :agent "research-team"
 :input {...}
 :output {...}
 :usage {:model-calls 7
         :input-tokens 42000
         :output-tokens 3900
         :duration-ms 12800}
 :trace-id "trace_..."
 :state {:karcarthy/type :conversation-state ...}
 :events [...]
 :error nil}
```

Statuses:

| Status | Meaning |
| --- | --- |
| `:completed` | Output satisfied the Agent's contract. |
| `:cancelled` | Root or ancestor cancellation propagated to the run. |
| `:failed` | Compilation, model, tool, contract, policy, or program failure. |

Completion is distinct from evaluation quality. An evaluator may score a
completed output poorly without treating the harness run as a protocol failure.

## Events

Every event contains `:type`, `:time-ms`, `:run-id`, and trace lineage. Events for
specific steps add `:agent-id`, `:parent-id`, `:depth`, or
`:tool-call-id`.

Stable event families:

```clojure
:run/started          :run/completed
:run/failed           :run/cancelled

:agent/started        :agent/completed
:agent/failed         :agent/handoff

:model/requested
:model/text-delta    :model/tool-call-delta
:model/completed      :model/failed

:state/checkpointed

:tool/approval-requested
:tool/started         :tool/completed
:tool/failed

:program/read         :program/expanded
:program/checked      :program/evaluated
:program/failed
```

Observers may redact or export events as they are emitted. Configurable
in-runtime payload retention remains roadmap work.

## Failure

```clojure
{:karcarthy/type :failure
 :kind :contract
 :phase :tool-output
 :message "Tool output did not satisfy ::search-output"
 :data {...}
 :cause exception
 :recoverable? false}
```

Failure kinds include `:read`, `:expand`, `:evaluation`, `:context`,
`:contract`, `:model`, `:tool`, `:guardrail`, `:approval`, `:budget`,
`:deadline`, `:cancellation`, and `:program`.

## Mathematical view

Let a Clojure Agent program be `p`, a task be `τ`, and its stochastic execution
trace be:

```text
ξ ~ Exec(p, τ)
```

An optimizer may search program forms under resource constraints:

```text
p* = argmaxₚ E[R(τ, ξ) - λ·cost(ξ) - μ·latency(ξ)]
```

The trace and program forms are first-class experimental data. Research can
measure behavior, invariants, or regressions without assuming that a candidate
program is an improvement.
