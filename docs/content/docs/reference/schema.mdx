---
title: Data model
description: Agent, Tool, Session, Run, event, limit, and failure values.
---

karcarthy values are ordinary Clojure values. Agent and Tool values may contain
functions and compiled code; Run events and outputs can be projected to an
external serialization format by the application.

## Agent

```clojure
{:karcarthy/type :agent
 :name "researcher"
 :config {...}
 :definition-ns 'example.research
 :source-form '(agent {...})
 :expanded-form '(...)
 :body nil}
```

The public configuration surface is:

| Key | Meaning |
| --- | --- |
| `:name` | Stable identifier; `defagent` supplies it from the symbol. |
| `:description` | Human- and model-facing description. |
| `:model` | Model identifier and adapter-specific options, or a resolver function. |
| `:instructions` | Model-visible string or resolver function returning a string. |
| `:context` | Contract for the local, non-model-visible context value. |
| `:input` | Agent input contract. |
| `:tools` | Tool values or a resolver function. |
| `:output` | Agent output contract. |
| `:output-schema` | Explicit model-facing JSON Schema. |
| `:max-turns` | Maximum model turns; defaults to 20. |
| `:stop-when` | Predicate accepting call metadata and final-response data. |
| `:guardrails` | Input and output guardrail functions. |
| `:limits` | Limits narrower than the current Run's grant. |
| `:metadata` | Application metadata retained in configuration. |

A custom-body Agent may omit `:model` and `:instructions`.

## Tool

```clojure
{:karcarthy/type :tool
 :name "web-search"
 :description "Search the web."
 :config {...}
 :source-form '(tool {...} [input] ...)
 :expanded-form '(...)
 :execute #object[clojure.lang.IFn ...]}
```

Tool input is validated before approval and the call. Tool output is
validated before it becomes visible to the model. `:to-model-output` may
project a rich application result into a smaller model-visible value.

## Dynamic configuration

Dynamic instructions, model and Tool resolvers, and guardrails receive an
immutable map describing the current Agent call:

```clojure
{:run-id "run_..."
 :agent-id "agent_..."
 :parent-id "agent_..."
 :depth 1
 :context {...}
 :limits {...}
 :usage {...}
 :agent agent}
```

Agent and Tool bodies do not receive this map. They use `context`, `model!`,
`emit!`, and `compile-agent!` directly.

## Session

A Session implements `karcarthy.session/Session`:

```clojure
(session-id session)       ; stable string
(get-items session)        ; chronological messages
(add-items! session items) ; append
(pop-item! session)        ; remove newest
(clear-session! session)   ; remove all
```

Stored items are normalized message maps with roles such as `:user`,
`:assistant`, and `:tool`. A Session stores conversation history; it does not
serialize in-flight Run stacks or arbitrary effects. `memory-session` stores a
vector in an atom; durable implementations choose their own storage.

## Limits

```clojure
{:model-calls 100
 :input-tokens 1000000
 :output-tokens 100000
 :agent-depth 8
 :parallelism 6
 :generated-forms 20
 :deadline-ms 300000}
```

Limits and usage counters are shared across the root run tree. A child may
lower inherited ceilings but cannot raise them.

## Run

```clojure
{:karcarthy/type :run
 :id "run_..."
 :status :completed
 :agent "research-team"
 :input {...}
 :output {...}
 :usage {:model-calls 7
         :input-tokens 42000
         :output-tokens 3900
         :generated-forms 1
         :duration-ms 12800}
 :events [...]
 :error nil}
```

`status` is `:completed`, `:failed`, or `:cancelled`. Completion means the
output passed the harness contracts and guardrails; an external evaluator may
still score it poorly.

The Run `id` joins its events. A caller-owned Session holds conversation
history across Runs.

## Events

Every event contains `:type`, `:time-ms`, and `:run-id`. Agent-specific events
also carry `:agent-id`, `:parent-id`, and `:depth`; Tool events may carry a
`:tool-call-id`.

Stable event families include:

```clojure
:run/started          :run/completed       :run/failed
:run/cancelled

:agent/started        :agent/completed     :agent/failed

:model/requested      :model/text-delta    :model/tool-call-delta
:model/stream-event   :model/completed     :model/failed

:tool/approval-requested  :tool/approval-resolved
:tool/approval-reused
:tool/started         :tool/completed      :tool/failed

:session/updated

:program/read         :program/expanded    :program/checked
:program/evaluated    :program/failed
```

The Run's `:observe` callback receives events as they occur.

## Failure

```clojure
{:karcarthy/type :failure
 :kind :contract
 :phase :tool-output
 :message "Value does not satisfy contract for tool-output"
 :data {...}
 :recoverable? false}
```

Failures cover reading, expansion, evaluation, instructions, Sessions,
contracts, models, Tools, guardrails, approvals, budgets, deadlines,
cancellation, and custom programs. Failed and cancelled Runs retain failure
data and events but have no output.
