Decide whether karcarthy belongs in your stack. This page states the problem
karcarthy takes on, the design bet it makes, and what that bet buys and
costs. It is not a getting-started path; for that, see the
[Quickstart](/docs/quickstart).

## The problem

Agent frameworks tend to bundle two jobs. One is the inner loop: calling the
model, executing tools, managing sessions. The other is orchestration: which
agent runs when, on what input, and what happens to the result.

Bundling has two costs. The inner loop gets reimplemented, even though
Claude, Codex, the OpenAI runtime, and ACP-compatible agents already run
model and tool calls well. And orchestration gets trapped in host-language
code: a plan that exists only as Python or TypeScript objects cannot be
stored in a database, diffed against another plan, generated by a model, or
rewritten before it runs.

## The bet

karcarthy does orchestration only, and represents it as data. An agent is an
EDN map. A workflow is a tree of nodes, and every node is a map tagged with
`:karcarthy/type`. Running a single agent is delegated to a runner, a
protocol with one operation.

```text
karcarthy: workflow data, routing, fan-out, loops, result shape
runner:   model calls, tool calls, sessions, provider-specific runtime
```

The same workflow runs offline, through Claude, through Codex, through
OpenAI, through any ACP-compatible agent, or through a process runner.
karcarthy is not a replacement for agent SDKs; it keeps multi-agent
coordination inspectable while delegating single-agent execution to systems
that already do it.

## What the bet buys

- **Serializable plans.** Workflows are EDN values with a JSON mirror
  (`edn-schema`, `json-schema`). Store them, version them, diff them, send
  them across a wire.
- **Rewritable plans.** `karcarthy.rewrite` transforms workflow data before
  it runs: `agents` collects the agent leaves, `over` rewrites matching
  nodes, and `configure` stamps a model or instruction suffix onto every
  agent. Rewrites never call runners and never use `eval`.
- **Model-authorable plans.** `karcarthy.self` parses agent-emitted workflows
  with `clojure.edn` — data only, never `eval` — and the `dynamic` node lets
  an agent define, patch, remove, call, spawn, and complete work mid-run.
- **Deterministic offline tests.** `mock-runner` runs any workflow with no
  network. karcarthy's own test suite runs offline.

## What it costs

karcarthy deliberately omits three things. The runner owns all of them.

- **No durable execution.** `run` is an in-process interpreter. There are no
  checkpoints; if the process dies mid-run, the run is gone.
- **No memory layer.** Nothing persists between runs. `continue` forwards a
  runner session id within a run, but conversation state lives in the runner.
- **No tool model.** karcarthy never defines, hosts, or executes tools. An
  agent's `:tools` is an allowlist passed to runners that support tool
  selection.

If you need durable workflows, persistent memory, or a framework-hosted tool
runtime, pair karcarthy with systems that provide them, or use those systems
directly.

## When to use it

| Need | Use karcarthy when | Prefer another system when |
| --- | --- | --- |
| Agent loop | You already have an SDK, CLI, Clojure function, or process command that can run one agent. | You need a batteries-included agent runtime first. |
| Workflow shape | You want the graph to be EDN/JSON data that can be stored, generated, diffed, and rewritten. | You want workflows only as Python/TypeScript objects. |
| Tools | Tool execution belongs to Claude, Codex, OpenAI, ACP/MCP config, or your process runner. | You want karcarthy itself to define and host tool schemas. |
| Multi-agent patterns | You want `pipe`, `branch`, `delegate`, `reduce`, `revise`, `route`, `continue`, and `dynamic` as portable values. | You want a mature framework-native graph runtime and UI. |
| Testing | You want deterministic offline orchestration tests with `mock-runner`. | You need provider-level simulation of model/tool behavior. |

## Comparison

| System | Strong at | karcarthy relationship |
| --- | --- | --- |
| Claude | Running single agents with tools, MCP, sessions, and CLI ergonomics. | Use `claude-runner`. |
| Codex | Running coding agents in non-interactive CLI mode. | Use `codex-runner`. |
| OpenAI | Agent loop, handoffs, tools, tracing, and structured SDK runtime. | Use `openai-runner` or mirror routing with `route`. |
| ACP agents | Editor-agnostic agent protocol over stdio JSON-RPC. | Use `acp-runner`; reuse one process across runs with `karcarthy.runner.acp/connect!` and `close!`. |
| DSPy | Signatures, metrics, and prompt/program optimization. | Complementary; karcarthy does not optimize prompts today. |
