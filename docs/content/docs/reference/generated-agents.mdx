---
title: Generated Agents
description: Source format, compilation pipeline, evaluation namespace, limits, events, and security for model-created Agents.
---

The zero-arity `(k/agent)` Tool is the model-facing use of karcarthy's
homoiconic representation. It accepts the same Clojure Agent form a developer
would write, treats that form as data through reading and macroexpansion, then
evaluates and runs the resulting Agent. Applications can use the same
compilation path directly with `compile-agent!`.

## Tool input

The Tool is named `agent` and accepts:

```json
{
  "source": "(agent {:name \"reviewer\" ...})",
  "input": "Review the proposed change."
}
```

`source` must contain exactly one top-level Clojure form whose evaluation
returns an Agent. `input` is passed to that Agent unchanged. Use a prompt string
for a model-backed Agent or any value accepted by a Clojure-body Agent.

## Accepted Agent forms

A standard model-backed Agent:

```clojure
(agent
 {:name "generated-reviewer"
  :model {:transport :responses :id "gpt-5.6"}
  :instructions "Review the change and return concrete findings."
  :input string?
  :output string?})
```

An Agent with a Clojure body:

```clojure
(agent
 {:name "generated-review-team"
  :input map?
  :output string?}
 [change]
 (let [reviews (->> [security-reviewer api-reviewer]
                    (mapv #(future (run! % change)))
                    (mapv deref)
                    (mapv :output))]
   (:output (run! editor {:change change :reviews reviews}))))
```

Symbols such as `security-reviewer` and `editor` must resolve in the evaluation
namespace.

## Compilation pipeline

```text
source string
    │
    ▼
read-agent-form
    │ Clojure form
    ▼
check-agent-form!
    │ macroexpanded, checked representation
    ▼
eval-agent-form!
    │ Agent value
    ▼
agent?
    │
    ▼
verified Agent
```

### Read

`read-agent-form` disables reader evaluation and rejects empty input or more
than one top-level form.

### Expand and check

`check-agent-form!` macroexpands executable positions in the current Run's
evaluation namespace. It preserves quoted forms and records the original and
expanded forms. Agent configuration and referenced contracts are validated
when the checked form is evaluated.

Clojure is dynamically typed. Boundary contracts validate Agent input/output,
Tool input/output, and local context; they are not a whole-program static type
system.

### Evaluate and verify

`eval-agent-form!` evaluates the checked form and requires the result to satisfy
`agent?`. The resulting Agent retains its source form, expanded form, and
definition namespace.

### Run the Agent

The `agent` Tool runs the generated Agent inside the current Run. It
inherits shared usage, deadline, cancellation, approvals, event recording, and
parallelism policy.

## Evaluation namespace

By default, generated code evaluates in the defining namespace of the parent
Agent. A root Run may select another namespace:

```clojure
(k/run! architect task
        {:evaluation-namespace 'example.generated})
```

The namespace receives Clojure core plus the public Agent-program forms needed
for composition, including `agent`, `tool`, `run!`, `context`, `model!`, and
`emit!`. Application Agents, Tools, macros, and contracts must be referred
or interned in that namespace before generated code can use them.

## Recursion and limits

A generated Agent can include `(agent)` in its own Tool set and create another
Agent. The root Run enforces:

```clojure
{:agent-depth 4
 :generated-forms 10
 :model-calls 50
 :input-tokens 500000
 :output-tokens 50000
 :parallelism 4
 :deadline-ms 300000}
```

`generated-forms` is charged before compilation, including failed attempts.
All descendants share token and model-call counters.

## Events

Compilation emits:

```clojure
:program/read
:program/expanded
:program/checked
:program/evaluated
:program/failed
```

These events include Run and Agent lineage. The read event contains exact
source text; expanded and evaluated events identify the forms and namespace.

## Failure behavior

Read, expansion, evaluation, and Agent-verification errors are returned as
structured Tool failures. The built-in Agent Tool does not retry or silently
repair invalid source. A parent Agent may make another Tool call with corrected
source within the remaining Run limits.

## Security

Generated source is intentionally evaluated as JVM Clojure. Reader evaluation
is disabled, but evaluated code has the capabilities of its namespace,
classloader, process, filesystem permissions, and network environment.

Treat `(k/agent)` as a code-execution capability. For untrusted models or task
input, run the harness in an appropriate OS process, container, or sandbox and
expose only the namespaces and Tools the task requires.

## Direct compilation API

Custom Agent bodies can use the same phases directly:

```clojure
(def form (k/read-agent-form source))
(def checked (k/check-agent-form! form))
(def generated (k/eval-agent-form! checked))

;; Equivalent convenience function:
(def generated (k/compile-agent! source))
```

Use the zero-arity `agent` Tool for normal model-driven creation and the direct
API when an application controls source acquisition or retry policy.
