Use the facade namespace for normal application code:

```clojure
(require '[karcarthy :as k])
```

## Agents

### `agent`

```clojure
(k/agent spec)
```

Builds an agent spec map. The map form is the primary API.

| Key | Type | Required | Meaning |
| --- | --- | --- | --- |
| `:name` | string | yes | Stable agent name used in results, logs, and mock responses. |
| `:instructions` | string | yes | Developer instructions for this agent spec. |
| `:description` | string | no | Short human-facing description. |
| `:model` | string | no | Runner-specific model selector. |
| `:tools` | vector of strings | no | Tool allowlist. Tools must already exist in the selected runner environment. |
| `:config` | map | no | Runner-specific configuration. |

Returns:

```clojure
{:karcarthy/type :agent
 :name "reviewer"
 :description "Use for launch-risk review."
 :instructions "Find launch risks."
 :model "sonnet"
 :tools ["mcp__feedback__search"]
 :config {:temperature 0.2}}
```

Example:

```clojure
(k/agent
 {:name "reviewer"
  :instructions "Find launch risks and propose mitigations."
  :model "sonnet"
  :tools ["mcp__feedback__search"]})
```

To derive a variant from existing agent data:

```clojure
(k/agent {:from reviewer
          :model "gpt-5.2"})
```

### `defagent`

```clojure
(k/defagent symbol spec)
```

Defines a var containing an agent. The symbol name becomes the agent `:name`.

```clojure
(k/defagent researcher
  {:instructions "Research the question and cite sources."
   :model "sonnet"})

(k/defagent claude-researcher
  {:from researcher
   :model "sonnet"})
```

### `agent?` and `explain-agent`

```clojure
(k/agent? value)
(k/explain-agent value)
```

`agent?` returns true for well-formed agent maps. `explain-agent` returns a
human-readable validation message or nil.

## Subagents

### `subagent`

```clojure
(k/subagent name description instructions & options)
```

Builds runner-native delegation config. A subagent is not a workflow node; it
is made available inside one runner call when the runner supports native
subagents or handoffs.

```clojure
(def security-reviewer
  (k/subagent "security-reviewer"
    "Use for auth, secrets, injection, permission, and data exposure review."
    "Review like a security owner. Return concrete findings with file refs."
    :tools ["Read" "Grep" "Glob"]
    :model "sonnet"
    :effort :high))
```

Common options:

| Option | Meaning |
| --- | --- |
| `:model` | Runner-native model selector. |
| `:tools` | Tool allowlist. |
| `:disallowed-tools` | Tool denylist where supported. |
| `:mcp-servers` | MCP servers available to the subagent. |
| `:permission-mode` / `:sandbox-mode` | Provider permission or sandbox controls. |
| `:max-turns` | Turn cap for providers that support it. |
| `:skills` | Skills preloaded or enabled by providers that support skills. |
| `:initial-prompt` | First message sent when the subagent starts, where supported (Claude `initialPrompt`). |
| `:memory` | Provider memory setting (Claude `memory`). |
| `:effort` / `:reasoning-effort` | Provider reasoning effort controls. |
| `:background?` / `:isolation` | Provider concurrency or isolation controls. |
| `:color` | Display color where supported (Claude `color`). |
| `:nicknames` | Vector of nickname candidates (Codex `nickname_candidates`). |
| `:hooks` | Hook configuration passed through where supported (Claude `hooks`). |
| `:config` | Extra provider config; merged beneath the generated Codex custom-agent config keys. |

Options with nil values are generally omitted from the built map, and non-nil
falsy values (for example `:background? false`) are kept. The coerced options
are the exception: the collection options (`:tools`, `:disallowed-tools`,
`:skills`, `:nicknames`) turn an explicit nil into `[]`, and `:background?`
turns an explicit nil into `false`. Unknown option keys throw.

Runner lowering:

| Runner | Native primitive |
| --- | --- |
| `claude-runner` | Claude Code `--agents` JSON. |
| `codex-runner` | None at run time; `codex-runner` ignores `:subagents` and runs one leaf agent per `codex exec` call. Lower with `karcarthy.runner.codex/subagent->config` instead (below). |
| `openai-runner` | OpenAI Agents SDK `handoffs`. |
| `karcarthy.runner.codex/subagent->config` | Codex custom agent config keys for `.codex/agents/` or `~/.codex/agents/`. |
| `process-runner` | Unsupported; process commands stay fixed runner config. |

## Workflows

### `pipe`

```clojure
(k/pipe & steps)
```

Runs steps in sequence. Each step receives the previous result's `:text`.

```clojure
(k/pipe researcher summarizer)
```

### `step`

```clojure
(k/step f & {:keys [name call?]})
```

Runs host Clojure code inside a workflow. By default, `f` receives the flowing
input string. With `:call? true`, `f` receives the step call map:
`{:input input :options options :node node}`.

```clojure
(k/pipe
 writer
 (k/step clojure.string/trim :name "trim")
 reviewer)
```

`step` may return text, a partial result map, or a full result map. It is
Clojure-local and not EDN/JSON portable.

### `branch`

```clojure
(k/branch workflows & {:keys [max-concurrency]})
```

Runs every workflow in the collection on the same input.

```clojure
(k/branch [product-reviewer engineering-reviewer security-reviewer]
          :max-concurrency 4)
```

### `delegate`

```clojure
(k/delegate planner worker & {:keys [max-concurrency]})
```

Reads planner EDN `{:subtasks [...]}` and runs the worker once per subtask.

```clojure
(k/delegate planner worker :max-concurrency 8)
```

### `reduce`

```clojure
(k/reduce source reducer & {:keys [max-concurrency]})
```

Runs branch/delegate work, then sends an EDN result summary to a reducer
workflow. `source` is a branch or delegate node, or a plain collection of
workflows, which is wrapped as `(k/branch source)` first.

```clojure
(k/reduce
  (k/branch [product-reviewer engineering-reviewer])
  brief-writer)

(k/reduce [product-reviewer engineering-reviewer] brief-writer)
```

The reducer receives EDN shaped like `{:input ... :subtasks ... :results [...]}`.

### `revise`

```clojure
(k/revise worker evaluator & {:keys [max-rounds]})
```

Runs a worker/evaluator loop until accepted or the round limit is reached. An
exhausted, unaccepted loop returns `:ok? false` and `:error :not-accepted`.

```clojure
(k/revise draft-writer critic :max-rounds 3)
```

Evaluator workflows reply with EDN verdict maps:

```clojure
{:accept? true}
{:accept? false :feedback "specific revision feedback"}
```

### `route`

```clojure
(k/route source routes & {:keys [default]})
```

With a route map, `source` replies with EDN `{:route key}` and the selected
workflow receives the original input.

```clojure
(k/route classifier
         {:launch launch-brief
          :incident incident-plan}
         :default launch-brief)
```

### `continue`

```clojure
(k/continue source continuation & {:keys [prompt]})
```

The continuation receives the source result text, or the `:prompt` override
when given. When the source result carries a `:session-id`, `continue` passes
it as the `:resume` option on the continuation's runner call, so runners with
session support keep the conversation (`claude-runner` turns it into
`--resume <id>`; `acp-runner` into `session/resume` or `session/load`). This
session threading is what distinguishes `continue` from a two-step `pipe`.

```clojure
(k/continue triage-agent specialist-agent)
```

### `dynamic`

> **Experimental.** The dynamic op protocol and prompt format may change
> between releases.

```clojure
(k/dynamic agent & {:keys [max-steps]})
```

Builds a workflow whose agent emits EDN ops while the run is in progress.
Run it with the same `run` function as any other workflow:

```clojure
(k/run {:runner runner
        :workflow (k/dynamic workflow-agent :max-steps 25)
        :input "audit this migration"})
```

Supported ops are:

```clojure
{:op :define
 :agent {:name "reader"
         :instructions "..."
         :model "sonnet"
         :tools ["Read"]
         :config {}}}
{:op :define :name "review" :workflow WORKFLOW}
{:op :patch :agent "reader" :merge {:instructions "..."}}
{:op :patch :workflow "review" :merge MAP}
{:op :remove :agent "reader"}
{:op :remove :workflow "review"}
{:op :call :agent "reader" :input "..."}
{:op :call :workflow "review" :input "..."}
{:op :spawn :agent "reader" :inputs ["a" "b"]}
{:op :spawn :workflow "review" :inputs ["a" "b"]}
{:op :complete :text "final answer"}
```

`:spawn` fans the target agent or named workflow out over `:inputs`; the op may
also carry `:max-concurrency` to cap parallel calls.
Failed `:call` or `:spawn` work must be retried successfully before `:complete`.

Use `agent-ref` and `workflow-ref` inside workflows defined during a dynamic
run when later patches should affect later calls.

```clojure
(k/agent-ref "reader")
(k/workflow-ref "review")
```

### `workflow?` and `defworkflow`

```clojure
(k/workflow? value)
(k/defworkflow symbol workflow)
```

Use these when storing, generating, or publishing workflows from application
code.

## Schemas

### `edn-schema` and `json-schema`

```clojure
k/edn-schema
k/json-schema
```

These are plain data reference values for the public workflow shapes. Use
`edn-schema` when documenting or generating EDN workflows, and `json-schema`
for the CLI JSON bridge.

```clojure
(keys k/edn-schema)
;=> (:agent :result :workflow :pipe :branch :delegate ...)

(get k/json-schema "$ref")
;=> "#/$defs/workflow"
```

## Rewrites

Workflow rewrites operate on EDN before execution. They never call runners and
never use `eval`.

### `agents`

```clojure
(k/agents workflow)
```

Returns the agent leaves in traversal order.

### `over`

```clojure
(k/over pred f workflow)
```

Applies `f` to every value in `workflow` where `pred` returns true, then
validates the rewritten workflow.

```clojure
(k/over k/agent?
        #(update % :instructions str "\n\nState assumptions before final answer.")
        workflow)
```

### `configure`

```clojure
(k/configure options workflow)
```

Applies common agent configuration in one pass. Supported keys:

| Key | Meaning |
| --- | --- |
| `:model` | Model selector string. |
| `:instructions/suffix` | Text appended to every agent instruction. |

```clojure
(def production-workflow
  (->> workflow
       (k/configure {:model "claude-sonnet-4"
                     :instructions/suffix "State assumptions before final answer."})))
```

## Execution

### `run`

```clojure
(k/run {:runner runner
        :workflow workflow
        :input input
        :options options})
```

Interprets workflow data through the runner passed to `run`.

| Key | Meaning |
| --- | --- |
| `:runner` | A value implementing the runner protocol. |
| `:workflow` | An agent or workflow node. |
| `:input` | The initial value given to the workflow. |
| `:options` | Runtime controls and runner overrides. Interpreter options include `:observe`, `:edn-retries`, run-wide `:max-concurrency`, `:run-timeout-ms`, and a zero-argument `:cancel?` predicate. |

Input rendering:

| Input | Runner prompt |
| --- | --- |
| `"review this"` | `"review this"` |
| `{:prompt "review this" :ticket 42}` | `"review this\n\nINPUT EDN:\n{:ticket 42}"` |
| `{:ticket 42}` | `"{:ticket 42}"` |

```clojure
(k/run {:runner (k/mock-runner)
        :workflow (k/pipe researcher summarizer)
        :input {:task "map" :subject "agent framework space"}})
```

Returns a result map:

```clojure
{:karcarthy/type :result
 :ok? true
 :agent "summarizer"
 :text "..."}
```

Top-level `run` throws for malformed input. Composite nodes convert child
exceptions into not-ok result maps.

### Observability

Pass `:observe` to receive OpenTelemetry-compatible span events as plain data:

```clojure
(k/run {:runner (k/mock-runner)
        :workflow (k/pipe researcher summarizer)
        :input "map the agent framework space"
        :options {:observe (fn [event] ...)}})
```

Events contain span ids, parent span ids, names, attributes, paths, status, and
duration metadata. They do not include prompt text or result text, and
observation is always best-effort.

```clojure
{:karcarthy/type :event
 :event :start                  ; :start, :finish, or :error
 :kind :workflow                ; or :agent
 :name "karcarthy.workflow.pipe"
 :span/kind :internal
 :span/id "..."
 :parent/span-id "..."
 :path [:steps 0]
 :attributes {"karcarthy.kind" "workflow"
              "karcarthy.workflow.type" "pipe"
              "karcarthy.path" "[:steps 0]"}}
```

For OpenTelemetry, start a span on `:start`, end it on `:finish` or `:error`,
use `:name` as the span name, and attach `:attributes`. karcarthy does not take
an OpenTelemetry dependency.

## Runners

### `mock-runner`

```clojure
(k/mock-runner)
(k/mock-runner respond-fn)
```

Offline deterministic runner for tests and examples.

```clojure
(k/mock-runner
 (fn [{:keys [agent prompt]}]
   (str "[" (:name agent) "] " prompt)))
```

### `fn-runner`

```clojure
(k/fn-runner f)
(k/fn-runner f options)
```

Wraps a Clojure function as a runner for agent specs. By default, the function
receives only the input string. With `{:call? true}`, it receives the agent call map:
`{:agent agent :input input :options options}`.

```clojure
(k/fn-runner clojure.string/upper-case)

(k/fn-runner
 (fn [{:keys [agent input]}]
   (str (:name agent) ": " input))
 {:call? true})
```

### `claude-runner`

```clojure
(k/claude-runner)
(k/claude-runner default-options)
```

Runs agent calls through Claude. The current implementation drives `claude -p`.
The model comes from the agent spec's `:model` field (lowered to `--model`), or
from a runner-level `:model` override.

```clojure
(k/claude-runner {:max-turns 4
                  :subagents [security-reviewer]})

(k/agent {:name "reviewer"
          :instructions "Find launch risks."
          :model "haiku"})
```

Common options:

| Option | Meaning |
| --- | --- |
| `:claude-bin` | Claude CLI executable; defaults to `"claude"`. |
| `:output-format` | `"json"` (default), `"text"`, or `"stream-json"`. |
| `:system-prompt-mode` | `:append` (default, `--append-system-prompt`) or `:replace` (`--system-prompt`). |
| `:max-turns` | Integer cap on agent turns. |
| `:permission-mode` | Permission mode, e.g. `:acceptEdits`, `:bypassPermissions`, `:default`. |
| `:resume` | Session id to resume (`--resume <id>`); the agent sees prior context. |
| `:continue?` | Resume the most recent session (`--continue`). |
| `:partial-messages?` | Stream token-level chunks (`--include-partial-messages`); only meaningful with `"stream-json"`. |
| `:subagents` | Vector of `k/subagent` values lowered to Claude Code `--agents` JSON. |
| `:prompt-via` | `:argv` (default) passes the prompt as an argument; `:stdin` pipes it to the process instead - use `:stdin` for prompts that can exceed OS argv limits, e.g. reduce fan-in. |
| `:extra-args` | Vector of strings appended verbatim (escape hatch for `--add-dir`, `--mcp-config`, `--disallowedTools`, ...). |
| `:dir` | Working directory for the `claude` process. |
| `:timeout-ms` | Force-kill a hung run after this many milliseconds. |
| `:on-event` | fn called with each stream event; switches the run to `--output-format stream-json`. |

### `codex-runner`

```clojure
(k/codex-runner)
(k/codex-runner default-options)
```

Runs agent calls through Codex. The current implementation drives `codex exec`.
The full prompt - agent name, instructions, and the flowing workflow input -
is written to stdin, with `-` as the positional prompt, so large inputs are
neither dropped nor limited by argv size.

```clojure
(k/codex-runner {:dir "."
                 :model "gpt-5"
                 :sandbox :read-only
                 :timeout-ms 120000})
```

Common options:

| Option | Meaning |
| --- | --- |
| `:codex-bin` | Codex CLI executable; defaults to `"codex"`. |
| `:dir` / `:cwd` | Working directory for the Codex process and `--cd`. |
| `:model` | Model override; takes precedence over the agent's `:model`. |
| `:sandbox` | Codex sandbox mode; defaults to `:read-only`. |
| `:color` | Color mode (`--color`); defaults to `:never`. |
| `:ephemeral?` | Pass `--ephemeral` (run without persisting a session); defaults to true. |
| `:skip-git-repo-check?` | Pass `--skip-git-repo-check`; defaults to true. |
| `:extra-args` | Additional `codex exec` args appended before the prompt. |
| `:env` | Extra environment variables. |
| `:timeout-ms` | Kill the subprocess after this many milliseconds. |
| `:trim?` | Trim stdout; defaults to true. |

### `acp-runner`

```clojure
(k/acp-runner {:command ["some-acp-agent"]})
(k/acp-runner command options)
```

Runs one Karcarthy agent call through an Agent Client Protocol subprocess over
stdio JSON-RPC. The ACP agent is the external process; Karcarthy folds the
agent spec's instructions into the prompt turn.

```clojure
(def acp
  (k/acp-runner {:command ["my-acp-agent"]
                 :cwd "."
                 :permission-policy :reject}))

(k/run {:runner acp
        :workflow (k/agent {:name "reviewer"
                            :instructions "Review this diff."
                            :model "sonnet"})
        :input "Check the current branch."})
```

Common options:

| Option | Meaning |
| --- | --- |
| `:command` | argv vector for the ACP agent process. |
| `:connection` | a connection from `karcarthy.runner.acp/connect!` reused across runs; without it, each run spawns and tears down its own agent process. |
| `:cwd` / `:dir` | working directory for `session/new`. |
| `:resume` | session id to resume, via ACP `session/resume` or `session/load` depending on agent capabilities; throws when the agent supports neither. Replayed history from `session/load` stays in `:raw :updates`; the returned `:text` is only the new turn. |
| `:mcp-servers` | ACP MCP server specs passed during session setup. `http`/`sse` specs require the agent's matching `mcpCapabilities` flag; stdio specs need no capability. |
| `:client-capabilities` | ACP client capabilities to advertise. Defaults to none. File reads/writes and terminals are served only when their corresponding capability is enabled. |
| `:auth-method` | id of an authentication method advertised during initialization; authentication runs before session creation. |
| `:auth-method-selector` | fn of the advertised authentication method vector, returning an id. |
| `:prompt-content` | vector of additional ACP content blocks appended to the text prompt; image, audio, and embedded resource blocks are capability-gated. |
| `:config-options` | session config option id/category to advertised value map; refreshed options are used after each change. |
| `:strict-config?` | throw if a requested config option is unavailable. |
| `:permission-policy` | `:reject` default, `:allow`, or `:cancel`. |
| `:permission-handler` | fn for ACP `session/request_permission`. |
| `:request-handler` | fn for custom agent-to-client JSON-RPC requests. |
| `:on-event` | fn called with each ACP message received. |
| `:timeout-ms` | max wait per ACP message before sending `session/cancel` for an active session. |
| `:cancel-grace-ms` | wait after cancellation before force-killing; defaults to 1000 ms. |

Session config: agents can carry ACP config under
`{:config {:acp {:config-options {...}}}}`, merged beneath the runner's
`:config-options`. The agent spec's `:model` is applied through ACP
`session/set_config_option` when the session advertises a `model` config
option, unless the merged config options already include a model entry.
`:strict-config?` applies to all of these.

Results map the ACP `stopReason` onto `:ok?`: only an explicit `end_turn` is
success; a missing reason, `refusal`, `cancelled`, `max_tokens`, and
`max_turn_requests` yield `{:ok? false}` with an `:error`, keeping
`:stop-reason` and partial `:text`. A timeout first requests `session/cancel`;
any permission request received while cancellation is in flight is answered
with `cancelled`.

#### Connection reuse: `connect!` and `close!`

`karcarthy.runner.acp/connect!` starts the ACP agent process and initializes
the protocol once. Passing the returned connection as `:connection` reuses it
across runs, so each run pays one `session/new` and one prompt turn instead of
a process spawn plus `initialize`. `close!` shuts the connection down.

```clojure
(require '[karcarthy.runner.acp :as acp])

(def conn (acp/connect! {:command ["some-acp-agent"]}))
(def runner (k/acp-runner {:connection conn}))
;; ... many runs ...
(acp/close! conn)
```

Prompt turns on one connection are serialized; for parallel branches, give each
worker its own connection. When the agent advertises
`sessionCapabilities.close`, each run closes its session after the prompt turn.
Malformed wire messages mark a reused connection unusable so a later run fails
immediately instead of reading a desynchronized stream.

Advertising `{:terminal true}` opts into running agent-requested commands on
the client machine. Karcarthy implements the full ACP terminal lifecycle,
bounds captured output by `outputByteLimit`, preserves valid UTF-8 when
truncating, and kills unreleased processes when the prompt turn ends.

### `process-runner`

```clojure
(k/process-runner command)
(k/process-runner command options)
```

Wraps a fixed local command. An argv vector executes directly; a string executes
through the configured shell. The command receives input on stdin and stdout
becomes result text.

```clojure
(k/process-runner ["tr" "a-z" "A-Z"])
(k/process-runner "tr a-z A-Z")
(k/process-runner ["ollama" "run" "llama3.2"]
                  {:timeout-ms 120000})
```

Options:

| Option | Meaning |
| --- | --- |
| `:trim?` | Trim stdout; defaults to true. |
| `:env` | Extra environment variables. |
| `:dir` | Working directory. |
| `:timeout-ms` | Kill the command after this many milliseconds. |
| `:shell` | argv prefix used for string commands; defaults to `["sh" "-c"]`. Deliberately not a login shell: profile scripts that print to stdout would corrupt the result text. Pass `["bash" "-lc"]` when a command needs the login environment. |

`process-runner` does not accept a function of `k/agent`. Different commands
mean different process runner values; choose one before calling `run`.

### `openai-runner`

```clojure
(k/openai-runner)
(k/openai-runner default-options)
```

## Results

```clojure
(k/result map)
(k/ok? result)
```

`result` fills in the standard result tag and defaults `:ok?` to true.

```clojure
(k/result {:agent "reviewer" :text "ready"})
;=> {:karcarthy/type :result
;    :ok? true
;    :agent "reviewer"
;    :text "ready"}
```

Composite workflows may add `:results`, `:subtasks`, `:rounds`, `:accepted?`,
or `:label`.
