title: Library API
description: Macros and functions for the native Clojure Agent harness.

The common API is exposed from `(require '[karcarthy :as k])`.

## Agents

### `agent`

```clojure
(k/agent config)
(k/agent config [input] body ...)
(k/agent)
```

With a config, constructs an Agent value and retains its original and expanded
forms. Without a body, the Agent uses the standard model/Tool loop. A
model-backed Agent requires `:name`, `:model`, and `:instructions`. A custom-body
Agent requires `:name` but may omit model configuration.

With no arguments, returns the model-facing `agent` Tool. That Tool accepts
`{:source <one (agent ...) form> :input <value>}`, compiles the form, verifies
that it produced an Agent, and runs it in the current Run. See
[Let the model define an Agent](/docs/agents#let-the-model-define-an-agent).

Supported configuration keys are:

```clojure
{:name "researcher"
 :description "Research one question."
 :model {:transport :responses :id "gpt-5.6"}
 :instructions "Research carefully."
 :context map?
 :input string?
 :tools [search-tool]
 :output ::report
 :output-schema {...}
 :max-turns 20
 :stop-when final-acceptable?
 :guardrails {:input [safe?] :output [grounded?]}
 :limits {:model-calls 30}
 :metadata {...}}
```

`:model`, `:instructions`, and `:tools` may be functions of immutable call
metadata. Instructions must resolve to a string. `:context` validates the local
context value; it is not model-visible prompt text.

For a model-backed Agent, the `run!` input is normally a prompt string. A
Clojure-body Agent receives its input unchanged. `:input` optionally validates
that boundary in either case.

### `defagent`

```clojure
(k/defagent name config)
(k/defagent name config [input] body ...)
```

Defines a var containing an Agent. The symbol supplies the default `:name`.
This is definition syntax for the same value constructed by `agent`, not a
different Agent type.

### Inspection

```clojure
(k/agent? value)
(k/source-form agent)
(k/expanded-form agent)
```

## Tools

### `tool` and `deftool`

```clojure
(k/tool config [input] body ...)
(k/deftool name config [input] body ...)
```

A Tool requires `:name`, `:description`, and `:input`. `deftool` supplies the
name from its symbol. Optional keys are `:input-schema`, `:output`,
`:output-schema`, `:approval`, `:enabled?`, `:guardrails`, `:timeout-ms`,
`:retry`, `:to-model-output`, and `:metadata`.

Tool arguments are validated before approval and the call. Tool results are
validated before they are returned to the model. `tool?` identifies Tool
values. Tool bodies have exactly one `[input]` binding.

### Hosted tools

```clojure
(k/hosted-tool :responses {:type "web_search"})
(k/responses-web-search)
(k/responses-web-search {:search_context_size "low"})
```

A hosted Tool is executed by its matching provider endpoint, not by local
Clojure code. `hosted-tool?` identifies this value.

### `as-tool`

```clojure
(k/as-tool agent)
(k/as-tool agent options)
```

Projects an Agent as a Tool while the parent retains control. Options may
override `:name`, `:description`, `:input`, `:output`, and `:approval`. A
`:context` function receives the parent context and Tool input and returns the
local context for the nested Agent.

## Running agents

### `run!`

```clojure
(k/run! agent input)
(k/run! agent input options)
```

Starts a root Run, validates local context and Agent input, calls the Agent,
and returns a Run value.

```clojure
{:context {...}
 :session session
 :limits {...}
 :observe event-fn
 :approval approval-handler
 :cancel cancellation-token
 :metadata {...}
 :model-transports {...}
 :evaluation-namespace 'example.generated}
```

Runs are stateless unless `:session` is supplied.

### Run from an Agent body

```clojure
(def other-run (k/run! other-agent input))
(:output other-run)
```

Agent bodies use the same `run!` function as application code. Each call starts
a complete, independent Run with separate limits, events, context, Session, and
cancellation. Clojure's normal functions, conditionals, `future`, and `deref`
provide composition. Agents entered through `as-tool` or `(agent)` instead
remain inside the current Run.

### `context`

```clojure
(k/context)
```

Returns the local, non-model-visible context value for the current Agent call.

## Sessions

```clojure
(def session (k/memory-session))
(def named (k/memory-session {:id "session_123" :items []}))

(k/session? session)
(k/session-id session)
(k/get-items session)
(k/add-items! session messages)
(k/pop-item! session)
(k/clear-session! session)
```

A Session is an ordered conversation store shared across root Agent runs.
`memory-session` is thread-safe and process-local. Durable stores implement the
`karcarthy.session/Session` protocol; karcarthy does not impose a file or
database format.

## Instructions and prompts

```clojure
(def instructions
  (k/prompt
   (k/system-prompt)
   (k/prompt-file "AGENTS.md")
   (fn [{:keys [context]}]
     (str "Current user: " (:user-id context)))))
```

`prompt` joins strings and call-metadata functions. Each function returns a
string or nil. `prompt-file` reads the named UTF-8 file. `system-prompt` reads
the packaged `resources/karcarthy/system.md`.

## Generated Agents

```clojure
(def form (k/read-agent-form source))
(def checked (k/check-agent-form! form))
(def agent (k/eval-agent-form! checked))
(def agent (k/compile-agent! source))
```

`read-agent-form` reads one form with reader evaluation disabled.
`check-agent-form!` macroexpands executable positions in the current Run's
evaluation namespace. `eval-agent-form!` intentionally evaluates the checked
form as JVM Clojure and verifies the resulting Agent. `compile-agent!` composes
the three phases and emits program lifecycle events. See
[Generated Agents](/docs/reference/generated-agents).

## Model access

### `model!`

```clojure
(k/model! normalized-request)
```

Calls the active model transport for a custom Agent body. Most Agents use the
standard loop or run another model-backed Agent.

### `fake-model`

```clojure
(k/fake-model (fn [request] {:type :final :output "done"}))
```

Creates a deterministic in-process model for tests and examples.

### Responses-compatible model

`:transport :responses` selects the built-in HTTP/SSE adapter:

```clojure
{:transport :responses
 :provider :openai
 :id "gpt-5.6"}
```

A Responses-compatible gateway changes endpoint and credentials without
changing harness semantics:

```clojure
{:transport :responses
 :provider :anthropic
 :id "anthropic/claude-model"
 :base-url "https://ai-gateway.vercel.sh/v1"
 :api-key-env "AI_GATEWAY_API_KEY"
 :headers {"X-Application" "karcarthy"}}
```

Supported endpoint options include `:base-url`, `:path`, `:url`, `:api-key`,
`:api-key-env`, `:headers`, `:auth?`, `:timeout-ms`, `:stream`, and
`:provider-options`. Direct OpenAI uses `RESPONSES_API_KEY` or
`OPENAI_API_KEY` when no credential option is supplied.

## Local capabilities

```clojure
(def tools (k/local-tools {:cwd "/workspace"}))
```

Returns ordinary `read`, `write`, `edit`, `bash`, and `search` Tools. Options
configure approval and output/time bounds. `search` requires ripgrep. `bash`
is not a sandbox.

## MCP

```clojure
(def connection
  (k/connect-mcp! {:name "server"
                   :command "server-command"
                   :args ["--stdio"]
                   :cwd "/workspace"}))
(def tools (k/mcp-tools connection))
(k/close-mcp! connection)
```

The client implements MCP `2025-11-25` over stdio. `mcp-tools` accepts
`:prefix` and `:approval`; approval defaults to `:always`. Commands execute
directly rather than through a shell. Always close the connection.

## ACP

```clojure
(k/serve-acp!
 {:models ["gpt-5.6" "gpt-5.5"]
  :agent (fn [{:keys [cwd mcp-tools model-id]}]
           (k/agent
            {:name "acp-agent"
             :model {:transport :responses :id model-id}
             :instructions (k/system-prompt)
             :tools (into (k/local-tools {:cwd cwd}) mcp-tools)
             :output string?}))})
```

Serves ACP v1 over stdio or supplied `:in` and `:out` streams. A factory
receives `{:cwd ... :mcp-tools ... :session-id ... :model-id ...}` and returns
an Agent. For a static Agent, its configured model ID is advertised
automatically. `:models` adds selectable model IDs; `:run-options` supplies base
Run options.

Model selection uses ACP session configuration options. The prompt response
includes `inputTokens`, `outputTokens`, and `totalTokens` aggregated across the
Run's model calls when each call reported usage. The field is currently marked
unstable by ACP. Tool updates carry standard ACP content as well as raw inputs
and outputs. Harbor's generic ACP adapter can convert these events into ATIF.

```bash
clojure -M -m karcarthy.acp your.namespace/agent-var
```

## Events

```clojure
(k/emit! {:type :application/decision :value value})
(k/events run)
```

`emit!` adds Run and Agent lineage, records the event, and calls the Run's
`:observe` function. `events` returns recorded events from a Run.

## Contracts

```clojure
(k/contract-valid? contract value)
(k/explain-contract contract value)
(k/contract->json-schema contract)
```

Contracts support Clojure specs, predicates, classes, sets, and basic JSON
Schema. See [Data model](/docs/reference/schema) for the value shapes.
