title: Generated Code
description: How model-authored Clojure is read, expanded, checked, evaluated, invoked, repaired, and recorded.
---

Generated behavior is Clojure source, not JSON or an EDN workflow language.
The model usually returns a string because model tool-call boundaries are
serialized, but the harness immediately reads that string into a Clojure form.

## Accepted source

The source must contain one top-level form whose evaluation returns an Agent:

```clojure
(agent
  {:name "generated-reviewer"
   :model {:transport :responses :id "gpt-5.6"}
   :context "Review the change and return concrete findings."
   :tools [read-file search-code]
   :input ::change
   :output ::review})
```

A custom program is equally valid:

```clojure
(agent
  {:name "generated-team"
   :input ::change
   :output ::review}
  [rt change]
  (let [reviews (await-all!
                 [(spawn! rt security-reviewer change)
                  (spawn! rt api-reviewer change)])]
    (invoke! rt editor {:change change :reviews reviews})))
```

## Compilation pipeline

```text
source string
    │
    ▼
read-agent-form
    │ Clojure form (data)
    ▼
macroexpand
    │ expanded form (data)
    ▼
expand + validate contracts
    │ checked form
    ▼
eval-agent-form!
    │ executable value
    ▼
agent?
    │
    ▼
invoke!
```

### Read

Reader evaluation is disabled while source is converted to a form. This
prevents the reader itself from executing `#=` forms. Reading is not the
security boundary—the program is intentionally evaluated later—but it keeps
the compilation phases explicit and observable.

### Expand

Normal Clojure macroexpansion runs in the configured evaluation namespace.
Generated source can use `agent`, application macros, and ordinary Clojure
macros available there. Both original and expanded forms are retained.

### Check

Checking and evaluation include:

- valid Clojure syntax and symbol resolution;
- macro argument contracts;
- agent and tool configuration contracts;
- input/output/environment contract references;
- policy checks selected by the Runtime;
- compiler analysis during evaluation.

Clojure is dynamically typed. “Type checking” here means compiler analysis and
explicit contracts at Agent, Tool, model, and environment boundaries. A static
analyzer can add stronger checks without changing the source language.

### Evaluate

The checked form is evaluated as JVM Clojure. For a trusted local research
harness, evaluation may have the full capabilities of its namespace and
process. Applications may select a restricted namespace, classloader, process,
or sandbox as policy, but the core programming model does not become a smaller
workflow language.

### Verify and invoke

The resulting value must satisfy `agent?`. It then runs through `invoke!` like
an authored Agent, inheriting Runtime lineage, cancellation, budgets,
observation, and approval handling. Root conversation state is explicit Run
input/output rather than an inherited hidden store.

## Recursive generation

A model-backed Agent may receive `(agent)` in its tool set; a custom program
may call the lower-level `compile-agent!` directly. No special recursion
feature is necessary:

```clojure
(let [child-source (k/invoke! rt architect task)
      child (k/compile-agent! rt child-source)]
  (k/invoke! rt child task))
```

The Runtime constrains recursion with shared `:agent-depth`, `:model-calls`,
`:generated-forms`, token, time, and concurrency budgets.

## Repair

Compilation failures are structured data:

```clojure
{:phase :evaluation
 :message "Unable to resolve symbol: fact-cheker"
 :form '(...)
 :line 8
 :column 19}
```

When compilation is exposed as a model tool, that failure becomes the normal
tool result. The model may submit corrected source on its next turn. Repair is
therefore ordinary tool-loop behavior, not a special dynamic protocol.

## Recording and replay

Every compilation records:

- exact source text and a content hash;
- the read form;
- macroexpanded form;
- expansion and contract results;
- evaluation namespace and available capability set;
- parent Agent and run lineage;
- result or structured failure;
- timing and resource usage.

These records allow experiments to compare source mutations, reproduce a
candidate, or verify that two runs evaluated the same program.

## What JSON is still for

JSON remains useful at external boundaries: ACP messages, HTTP requests,
observation export, evaluation datasets, and JSON-compatible outputs. It is
not the representation of executable orchestration.
