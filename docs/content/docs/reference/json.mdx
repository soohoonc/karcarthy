Run agents and workflow files from the `karcarthy` command, and drive
karcarthy from any language over a stdin/stdout JSON protocol. This page
covers the commands, the request and response JSON, and every workflow shape
the JSON bridge accepts.

Build the standalone CLI once:

```bash
clojure -T:build uber
```

Then use `bin/karcarthy` as a normal command:

```bash
./bin/karcarthy agent echo --instructions "Echo the input." hi
./bin/karcarthy run examples/workflows/echo.json hi
./bin/karcarthy demo
./bin/karcarthy --help
```

In a source checkout, the launcher falls back to running from source when the
standalone jar is not built. For installed copies, set
`KARCARTHY_JAR=/path/to/karcarthy-0.0.2-standalone.jar`.

## Commands

### agent

Run one agent without writing a workflow file:

```bash
./bin/karcarthy agent reviewer \
  --instructions "Find the launch risk and answer in one sentence." \
  "Enterprise SSO is ready for staged beta."
```

By default, the command prints the result text. Add `--json` when a caller
needs the full result object:

```bash
./bin/karcarthy agent echo --instructions "Echo the input." hi --json
```

Useful options:

- `--runner mock|claude`
- `--model NAME`
- `--tool NAME` (repeatable)
- `--mock-response AGENT=TEXT` (repeatable)
- `--input TEXT`

### run

Run a workflow JSON file. The file is either a raw workflow object:

```json
{
  "type": "agent",
  "name": "echo",
  "instructions": "Echo the input."
}
```

or a full request object with `workflow`, `input`, and runner fields.

```bash
./bin/karcarthy run examples/workflows/echo.json hi
./bin/karcarthy run examples/workflows/echo.json --input "hi" --json
```

The JSON workflow schema covers portable workflow nodes. Runner configuration
such as `subagent` definitions is supplied by Clojure runner construction or a
runner-specific config file, not embedded as workflow nodes.

### json

The `json` command is the machine protocol: request JSON on stdin, result
JSON on stdout.

```bash
./bin/karcarthy json < request.json
```

## Request shape

```json
{
  "workflow": { "type": "agent", "name": "echo", "instructions": "Echo." },
  "input": "hello",
  "runner": "mock",
  "mock-responses": {
    "echo": "hello back"
  }
}
```

`runner` defaults to `mock`. `claude` uses the Claude runner with sub-agent
defaults.

`mock-responses` is optional. With the mock runner, it maps agent names to
deterministic response text, which lets examples and tests show realistic
output without calling a live model. Agents not listed fall back to the
default echo-style mock response.

`tools` is a per-agent field, not a top-level request key: set it on agent
nodes inside `workflow`, or with repeated `--tool` flags on the `agent`
command. It is a runner allowlist. karcarthy passes the names to runners that
support tool selection; it does not create tool servers or bind MCP
configuration. The mock runner ignores tools.

## Workflow shapes

The same shapes are available as a JSON Schema value from Clojure:

```clojure
(require '[karcarthy :as k])

k/json-schema
```

JSON workflows include portable workflow nodes only. Clojure-local `step`
nodes store function values, so they are available from Clojure code but not
from the JSON bridge.

### agent

```json
{
  "type": "agent",
  "name": "reviewer",
  "instructions": "Find launch risks.",
  "model": "sonnet",
  "tools": ["mcp__feedback__search"]
}
```

Runner selection is not an agent field. It comes from the top-level `runner`
request key shown above.

### pipe

Runs each step in order:

```json
{
  "type": "pipe",
  "steps": [
    { "type": "agent", "name": "researcher", "instructions": "Research." },
    { "type": "agent", "name": "writer", "instructions": "Write." }
  ]
}
```

### branch

Runs every branch on the same input:

```json
{
  "type": "branch",
  "branches": [
    { "type": "agent", "name": "api", "instructions": "Review API." },
    { "type": "agent", "name": "docs", "instructions": "Review docs." }
  ]
}
```

### delegate

Planner/worker shape:

```json
{
  "type": "delegate",
  "planner": { "type": "agent", "name": "planner", "instructions": "Reply with EDN: {:subtasks [\"...\"]}." },
  "worker": { "type": "agent", "name": "worker", "instructions": "Handle one subtask." }
}
```

### reduce

Runs branch or delegate work, then passes an EDN summary to a reducer:

```json
{
  "type": "reduce",
  "source": {
    "type": "branch",
    "branches": [
      { "type": "agent", "name": "api", "instructions": "Review API." },
      { "type": "agent", "name": "docs", "instructions": "Review docs." }
    ]
  },
  "reducer": { "type": "agent", "name": "writer", "instructions": "Synthesize the EDN result summary." }
}
```

### revise

Worker/evaluator loop:

```json
{
  "type": "revise",
  "worker": { "type": "agent", "name": "writer", "instructions": "Draft." },
  "evaluator": { "type": "agent", "name": "critic", "instructions": "Reply with EDN: {:accept? true} or {:accept? false :feedback \"...\"}." },
  "max-rounds": 3
}
```

### route

Choice map keyed by the source result:

```json
{
  "type": "route",
  "source": { "type": "agent", "name": "classifier", "instructions": "Classify." },
  "routes": {
    "launch": { "type": "agent", "name": "launch", "instructions": "Handle launch." },
    "incident": { "type": "agent", "name": "incident", "instructions": "Handle incident." }
  },
  "default": { "type": "agent", "name": "fallback", "instructions": "Handle fallback." }
}
```

The classifier result text should be EDN such as `{:route "launch"}` because
JSON route labels are strings.

### continue

Direct handoff from one workflow to another:

```json
{
  "type": "continue",
  "source": { "type": "agent", "name": "triage", "instructions": "Triage." },
  "to": { "type": "agent", "name": "specialist", "instructions": "Continue." }
}
```

### dynamic

Agent-driven orchestration:

```json
{
  "type": "dynamic",
  "agent": { "type": "agent", "name": "lead", "instructions": "Orchestrate the review." },
  "max-steps": 25
}
```

`max-steps` is optional and defaults to `25`. The agent replies with EDN ops
(`:define`, `:patch`, `:remove`, `:call`, `:spawn`, `:complete`); the op
shapes are documented on the [Data Shapes](/docs/reference/schema#dynamic)
page.

## Response

`karcarthy json` and command runs with `--json` return the karcarthy result
map encoded as JSON:

```json
{
  "type": "result",
  "ok?": true,
  "agent": "writer",
  "text": "..."
}
```

The EDN result tag `:karcarthy/type` serializes as `"type"`: the JSON encoder
drops keyword namespaces.

If a JSON request fails to parse or run, the protocol returns an object with
`ok: false`, `error`, and `exception`.
