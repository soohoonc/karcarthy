---
title: Limits and safety
description: Bound a Run and understand the authority of generated code and Tools.
---

Runs combine model calls, Tool effects, concurrency, and generated JVM code. A
Run should state the resources and authority available to that work.

## Bound the Run

```clojure
(k/run! agent input
        {:limits {:model-calls 40
                  :input-tokens 500000
                  :output-tokens 50000
                  :depth 4
                  :parallelism 4
                  :agent-forms 8
                  :deadline-ms 300000}})
```

Nested Agents created through `:agents` or the built-in `agent` Tool consume
the same budgets. A separate `run!` call creates a separate Run and separate
limits.

| Limit | Bounds |
| --- | --- |
| `:model-calls` | Completed and attempted model turns |
| `:input-tokens` / `:output-tokens` | Reported model usage |
| `:depth` | Nested Agent calls |
| `:parallelism` | Concurrent nested work |
| `:agent-forms` | Submitted runtime Agent forms, including failed attempts |
| `:deadline-ms` | Wall-clock time for the Run |

## Cancel work

Pass a cancellation token through `:cancel`. Cancellation propagates through
the current Run and produces a cancelled Run result with its events retained.

## Approve effects

Tools may require approval on every call or once per Run. Without an approval
handler, a required approval fails closed. Approval controls Tool calls; it
does not reduce the operating-system authority of the process that implements
the Tool.

## Generated source is trusted code

Reader evaluation is disabled while source is read, and karcarthy requires one
valid Agent form. The checked form is then evaluated as JVM Clojure. These
checks prevent malformed Agent submissions; they are not a sandbox.

Run untrusted generation in an isolated process with restricted filesystem,
network, credentials, and namespaces. MCP servers and shell Tools require the
same treatment: they execute with the authority of their processes.
