title: Tools
description: Give an Agent callable functions with validated boundaries.
---

A Tool is a Clojure function a model can call. Its name, description, and input
schema are visible to the model.

```clojure
(k/deftool lookup-user
  {:description "Look up a user by ID."
   :input-schema {:type "object"
                  :properties {"user-id" {:type "string"}}
                  :required ["user-id"]
                  :additionalProperties false}
   :output-schema map?}
  [{:keys [user-id]}]
  (lookup! (:db (k/context)) user-id))
```

Expose the Tool through an Agent:

```clojure
(k/defagent support
  {:model "gpt-5.6"
   :instructions "Resolve the user's request."
   :tools [lookup-user]
   :output-schema string?})
```

## Tool calls

For each model turn, karcarthy sends the Tool's name, description, and JSON
input schema. When the model returns a Tool call, karcarthy:

1. validates its input;
2. runs input guardrails;
3. requests approval when required;
4. calls the Tool function;
5. validates its output and runs output guardrails; and
6. returns the result to the model.

An exception from ordinary Tool work becomes an error result that the model can
inspect and recover from on its next turn. Configuration, schema, guardrail,
approval, budget, deadline, and cancellation failures stop the Run.

## `tool` and `deftool`

Use `deftool` for a top-level definition. Use `tool` to construct the same
value inside an expression:

```clojure
(def lookup-user
  (k/tool
   {:name "lookup-user"
    :description "Look up a user by ID."
    :input-schema {:type "object"
                   :properties {"user-id" {:type "string"}}
                   :required ["user-id"]
                   :additionalProperties false}
    :output-schema map?}
   [{:keys [user-id]}]
   (lookup! (:db (k/context)) user-id)))
```

The body receives only the validated input and may use ordinary Clojure. Read
local dependencies with `(k/context)`; see [Context](/docs/context).

## Schemas and guardrails

`:input-schema` is the source of truth for both local validation and the
model-facing interface. It may be a predicate, spec, or JSON Schema, but it
must be convertible to JSON Schema. Use an explicit object-shaped schema for
normal function Tools.

Explicit schemas use JSON Schema Draft 2020-12 by default. Set `$schema` to
select Draft 4, 6, 7, or 2019-09. The optional `:output-schema` validates the
Tool result.

Input and output guardrails receive `[run-context value]`:

```clojure
{:input-guardrails [allowed-query?]
 :output-guardrails [safe-result?]}
```

## Approvals

Set `:needs-approval` to `:never`, `:once`, `:always`, a boolean, or a function
of `[run-context input]`.

```clojure
(k/run! support prompt
        {:approval
         (fn [{:keys [tool input]}]
           (confirm? (:name tool) input))})
```

Without an approval handler, required approval fails closed. Approval governs a
Tool call; it does not reduce the operating-system authority of the Tool or its
process.

## Tool sources

All callable functions enter an Agent through `:tools`, but they may execute in
different places:

| Source | Execution |
| --- | --- |
| `tool` / `deftool` | Clojure function in the karcarthy process |
| `local-tools` | Filesystem, shell, and search functions rooted at a directory |
| Provider-hosted Tool | Selected model endpoint |
| MCP Tool | External MCP server process |

```clojure
(def tools
  (k/local-tools
   {:cwd "/workspace/project"
    :needs-approval {:read :never
                     :search :never
                     :write :always
                     :edit :always
                     :bash :always}}))
```

`local-tools` supplies `read`, `write`, `edit`, `bash`, and `search`. File Tools
remain under `:cwd`, including after symlink resolution. `bash` is bounded by
timeout and output size but is not a sandbox.

See [Models](/docs/protocols/models#provider-hosted-tools) for hosted Tools and
[MCP](/docs/protocols/mcp) for remote Tools.
