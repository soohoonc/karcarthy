---
title: Port Patterns
description: Express familiar agent framework patterns as karcarthy workflow data.
---

karcarthy is useful when you like a framework's control-flow pattern but want
the plan to remain inspectable data. The helpers in `karcarthy.patterns`
compile familiar shapes back to ordinary `pipe`, `map`, `iterate`, `bind`, and
state-graph workflow data.

All Clojure snippets below assume:

```clojure
(require '[karcarthy :as k])
```

Run the offline overview:

```bash
clojure -M -e '(load-file "examples/clojure/mappings.clj")'
```

## Pattern map

| If you know this | Use this in karcarthy | Core workflow |
| --- | --- | --- |
| CrewAI sequential crews | `crew` | `pipe` |
| CrewAI parallel/hierarchical review | `crew` with process options | `map` plus optional `reduce` |
| AutoGen round-robin group chat | `group-chat` | transcript through `pipe` |
| OpenAI-style specialist handoff | `bind` | route by classifier output |
| Google ADK sequential/concurrent/loop agents | `workflow-agent` | `pipe`, `map`, `iterate` |
| LangGraph state machines | `state-graph` | finite graph with state updates |

## Specialist routing

OpenAI-style specialist routing is just `bind`: one agent classifies, then the
selected specialist receives the original input.

```clojure
(def workflow
  (k/bind triage
          {"refund" refund-agent
           "sales" sales-agent
           "support" support-agent}
          :default support-agent))
```

Equivalent JSON shape:

```json
{
  "type": "bind",
  "source": { "type": "agent", "name": "triage", "instructions": "Classify as refund, sales, or support." },
  "routes": {
    "refund": { "type": "agent", "name": "refund", "instructions": "Handle refunds." },
    "sales": { "type": "agent", "name": "sales", "instructions": "Handle sales questions." },
    "support": { "type": "agent", "name": "support", "instructions": "Handle support." }
  },
  "default": { "type": "agent", "name": "support", "instructions": "Handle support." }
}
```

Run it:

```bash
python3 examples/python/openai/demo.py
npx tsx examples/typescript/openai/demo.ts
```

## Sequential crew

CrewAI's sequential process maps to `pipe`: each task-specific agent receives
the previous task's result text.

```clojure
(k/pipe market-researcher risk-analyst brief-writer)
```

The `crew` helper lets you keep task maps at the edge:

```clojure
(k/crew [{:id :market :agent researcher :description "Find market signal."}
         {:id :risk :agent analyst :description "Name the main risk."}
         {:id :brief :agent writer :description "Write the final brief."}]
        :process :sequential)
```

Run it:

```bash
python3 examples/python/crewai/demo.py
npx tsx examples/typescript/crewai/demo.ts
```

## Round-robin group chat

AutoGen-style deterministic group chat can be represented as a transcript
threaded through `pipe`.

```clojure
(k/group-chat [planner builder reviewer] :rounds 1)
```

Use this when the speaker order is known. Use `bind` when an agent should pick
the next specialist dynamically.

Run it:

```bash
python3 examples/python/autogen/demo.py
npx tsx examples/typescript/autogen/demo.ts
```

## Workflow agents

Google ADK-style workflow agents are deterministic orchestration nodes:

```clojure
(k/workflow-agent :sequential [planner worker reviewer])
```

Mapping:

| ADK-style type | karcarthy node |
| --- | --- |
| sequential | `pipe` |
| concurrent | `map` |
| loop | `iterate` |

## State graph

LangGraph-style state graphs are the one pattern that is not just a functional
combinator. Use `state-graph` when named nodes update shared state and edges
choose the next node.

```clojure
(k/state-graph
 {:start :triage
  :nodes {:triage triage-node
          :launch launch-node
          :incident incident-node}
  :edges {:triage {"launch" :launch
                   "incident" :incident}}})
```

State graphs are useful for finite business processes. Prefer `pipe`, `map`,
`iterate`, and `bind` when those simpler values describe the flow.
