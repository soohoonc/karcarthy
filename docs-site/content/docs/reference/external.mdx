These links are the outside protocol and SDK surfaces used to keep karcarthy's
public vocabulary grounded in existing agent systems.

## Agent Client Protocol

| Reference | Why it matters |
| --- | --- |
| [Session setup](https://agentclientprotocol.com/protocol/v1/session-setup) | Sessions are conversation state, including context, history, and MCP server setup. |
| [Prompt turn](https://agentclientprotocol.com/protocol/v1/prompt-turn) | Defines the user-turn lifecycle, updates, tool calls, stop reasons, and cancellation. |
| [Tool calls](https://agentclientprotocol.com/protocol/v1/tool-calls) | Treats tools as runtime events rather than workflow node definitions. |
| [Session config options](https://agentclientprotocol.com/protocol/v1/session-config-options) | Keeps model, mode, and reasoning selectors in session or adapter config. |
| [Session modes](https://agentclientprotocol.com/protocol/v1/session-modes) | Older compatibility surface; not a core karcarthy resource. |

Takeaways:

- `context` should be session/message state, not a broad resource kind.
- tool calls are adapter/runtime events, not graph IR.
- permission requests belong to the host adapter boundary.

## OpenAI Agents SDK and Responses API

| Reference | Why it matters |
| --- | --- |
| [Using tools](https://developers.openai.com/api/docs/guides/tools) | Hosted tools, function tools, remote MCP tools, and Agents SDK usage. |
| [Tool search](https://developers.openai.com/api/docs/guides/tools-tool-search) | Deferred tool loading through namespaces or MCP servers. |
| [Orchestration and handoffs](https://developers.openai.com/api/docs/guides/agents/orchestration) | Distinguishes specialist ownership from agents-as-tools. |
| [Running agents](https://developers.openai.com/api/docs/guides/agents/running-agents) | Agent loop, sessions, continuation, and streaming. |
| [Results and state](https://developers.openai.com/api/docs/guides/agents/results) | Final output, replay history, ownership, interruptions, and resumable state. |
| [Integrations and observability](https://developers.openai.com/api/docs/guides/agents/integrations-observability) | MCP wiring and traces for model calls, tools, handoffs, and guardrails. |

Takeaways:

- MCP, hosted tools, and function tools should stay in SDK/provider config until
  karcarthy has a concrete reason to interpret them.
- `map` can target agents-as-tools, multiple SDK runs, or graph branches.
- final output, emitted updates, and resumable state should stay separate.

## Claude Agent SDK and Claude Code

| Reference | Why it matters |
| --- | --- |
| [MCP in the SDK](https://code.claude.com/docs/en/agent-sdk/mcp) | Shows MCP configured in code, project files, or remote transports. |
| [Subagents in the SDK](https://code.claude.com/docs/en/agent-sdk/subagents) | Defines programmatic subagents with prompts, tools, and context isolation. |
| [Claude Code subagents](https://code.claude.com/docs/en/sub-agents) | Shows filesystem agent definitions, scoped MCP, permission modes, and skills. |
| [Model Context Protocol overview](https://claude.com/docs/connectors/building/mcp) | Frames MCP around tools, resources, prompts, servers, and security boundaries. |

Takeaways:

- subagents support context isolation and final-message returns, which maps well
  to explicit `map` and `reduce` synthesis.
- MCP servers should remain scoping boundaries for tools and resources.
- permission modes belong in adapter configuration, not workflow data.

## Agent orchestration frameworks

| Reference | Why it matters |
| --- | --- |
| [LangGraph Graph API](https://docs.langchain.com/oss/python/langgraph/graph-api) | State, nodes, edges, and conditional graph transitions. |
| [CrewAI crews](https://docs.crewai.com/en/concepts/crews) | Agents, tasks, and crew processes. |
| [AutoGen group chat](https://autogenhub.github.io/autogen/docs/reference/agentchat/groupchat/) | Multi-agent chat coordination and speaker selection. |
| [Google ADK workflow agents](https://google.github.io/adk-docs/agents/workflow-agents/) | Sequential, concurrent, and loop-style deterministic agents. |
| [DSPy](https://dspy.ai/) | Typed signatures, modules, metrics, and optimization. |

Takeaways:

- recognizable orchestration patterns should compile to the small workflow data
  vocabulary instead of becoming new public concepts.
- typed input/output signatures are future work once validation needs them.
- metrics and optimization can help agents patch behavior, but they should not
  expand the runtime surface before the interpreter has concrete behavior for
  them.

## Agent configuration UX

| Reference | Why it matters |
| --- | --- |
| [OpenClaw agent configuration](https://docs.openclaw.ai/gateway/config-agents) | Agent defaults, per-agent overrides, workspaces, skills, context injection, truncation notices, and context budgets. |
| [OpenClaw system prompts](https://open-claw.bot/docs/concepts/system-prompt/) | Shows a fixed-section prompt assembled from workspace, skills, docs, and safety boundaries. |
| [Hermes Agent docs](https://hermes-agent.app/en/docs) | Highlights setup, active config, skill registries, self-checking through doctor commands, and self-learning loops. |
| [Pi](https://hey.pi.ai/) | Consumer UX example where tone, curiosity, decision support, voice, and emotional intelligence are core product behavior. |

Takeaways:

- prompt text should be rendered from structured product configuration, not
  handwritten as tiny role labels.
- workspace context, tools, skills, memory, and safety boundaries are separate
  inputs that can compile into one agent instruction.
- karcarthy should expose simple workflow data; products built on it can provide
  richer agent configuration screens and templates.
