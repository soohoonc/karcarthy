---
title: State
description: Store named agents, workflows, calls, and history as data.
---

There should not be a separate static-versus-dynamic split. karcarthy treats
the running system as editable data: named agents, named workflows, and the
history of operations applied to them.

```text
state -> operation -> state
```

Pydantic AI, Codex CLI, Claude Agent SDK, OpenAI Agents SDK, local models, and
command processes still own the inner agent loop. karcarthy only coordinates
the outer workflow and state changes.

## State

A running system needs three core registries:

```clojure
{:agents {"planner" {:karcarthy/type :agent
                     :name "planner"
                     :instructions "..."}}
 :workflows {"main" {:karcarthy/type :pipe
                     :steps [{:karcarthy/type :agent-ref
                              :name "planner"}]}}
 :history [{:operation {...}
            :result {...}}]}
```

Everything else stays where it belongs until there is concrete behavior for it:

- tools live on agents, Agent SDKs, CLIs, or their adapters;
- integrations live in SDK/CLI options or MCP config;
- context is input and history passed to the current call;
- safety boundaries live in the host application or adapter.

## Operations

Keep the operation surface small.

| Operation | Meaning |
| --- | --- |
| `:put` | Create or replace an agent or workflow. |
| `:patch` | Merge changes into an existing agent or workflow. |
| `:remove` | Remove an agent or workflow. |
| `:call` | Invoke an agent or workflow. |
| `:complete` | End the current top-level run. |

```clojure
{:op :put
 :resource {:kind :agent
            :id "critic"
            :instructions "Find flaws in the current plan."}}

{:op :patch
 :kind :agent
 :id "critic"
 :merge {:instructions "Find flaws and propose one better plan."}}

{:op :call
 :target "critic"
 :input "Review the current approach."}
```

## Production requirements

Production readiness means the state layer can survive and explain itself:

- append-only operation log;
- state snapshots;
- replay from the log;
- fork from a prior operation;
- crash recovery;
- trace viewer over operations and calls;
- resource accounting for model calls, latency, tokens, and cost;
- schema normalization so the agent can edit state reliably.

Do not add a new resource kind until karcarthy needs to interpret it. The
smallest useful surface is agents, workflows, calls, and history.
