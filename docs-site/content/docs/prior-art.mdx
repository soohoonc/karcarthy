This page is context, not API reference. It records the existing systems and
names that shaped karcarthy's public vocabulary.

Most agent frameworks define agents, tools, and workflows as objects or
decorators in a host language, and bundle the agent loop. karcarthy makes a
different pair of choices: the workflow is plain data, and the loop is delegated
to an agent SDK, agent framework, coding-agent CLI, or command adapter.

## Comparison

| Choice | PydanticAI / DSPy / Agno / Vercel AI SDK | karcarthy |
| --- | --- | --- |
| Define an agent | Host-language object, module, decorator, or typed declaration. | A data map: `{:karcarthy/type :agent ...}`. |
| Agent loop | The framework or SDK runs it. | The selected SDK, framework, CLI, or adapter runs it. |
| Tools | Typed functions, schemas, callables, or SDK tool definitions. | Tool allowlists and execution are handed to that SDK/CLI. |
| Multi-agent | Teams, handoffs, graphs, chats, or workflows. | Functional workflow values: `pipe`, `map`, `reduce`, `iterate`, `bind`. |
| Structured output | Pydantic models, Zod, signatures, or SDK-specific schemas. | Parse the reply today; typed signatures are future work. |
| Streaming | Framework-specific events or tokens. | SDK/CLI-specific, with Claude CLI streaming hooks today. |
| Observability | Framework-specific tracing and dashboards. | OpenTelemetry spans via `karcarthy.otel/instrument`. |

## When karcarthy fits

Use karcarthy when you want orchestration that is inspectable, serializable,
generatable, and transformable as data. It is especially useful when the graph
itself is something an agent might create or edit.

Choose another framework when the main need is a mature batteries-included
runtime, typed Python tools, a prompt optimizer, or a first-class React
streaming UI.

These are not strictly either-or. karcarthy is a thin coordination layer, and
those frameworks are candidates to run inside it as command or SDK-backed
adapters.

## Agent SDKs and protocols

| Reference | Why it matters |
| --- | --- |
| [Agent Client Protocol](https://agentclientprotocol.com/protocol/v1/session-setup) | Separates conversation state, prompt turns, tool calls, and client capabilities. |
| [OpenAI Agents SDK orchestration](https://developers.openai.com/api/docs/guides/agents/orchestration) | Describes specialist routing, handoffs, and agents-as-tools. |
| [OpenAI Agents SDK runs](https://developers.openai.com/api/docs/guides/agents/running-agents) | Defines the agent loop, continuation, and streaming boundary. |
| [Claude Agent SDK MCP](https://code.claude.com/docs/en/agent-sdk/mcp) | Keeps MCP configuration at the SDK boundary. |
| [Claude subagents](https://code.claude.com/docs/en/agent-sdk/subagents) | Shows programmatic subagents with prompts, tools, and context isolation. |

## Orchestration frameworks

| Reference | Why it matters |

| [LangGraph Graph API](https://docs.langchain.com/oss/python/langgraph/graph-api) | State, nodes, edges, and conditional graph transitions. |

## Agent configuration UX

| Reference | Why it matters |
| --- | --- |
| [OpenClaw agent configuration](https://docs.openclaw.ai/gateway/config-agents) | Agent defaults, per-agent overrides, workspace context, skills, tools, and context budgets. |
| [OpenClaw system prompts](https://open-claw.bot/docs/concepts/system-prompt/) | Fixed-section prompts assembled from workspace, skills, docs, and safety boundaries. |
| [Hermes Agent docs](https://hermes-agent.app/en/docs) | Setup, active config, skill registries, doctor commands, and self-learning loops. |
| [Pi](https://hey.pi.ai/) | Consumer UX reference for tone, decision support, and conversational behavior. |

The product can expose rich configuration screens. karcarthy should keep the
orchestration layer small: agents, workflow data, adapters, operations, and
results.
